# ADR-001: Plugin Contract — fetch+map only, no DB writes

## Контекст

При проектировании плагинной архитектуры нужно выбрать: могут ли плагины напрямую писать в БД, или только возвращают данные оркестратору.

## Драйверы решения

- Изоляция плагинов: плагин должен работать без знания о persistence-слое.
- Тестируемость: плагины тестируются без БД, только с mock-транспортом.
- Повторные запуски: оркестратор контролирует idempotency и checkpoints.
- Расширяемость: новый плагин не должен требовать знания о схеме БД.

## Рассмотренные варианты

| Вариант | Плюсы | Минусы | Решение |
| --- | --- | --- | --- |
| Плагин пишет в БД напрямую | Меньше слоёв | Tight coupling, сложные тесты | Отклонён |
| Плагин только fetch+map, Job пишет в БД | Изоляция, простые тесты, чистый контракт | Дополнительный слой Job | **Принято** |

## Решение

Плагины (`plugins/`) реализуют контракт `Collect::Plugin#sync_step`: получают конфигурацию, выполняют fetch из внешнего источника, возвращают структурированный результат (raw items). Запись в БД — исключительно ответственность Job (`jobs/`). Плагины не имеют зависимостей на ActiveRecord или схему БД.

Контракт зафиксирован в `core/lib/collect/plugin.rb`.

## Последствия

### Положительные

- Плагины тестируются изолированно без БД (используя `NullPlugin`).
- Job контролирует checkpoint, retry и idempotency.
- Новый плагин добавляется без знания о persistence.

### Отрицательные

- Каждый новый источник требует Job + Plugin пары.
- Нельзя сохранять промежуточные результаты внутри плагина.

### Нейтральные / организационные

- `core/README.md` — canonical doc для контракта плагина.
- `coding-style.md` закрепляет правило: клиенты/плагины — только fetch+маппинг.
