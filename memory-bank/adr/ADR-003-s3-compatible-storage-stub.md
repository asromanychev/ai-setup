# ADR-003: S3-compatible Storage as MVP Stub

## Контекст

Сервис должен хранить сырые payload и вложения (медиафайлы из Telegram и других источников). Нужно определить стратегию хранения на этапе MVP.

## Драйверы решения

- MVP-фокус: приоритет — `CollectionTask` lifecycle и текстовые данные.
- S3-хранилище требует инфраструктуры (bucket, credentials, upload-логики).
- Не блокировать MVP ради вложений, которые потребители пока не используют.
- Архитектура должна позволять включить реальное S3 позже без рефакторинга контракта.

## Рассмотренные варианты

| Вариант | Плюсы | Минусы | Решение |
| --- | --- | --- | --- |
| Реализовать S3-upload в MVP | Полная функциональность | Блокирует MVP, инфра-зависимость | Отклонён |
| Хранить всё в PG (включая бинари) | Простота | Плохо масштабируется, нарушает разделение | Отклонён |
| S3-совместимый contract + stub в MVP | Контракт есть, реализация позже | Вложения не загружаются до post-MVP | **Принято** |

## Решение

В MVP: S3-переменные присутствуют в ENV contract (`ADR-001`), ссылки на объекты сохраняются в PG, но реальная загрузка не выполняется (заглушка). После MVP: включить реальный S3-upload без изменения модели данных.

## Последствия

### Положительные

- MVP не блокируется инфра-зависимостью.
- Контракт (ссылки в PG, S3 для бинарей) стабилен с первого дня.

### Отрицательные

- Вложения недоступны в MVP.
- Нужно явно документировать ограничение для потребителей.

### Нейтральные / организационные

- `ops/config.md` содержит S3 ENV contract.
- `prd/PRD.md` — Out of Scope: «S3-загрузка вложений».

## Follow-up

- Post-MVP issue: реализовать реальный S3-upload.
