## 3. Качество результата агента

**Итог:** по документам и code review реализация выглядит эквивалентной спецификации, но финальная runtime-верификация неполная: `core/lib` зелёный, Rails model specs в текущем окружении не поднимаются из-за подключения к БД.

**Что хорошо:**
- После очистки Brief агент нормально держал Problem/Solution separation и дальше уже не возвращался к ранним доменным ошибкам.
- В Spec к финальной версии хорошо материализованы adversarial сценарии: `{}` vs `nil`, мутация `position`, direct SQL для ambiguity, независимость разных source.
- Code review быстро сошёлся: ко второй итерации остались только low-risk runtime gates, без доказуемых багов по diff.

**Где потребовалась правка:**
- Агент системно недооценивал `precondition hygiene`: внешний `CheckpointAmbiguityError` несколько циклов трактовался как обычная часть реализации.
- В Plan агент несколько раз писал желаемый тест, не показывая, как сделать его исполнимым в реальной схеме с unique index.
- Финальный runtime gate в документах был доведён до чистого вида, но реальная среда всё равно показала инфраструктурную зависимость на Postgres, которую статический цикл не закрыл.

**Итоговая оценка качества: 7.5/10.** По аналитике и коду результат сильный, но это не такой чистый end-to-end outcome, как в `0001`: главный остаточный риск лежит не в бизнес-логике, а в не подтверждённом model-runtime.

## 4. Что нужно изменить в промптах

Рекомендации основаны на `cycle-analysis-1.md`.

### `01-1-generate-brief.md`
**Что добавить:** для каждого термина в Success Criteria, который задаёт выборку, возобновление или error reaction, агент обязан выписать наблюдаемое различие состояний и наблюдаемый исход; если этого сделать нельзя без ухода в Solution Space, термин должен быть удалён из SC или объявлен блокирующим open question.

### `02-1-review-brief.md`
**Что добавить:** отдельное блокирующее замечание `solution-artifact leakage`, если критерии успеха начинают перечислять миграции, тесты, фикстуры, БД-движок или другие инженерные артефакты вместо поведения системы.

### `01-2-generate-spec.md`
**Что добавить:** обязательный grounding-check для каждого нового helper, bootstrap-механизма, загрузочного пути и внешнего error-класса: `файл существует / зависимость есть / механизм уже работает`. Если нет — выносить в precondition или отдельный issue.

### `02-2-review-spec.md`
**Что добавить:** отдельные проверки `module-budget` и `precondition hygiene`. Если документ лезет больше чем в 3 модульные зоны или оформляет внешний prerequisite как обычную дельту, это должно быть блокирующим fail.

### `01-3-generate-plan.md`
**Что добавить:** если спека требует внешний prerequisite или повреждённое состояние, недостижимое в штатной схеме, план обязан материализовать это через отдельный gate или локальный test-setup шаг. Иначе edge-case считается непокрытым.

### `02-3-review-plan.md`
**Что добавить:** жёсткую проверку на чистый final runtime gate. Последний шаг плана должен содержать только команды и критерий прохождения; любое `если упало — поправить здесь же` должно считаться блокирующим нарушением атомарности.

### `01-4-generate-code.md`
**Что добавить:** перед началом правок агент обязан сверять allowlist файлов из `spec.md`/`plan.md` с фактическими изменениями. Если для прохождения AC нужен файл вне allowlist или закрытие внешнего prerequisite, реализация должна останавливаться, а не молча расширять scope.

### `02-4-review-code.md`
**Что добавить:** перед verdict по эквивалентности code review должен собирать `scope-and-runtime context`: разрешённые файлы из плана, materialized schema, test helpers и фактический diff. Это уменьшит ложные риски и одновременно лучше ловит scope breach.

## 5. Наблюдения по Context Engineering

- **Curated context всё ещё работает лучше raw context:** как только Brief перестал течь в Solution Space, следующие артефакты стали спорить уже не о домене, а о реальных execution-гейтах.
- **Главный источник churn в `0002` был не в предметной логике, а в executable grounding:** precondition, test bootstrap, DB state simulation и final runtime gate.
- **Scope lock важнее, чем кажется:** как только `CheckpointAmbiguityError` был формально вынесен в precondition, Spec и Plan стали заметно чище.
- **Статический анализ без materialized runtime ограничен:** code review показал, что логика выглядит правильно, но текущий запуск model specs всё равно уткнулся в инфраструктуру БД. Для Rails-задач финальный verdict без живого runtime менее надёжен, чем для `core/lib`.
