## Brief

Файл для обновления: `01-1-generate-brief.md`
Условие (Триггер): если критерий успеха опирается на доменный термин, который определяет выборку, возобновление или защитную реакцию системы (`должен попасть в следующий запуск`, `контрольная точка`, `сигнализирует об ошибке`).
Действие (Что добавить в промпт): перед сохранением агент обязан выписать для каждого такого термина одно наблюдаемое различие состояний и один наблюдаемый исход; если термин нельзя определить без ухода в Solution Space, его нужно убрать из Success Criteria или пометить документ как неготовый к утверждению.
Избегать (Граница/Антипаттерн): не оставляй в критериях успеха слова, смысл которых раскрывается только через круговое определение, внутренний статус или неявную реакцию уровня `не оставляет незамеченным`.
Обоснование: это предотвращает основной churn brief-цикла, где документ несколько раз переписывался из-за недоопределённых доменных границ.

Файл для обновления: `02-1-review-brief.md`
Условие (Триггер): если в Brief критерии успеха начинают перечислять миграции, фикстуры, тесты, БД-движок, уровни проверки или другие инженерные артефакты.
Действие (Что добавить в промпт): ревью должно ставить блокирующее замечание `solution-artifact leakage` и требовать заменить каждый такой пункт на наблюдаемое системное поведение, не переходя к wording-правкам, пока Success Criteria не очищены от implementation evidence.
Избегать (Граница/Антипаттерн): не считать инженерный артефакт приемлемым только потому, что он «помогает проверить» критерий или «объясняет реализацию».
Обоснование: это устраняет повторявшуюся ошибку первых версий Brief, где критерии успеха подменялись тестовыми и инфраструктурными деталями.

## Spec

Файл для обновления: `01-2-generate-spec.md`
Условие (Триггер): если в Grounding, Scope или тестовой архитектуре упоминаются новые helper-файлы, gem-зависимости, загрузочные пути, bootstrap-механизмы или инфраструктурные классы вне основной фичи.
Действие (Что добавить в промпт): агент обязан для каждого такого элемента выполнить grounding-check `файл существует / зависимость есть в Gemfile / механизм загрузки уже работает`; если проверка не проходит, нужно либо явно вынести элемент в precondition или отдельный issue, либо сузить спецификацию до уже существующих модульных зон.
Избегать (Граница/Антипаттерн): не описывай `database_cleaner`, `config/application.rb`-интеграцию, `core/lib`-подключение или внешние error-классы как часть фичи без проверки, что они реально есть и входят в scope.
Обоснование: это убирает главный источник spec-итераций, где документ несколько раз ссылался на несуществующую тестовую и загрузочную инфраструктуру.

Файл для обновления: `02-2-review-spec.md`
Условие (Триггер): если спецификация расширяется за пределы доменной фичи и начинает затрагивать более трёх модульных зон или включает внешний prerequisite как обычную дельту реализации.
Действие (Что добавить в промпт): ревью обязано отдельно проверять `module-budget` и `precondition hygiene`: считать модульные зоны по Grounding/Scope, а любой внешний prerequisite требовать оформить как явный precondition с условием остановки, а не как часть текущей реализации.
Избегать (Граница/Антипаттерн): не пропускай спеку только потому, что все отдельные требования выглядят разумными поодиночке; если документ смешал доменную фичу и инфраструктурную обвязку, это блокирующий scope-defect.
Обоснование: это предотвращает поздние plan/code-конфликты, когда внешняя зависимость и инфраструктурные изменения попадали в текущий issue без ясной границы.

## Plan

Файл для обновления: `01-3-generate-plan.md`
Условие (Триггер): если активная spec содержит внешний prerequisite, повреждённое состояние, которое нельзя получить в штатной схеме, или edge-case, нарушающий один из DB-инвариантов.
Действие (Что добавить в промпт): план обязан материализовать это через отдельный наблюдаемый gate или локальный test-setup шаг вида `как проверить prerequisite` / `как временно создать запрещённое состояние` / `как восстановить инвариант после теста`; без такого шага edge-case считается непокрытым.
Избегать (Граница/Антипаттерн): не включай в план тест или implementation step, который невозможен после собственных же миграций и ограничений схемы, и не оставляй prerequisite текстовым предупреждением без исполнимой проверки.
Обоснование: это снимает повторявшиеся замечания про недостижимый ambiguity-case и неисполняемый prerequisite на `Collect::CheckpointAmbiguityError`.

Файл для обновления: `02-3-review-plan.md`
Условие (Триггер): если последний шаг плана запускает команды, но одновременно разрешает дополнительные правки или исправления «по месту».
Действие (Что добавить в промпт): ревью должно требовать чистый final runtime gate: отдельный финальный шаг только с командами и критериями прохождения; любое `если упало — исправить здесь же` должно считаться блокирующим нарушением атомарности, а не stylistic issue.
Избегать (Граница/Антипаттерн): не считать план завершённым, если verification смешан с дальнейшей разработкой или если test bootstrap не даёт воспроизводимого первого запуска.
Обоснование: это устраняет churn поздних plan-итераций, где runtime gate был смешан с доработкой и не давал однозначной точки остановки.

## Code

Файл для обновления: `01-4-generate-code.md`
Условие (Триггер): если `spec.md` или `plan.md` помечают зависимость как precondition / out of scope или ограничивают допустимый набор файлов для реализации.
Действие (Что добавить в промпт): перед правкой агент обязан сравнить плановый allowlist файлов с фактическими изменениями; если для прохождения AC требуется файл вне allowlist или закрытие внешнего prerequisite, реализация должна остановиться и вернуть пользователя к пересмотру spec/plan, а не молча расширять scope.
Избегать (Граница/Антипаттерн): не «лечи» отсутствующий prerequisite правкой исключённого из scope файла и не добавляй функционально безвредные, но внеплановые изменения ради удобства запуска.
Обоснование: это предотвращает межэтапный дефект, когда реализация тихо залезает в внешний infrastructure scope вместо соблюдения утверждённых границ.

Файл для обновления: `02-4-review-code.md`
Условие (Триггер): если code review оценивает эквивалентность по diff, а часть рисков зависит от materialized schema, test helper или фактического набора изменённых файлов относительно `plan.md`.
Действие (Что добавить в промпт): ревью обязано сначала собрать `scope-and-runtime context`: прочитать список разрешённых файлов из `plan.md`, materialized артефакты вроде `db/schema.rb` и реальные test helpers; только после этого разделять `provable defect`, `scope breach` и `runtime-unknown gate`.
Избегать (Граница/Антипаттерн): не повышай severity на предположениях о среде, которые снимаются чтением уже сгенерированных файлов, и не пропускай внеплановые изменения только потому, что они не ломают функциональные AC.
Обоснование: это уменьшает churn code-review и одновременно ловит нарушения scope, которые текущий цикл частично пропустил.

## Кросс-цикловая оптимизация

Правило для системного промпта / workflow: перед утверждением каждого этапа агент обязан проходить двойной gate `traceability + executable grounding`: каждый термин из Success/FR/Invariant должен иметь наблюдаемое определение, а каждый новый файл/зависимость/bootstrapping-ход должен быть подтверждён существующим репозиторием или явно вынесен в precondition.

Правило для системного промпта / workflow: перед code и code-review обязателен `scope lock`: зафиксировать allowlist файлов из активных `spec.md` и `plan.md`, сравнить его с фактическими изменениями и отдельно эскалировать любые внеплановые правки, даже если они выглядят безопасными или «необходимыми для запуска».
