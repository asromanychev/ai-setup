## Замечания

### Замечание 1 — Шаг 8 | Отсутствует тест каскадного удаления (блокирующее)

**Затронут:** шаг 8 (`spec/models/sync_checkpoint_spec.rb`)

**Проблема логики / выполнимости:**  
`spec.md` содержит явный AC и Инвариант 5:

> "После `source.destroy` — `SyncCheckpoint.where(source_id: source.id).count == 0`"  
> "Консистентность FK: после удаления `Source` связанный `SyncCheckpoint` отсутствует в БД."

Шаг 8 перечисляет покрываемые сценарии: обязательность `source`, запрет `position: nil`, допустимость `{}`, поведение при 0/1 записях, ambiguity-триггер. Каскадное удаление (`ON DELETE CASCADE` из миграции шага 3) в этом перечне отсутствует.  

Это не stylistic improvement — инвариант FK-консистентности специфицирован явно, задаётся на уровне БД-ограничения (миграция шага 3), и единственный способ убедиться, что оно работает, — это тест. Без него acceptance gates не закрыты.

**Как исправить:**  
В описание шага 8 добавить сценарий:

> "каскадное удаление: `source.destroy` → `SyncCheckpoint.where(source_id: source.id).count == 0` (проверяет `ON DELETE CASCADE` из миграции шага 3)"
