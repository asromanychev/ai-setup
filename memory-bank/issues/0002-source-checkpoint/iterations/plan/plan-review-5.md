# Plan Review — Issue #0002 (plan-v5.md)

## Общая оценка

**2 блокирующих замечания.** План стал достаточно конкретным по миграциям, моделям и model specs, но всё ещё содержит один неисполняемый внешний gate и один неатомарный финальный шаг.

---

## Замечания

### Замечание 1 — Prerequisite перед шагом 1 | Внешняя зависимость не материализована в наблюдаемую проверку (блокирующее)

**Затронут:** prerequisite перед шагом 1, шаги 3, 7, 8, 9

**Проблема логики / выполнимости:**

План начинается с условия: реализация стартует только после того, как внешний infrastructure issue добавит `Collect::CheckpointAmbiguityError < Collect::Error` в `core/lib/collect/errors.rb`. В текущей кодовой базе этого класса нет: в `core/lib/collect/errors.rb` определены только `Collect::Error`, `Collect::ContractError` и `Collect::UnknownPluginError`.

Это допустимый prerequisite, но сейчас он оформлен только текстовым предупреждением. По правилам шаблона условный шаг или gate должен иметь источник решения раньше самого шага и быть наблюдаем через явную проверку. Здесь такого источника нет:

1. нет отдельного шага, который проверяет наличие `Collect::CheckpointAmbiguityError`;
2. нет команды, которая останавливает исполнение плана, если prerequisite не выполнен;
3. шаги 3, 7, 8 и 9 уже зависят от этого класса, но план не говорит, как именно исполнитель должен установить, что зависимость закрыта.

В результате план остаётся partially executable: его нельзя безопасно запускать на текущем checkout без догадки "проверить руками, появился ли класс".

**Как исправить:**

Нужно превратить prerequisite в явный наблюдаемый gate до implementation steps. Минимальный вариант:

- добавить отдельный нулевой шаг или gate-шаг с конкретной проверкой существования `Collect::CheckpointAmbiguityError` в `core/lib/collect/errors.rb`;
- явно указать, что при отрицательном результате реализация по этому плану не начинается;
- только после этого оставлять шаги 1-9 как собственно implementation plan.

Если команда хочет закрывать этот класс в рамках того же плана, тогда prerequisite нужно убрать и переписать как обычную дельту по `core/lib/collect/errors.rb`, но это уже потребует синхронизации со `spec-v6.md`, где такой change вынесен за scope issue.

### Замечание 2 — Шаг 9 | Финальный runtime gate смешан с дополнительными правками и перестаёт быть gate-шагом (блокирующее)

**Затронут:** шаг 9 (`app/models/source.rb`, `app/models/sync_checkpoint.rb`, `spec/models/source_spec.rb`, `spec/models/sync_checkpoint_spec.rb`, `db/schema.rb`)

**Проблема логики / выполнимости:**

Шаблон требует, чтобы в конце плана был отдельный runtime gate с конкретными командами, подтверждающими acceptance gates. В `plan-v5.md` команды действительно перечислены, но шаг 9 одновременно разрешает:

- запускать миграции и тесты;
- при красных тестах "вносить только те точечные правки в модели/спеки, которые подтверждены failing checks";
- фиксировать `db/schema.rb`.

Это нарушает сразу два требования:

1. **Атомарность:** шаг нельзя независимо выполнить и проверить, потому что он содержит и верификацию, и неограниченный цикл дополнительных изменений.
2. **Delta-first adequacy / Final runtime gate:** вместо чистого описания проверочной дельты шаг фактически говорит "если не прошло, продолжай реализацию здесь же".

На практике такой шаг размывает границу между завершением реализации и новым витком разработки: после падения тестов исполнитель уже не знает, считать ли шаг 9 gate-ом, debug-шагом или местом для дополнительных изменений в production-коде.

**Как исправить:**

Нужно разделить шаг 9 на две разные сущности:

- отдельный финальный runtime gate, который содержит только команды уровня `db:prepare`/`rspec` и критерий успеха;
- если команда хочет оставить пост-gate коррекции, их нужно описывать не внутри финального шага, а как возврат к конкретным предыдущим шагам реализации по наблюдаемому failing check.

В финальной версии плана последний шаг должен быть чистой проверкой: выполнить команды, убедиться, что model specs и `spec/core/collect` зелёные, а coverage gate проходит. Без формулировки "при падении тестов исправить тут же".
