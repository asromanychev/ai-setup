## Замечания

### Замечание 1 — Шаг 8 | Ambiguity-триггер не наблюдаем после шага 3 (блокирующее)

**Затронут:** шаг 8 (`spec/models/sync_checkpoint_spec.rb`)

**Проблема логики / выполнимости:**

Шаг 3 явно требует создать unique index на `sync_checkpoints.source_id`, а шаг 8 предлагает проверить сценарий
`source.sync_checkpoint` при `count > 1` через "прямой SQL или иной обход AR-валидаций".

В текущем плане это не сходится:

1. После шага 3 база обязана запрещать вторую строку с тем же `source_id` на уровне unique index.
2. "Прямой SQL" обходит AR-валидации, но не обходит unique index.
3. Значит описанный в шаге 8 trigger не является наблюдаемым в той схеме, которую сам план создаёт.

Это блокирующее замечание, а не stylistic improvement: ревью требует, чтобы условные и edge-case шаги опирались на реально достижимый источник решения. Сейчас шаг 8 описывает тест, который нельзя выполнить в итоговой схеме без отдельной, заранее описанной дельты по временному ослаблению/обходу DB-invariant.

**Как исправить:**

Нужно переписать шаг 8 так, чтобы ambiguity-сценарий имел конкретный и выполнимый trigger. Подойдут только явно описанные варианты, например:

- либо временно снять/обойти unique index внутри тестового setup и затем восстановить его;
- либо зафиксировать отдельный технический способ симуляции повреждённого состояния БД, который наблюдаемо создаёт `count > 1`;
- либо, если команда не готова поддерживать такой setup, вынести AC/spec-требование про "прямой SQL" на пересмотр, потому что оно противоречит DB-invariant из шага 3.

В текущем виде формулировка "прямой SQL или иной обход AR-валидаций" недостаточна: она не задаёт конкретную дельту и не делает шаг исполнимым.
