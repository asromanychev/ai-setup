# Brief: Персистентность источников и контрольных точек синка

## 1. Цель и Проблема

Оркестратор ingestion не может возобновить инкрементальный обход источника данных, потому что в системе отсутствует постоянное хранилище двух вещей:

1. **Какие источники зарегистрированы** и каков их текущий статус (например, активен ли источник для синка).
2. **До какого момента обработан каждый источник** — позиция, начиная с которой оркестратор должен продолжить при следующем запуске.

Наблюдаемые симптомы:
- При перезапуске процесса sync оркестратор не знает, откуда продолжить: либо начинает с начала и дублирует уже обработанные данные, либо пропускает их произвольно.
- Нет механизма определить, какие из зарегистрированных источников ожидают синка, без запроса к внешнему API.

## 2. Для кого

Основной стейкхолдер — ядро системы (оркестратор), которое управляет инкрементальным синком источников.

На практике это затрагивает разработчика, который:
- подключает новые источники данных;
- пишет тесты для оркестрации синка.

Ему нужна явная и постоянная граница между «что уже обработано» и «что предстоит обработать», доступная без обращения к внешним API и без знания о конкретном типе источника.

Задача является вторым шагом в бэклоге и зависит от контракта плагина (#1): без согласованного контракта неизвестно, что именно фиксировать как состояние для возобновления.

## 3. Доменный контекст

- **Источник (Source)** — зарегистрированная пара «тип плагина + внешний идентификатор» (например, тип `telegram` и id канала). Источник имеет статус, отражающий его готовность к синку (активен / не активен).
- **Контрольная точка синка (SyncCheckpoint)** — сохранённое состояние, фиксирующее, до какого момента источник был успешно обработан. Контрольная точка привязана к конкретному источнику: у каждого источника своя точка возобновления, независимая от других.
- **Инвариант**: для одного источника существует не более одной активной контрольной точки. Нарушение этого инварианта делает результат возобновления синка неопределённым — оркестратор не может выбрать, с какой из точек продолжать.
- Содержимое контрольной точки — состояние, достаточное плагину для продолжения с места остановки. Конкретный формат этого состояния определяется контрактом плагина (#1), а не данной задачей.
- Источники должны быть доступны для выборки по критерию «активен и ожидает синка», чтобы оркестратор мог планировать работу.

## 4. Технический контекст

- Стек: Ruby on Rails, PostgreSQL — хранилище метаданных источников и контрольных точек синка.
- Контракт плагина зафиксирован в задаче #1; данная задача зависит от него как от вышестоящей зависимости — формат состояния, которое фиксируется как checkpoint, определяется там.
- Хранение контрольных точек — в реляционной БД (PostgreSQL); S3 для них не используется.
- Тесты должны работать без внешних сервисов: фабрики и фикстуры создают тестовые записи без сети и без реального плагина.
- Миграции схемы должны быть обратимы, чтобы изменения можно было откатить без потери данных.

## 5. Критерий успеха

Задача считается выполненной, когда:

1. Оркестратор может получить список источников, ожидающих синка, без обращения к внешним API (определяется по статусу источника из хранилища).
2. Для каждого источника оркестратор может прочитать последнее сохранённое состояние синка — либо получить его, либо получить явный ответ «синк ещё не начинался».
3. Инвариант «не более одной активной контрольной точки на источник» является проверяемым: его нарушение предотвращается или обнаруживается на уровне хранилища или модели, и это поведение покрыто автоматическим тестом.
4. Фабрики/фикстуры позволяют создать в тесте источник с контрольной точкой и без неё, без внешних зависимостей.
5. Миграции применяются и откатываются без ошибок в изолированном окружении.

## 6. Что не в скоупе

- Хранение данных, специфичных для Telegram: колонки, выходящие за рамки типа плагина + внешнего идентификатора + JSON-метаданных.
- Логика оркестрации синка: планирование запусков, очередь, retry-политика.
- Интеграция с фоновыми джобами (Sidekiq, ActiveJob).
- S3-хранилище payload и вложений.
- Определение конкретного формата состояния внутри контрольной точки — это ответственность контракта плагина из #1.
- Поддержка более одного типа плагина: для проверки инварианта достаточно одного типа источника.
