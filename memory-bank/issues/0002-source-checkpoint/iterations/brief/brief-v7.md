# Brief: Source Checkpoint Persistence

## 1. Цель и Проблема

Оркестратор первого ingestion-слайса должен уметь возобновлять обход источника не с нуля, а с последней успешно сохранённой позиции. Без этого у системы нет внутреннего способа различить два наблюдаемых состояния:

- источник включён в следующий запуск синка или не должен в него попасть;
- у включённого источника уже есть позиция возобновления или синк по нему ещё не начинался.

Наблюдаемые симптомы:
- после перезапуска сервиса состояние источников нельзя восстановить только из собственной БД;
- повторный sync вынужден либо начинать заново, либо полагаться на внешнюю память вызывающего кода;
- повреждённое состояние с несколькими checkpoint для одного источника не имеет фиксированного способа быть обнаруженным как ошибка.

## 2. Стейкхолдер

Основной стейкхолдер — orchestration-слой ingestion, который выбирает источники для следующего шага и передаёт плагину checkpoint на вход.

Практически это влияет на разработчика, который:
- подключает новый источник;
- проверяет инкрементальный sync после перезапуска;
- диагностирует дубли или пропуски данных.

Ему нужен наблюдаемый и воспроизводимый ответ на вопросы:
- какие `Source` участвуют в ближайшем sync;
- существует ли checkpoint для конкретного source;
- является ли состояние source повреждённым и должно ли оно останавливать продолжение.

## 3. Доменный контекст

- `Source` — зарегистрированная пара `plugin_type + external_id`, представляющая внешний источник данных.
- `sync_enabled` — признак того, должен ли источник попасть в следующий запуск sync.
- `SyncCheckpoint` — последняя сохранённая позиция возобновления для конкретного source.
- Для одного source допустимы только два штатных состояния checkpoint: его нет вовсе или он ровно один.
- Состояние, в котором у одного source одновременно более одного checkpoint, считается неоднозначным и не должно трактоваться как валидная позиция продолжения.
- Формат содержимого checkpoint определяется контрактом плагина и не является частью этой feature.

## 4. Технический контекст

- Исторический первый vertical slice проекта реализован вокруг моделей `Source` и `SyncCheckpoint`.
- В кодовой базе уже есть Rails/ActiveRecord слой и модельные тесты для этой области.
- В roadmap проекта текущий продукт уже мигрирует к `CollectionTask`, но issue `0002` остаётся каноническим артефактом для legacy-slice про `sources`.
- Проблема этой feature не в API, не в фоновых job и не в storage payload; она ограничена собственным persisted state оркестратора.

## 5. Когда и зависимости

- Срочность: историческая core feature первого слайса; нужна как upstream-артефакт для orchestration и raw item persistence.
- Блокирует: корректное описание legacy-оркестрации поверх `sources`.
- Блокируется: контрактом плагина из issue `0001`, который определяет допустимый смысл checkpoint state.
- Порядковые зависимости: `0001` -> `0002` -> `0003`.

## 6. Критерии успеха

1. Что наблюдаем: система может выбрать из собственной БД только те `Source`, у которых `sync_enabled = true`.
   Как измеряем: выборка не зависит от внешнего API или внешней памяти процесса.
2. Что наблюдаем: для конкретного source система различает состояния `checkpoint отсутствует` и `checkpoint существует`.
   Как измеряем: после чтения persisted state результат однозначно равен либо `nil`, либо одному checkpoint-объекту.
3. Что наблюдаем: после сохранения checkpoint следующий sync может использовать сохранённую позицию вместо старта с нуля.
   Как измеряем: после перезапуска/нового чтения persisted state сохранённый `position` читается без потери значения.
4. Что наблюдаем: повреждённое состояние с несколькими checkpoint для одного source не маскируется под штатное продолжение.
   Как измеряем: чтение такого source сигнализирует ошибку неоднозначности, а не выбирает checkpoint молча.

## 7. Вне скоупа

- orchestration-логика выполнения sync;
- `sync_step!`, вызов плагина и обработка его результата;
- фоновые job, retry, очереди;
- хранение raw payload и вложений;
- provider-specific поля сверх идентичности source и checkpoint;
- валидация внутреннего формата `position`.
