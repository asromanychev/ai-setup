# Brief: Персистентность источников и контрольных точек синка

## 1. Цель и Проблема

Оркестратор ingestion не может корректно возобновить инкрементальный обход источника данных, потому что в системе отсутствует постоянная запись двух вещей:

1. Какие источники зарегистрированы и должны участвовать в последующих запусках синка.
2. До какого момента уже обработан каждый источник, чтобы следующий запуск продолжал работу с корректной позиции.

Наблюдаемые симптомы:
- При перезапуске процесса sync оркестратор не знает, откуда продолжить: либо начинает с начала и повторно обрабатывает уже пройденный диапазон, либо продолжает с неверного места и пропускает часть данных.
- Оркестратор не может, опираясь только на внутреннее состояние системы, отделить источники, которые должны попасть в следующий запуск синка, от источников, которые зарегистрированы, но не должны в него входить.

## 2. Для кого

Основной стейкхолдер — ядро системы (оркестратор), которое управляет инкрементальным синком источников.

На практике это затрагивает разработчика, который:
- подключает новые источники данных;
- отлаживает и проверяет сценарии возобновления синка.

Ему нужна однозначная граница между «источник должен быть взят в следующий запуск» и «источник не должен быть взят в следующий запуск», а также между «по источнику уже есть позиция возобновления» и «синк по нему ещё не начинался».

Задача является вторым шагом в бэклоге и зависит от контракта плагина (#1): без согласованного контракта неизвестно, какое состояние считается достаточным для возобновления.

## 3. Доменный контекст

- **Источник (Source)** — зарегистрированная пара «тип плагина + внешний идентификатор» (например, тип `telegram` и id канала).
- **Источник, который должен попасть в следующий запуск синка** — зарегистрированный источник, находящийся в состоянии «включён в sync». Для такого источника после перезапуска системы оркестратор обязан включать его в выборку ближайшего запуска синка. Источник в состоянии «исключён из sync» в эту выборку не входит, пока его состояние не изменено.
- **Контрольная точка синка (SyncCheckpoint)** — сохранённое состояние, фиксирующее, до какого момента источник был успешно обработан. Контрольная точка привязана к конкретному источнику: у каждого источника своя точка возобновления, независимая от других.
- Для одного источника система может иметь либо ни одной контрольной точки, либо ровно одну контрольную точку, которая используется для следующего возобновления. Если система одновременно видит для одного источника более одной такой контрольной точки, результат возобновления становится неоднозначным.
- Содержимое контрольной точки — состояние, достаточное плагину для продолжения с места остановки. Конкретный формат этого состояния определяется контрактом плагина (#1), а не данной задачей.

## 4. Технический контекст

- Проект использует Ruby on Rails; оркестратор уже существует как часть ingestion-пайплайна.
- В текущей системе отсутствует внутренний постоянный источник истины, из которого оркестратор может между запусками прочитать перечень зарегистрированных источников и последнюю позицию обработки по каждому из них.
- В текущей системе отсутствует возможность определить состав источников для следующего запуска, опираясь только на внутреннее состояние системы.
- Контракт плагина зафиксирован в задаче #1; данная задача зависит от него как от вышестоящей зависимости, потому что именно он задаёт границу того, какое состояние считается достаточным для возобновления.

## 5. Критерий успеха

Задача считается выполненной, когда:

1. Оркестратор может получить из внутреннего состояния системы список источников, которые должны попасть в следующий запуск синка, без обращения к внешним API.
2. Для каждого такого источника оркестратор может однозначно различить два состояния: «по источнику уже есть сохранённая позиция возобновления» и «синк по источнику ещё не начинался».
3. Если по источнику есть сохранённая позиция возобновления, оркестратор может прочитать её после перезапуска процесса и продолжить работу не с начала, а с этой позиции.
4. Если система сталкивается с состоянием, в котором для одного источника одновременно существуют несколько контрольных точек, каждая из которых претендует на использование для следующего возобновления, она сигнализирует об ошибке и не продолжает возобновление по такому источнику как по однозначно определённому.

## 6. Что не в скоупе

- Логика оркестрации синка: планирование запусков, очередь, retry-политика.
- Интеграция с фоновыми джобами (Sidekiq, ActiveJob).
- Хранение payload и вложений источника.
- Определение конкретного формата состояния внутри контрольной точки — это ответственность контракта плагина из #1.
- Расширение доменной модели данными, специфичными для отдельного внешнего провайдера, сверх идентичности источника и его состояния синка.
