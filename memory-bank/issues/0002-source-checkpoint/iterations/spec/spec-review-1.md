# Spec Review: Source и SyncCheckpoint (Issue #0002)

Проверена спецификация `spec-v1.md`.

## Вердикты по критериям

### 1. T (Testable): Fail

**Цитата из спеки:** `upsert_checkpoint!` использует `SyncCheckpoint.upsert({source_id: id, position: position.to_json}, unique_by: :source_id)` или AR `upsert` с `returning:`.  
**Проблема для AI-агента:** это требование к способу реализации, а не к наблюдаемому поведению. По нему нельзя писать поведенческий автотест без привязки к внутреннему коду, а `или` оставляет агенту две разные реализации с разными побочными эффектами.  
**Вариант исправления:** заменить на поведенческий контракт: `повторный вызов upsert_checkpoint! с тем же source сохраняет одну запись и делает доступным последнее переданное значение position`.

**Цитата из спеки:** `Все существующие тесты в spec/core/ остаются зелёными.`  
**Проблема для AI-агента:** это мета-требование к процессу проверки, а не acceptance criterion фичи. Оно не проверяет поведение `Source`/`SyncCheckpoint` и не помогает понять, что именно должно работать.  
**Вариант исправления:** оставить это отдельным release-gate вне AC, а в AC перечислить только наблюдаемое поведение доменной модели.

**Цитата из спеки:** `Инварианты уникальности источника и единственности контрольной точки не нарушены.`  
**Проблема для AI-агента:** критерий слишком агрегированный. Если он падает, непонятно, какое именно поведение нарушено и каким тестом это должно фиксироваться.  
**Вариант исправления:** разбить на отдельные AC: уникальность `(plugin_type, external_id)`, одна checkpoint на `source_id`, каскадное удаление, независимость checkpoint разных источников.

### 2. A (Ambiguous-free): Pass

Явных расплывчатых формулировок вроде `быстро`, `удобно`, `при необходимости` или `и т.д.` в спецификации нет. Основные требования сформулированы достаточно предметно.

### 3. U (Uniform): Fail

**Цитата из спеки:** `Состояние empty (Source.for_sync): применяется.`  
**Проблема для AI-агента:** empty-состояние описано только для списка источников, но не для чтения checkpoint, хотя дальше AC различают `source.sync_checkpoint == nil` и наличие объекта. Неприменимость/применимость состояний должна быть материализована для каждого значимого read-path, иначе агент легко недоопределит поведение чтения.  
**Вариант исправления:** в разделе состояний явно добавить для `source.sync_checkpoint` два состояния: `empty` (`nil`, checkpoint ещё не создавался) и `success` (ровно один `SyncCheckpoint` возвращается).

**Цитата из спеки:** `SyncCheckpoint.create! без source_id | ActiveRecord::RecordInvalid: FK constraint`  
**Проблема для AI-агента:** тут смешаны два разных failure path: AR-валидация и ошибка ограничения БД. Для реализации и тестов это неоднозначно: неясно, ожидается ли `RecordInvalid`, `NotNullViolation`, `InvalidForeignKey` или обязательна именно AR-валидация.  
**Вариант исправления:** зафиксировать один путь: либо `belongs_to :source` валидирует присутствие и тест ожидает `ActiveRecord::RecordInvalid`, либо сценарий про прямой SQL отдельно ожидает DB-level exception.

**Цитата из спеки:** в инвариантах есть `Независимость контрольных точек`, но в `Сценариях ошибок и состояний` и AC нет отдельного сценария на обновление двух разных `Source`.  
**Проблема для AI-агента:** invariant-to-AC completeness нарушен. Один из ключевых инвариантов сформулирован, но не материализован в falsifiable scenario, который может упасть при нарушении.  
**Вариант исправления:** добавить сценарий и AC: `обновление checkpoint у source A не меняет checkpoint у source B`.

### 4. S (Scoped): Pass

Спецификация описывает одну фичу, укладывается в ограничение по объёму (`1231` слов) и не расползается на оркестрацию, фоновые джобы или провайдер-специфичную логику.

### 5. Scope явно ограничен: Pass

Раздел Scope чётко отделяет то, что входит в задачу, от того, что запрещено реализовывать в рамках issue: orchestration, jobs, payload storage, provider-specific data и API вынесены за границы.

### 6. Инварианты перечислены: Fail

**Цитата из спеки:** `4. Независимость контрольных точек: изменение position одного источника не затрагивает position другого источника.`  
**Проблема для AI-агента:** этот инвариант перечислен, но из текущей спеки нельзя вывести обязательный тест, который гарантированно упадёт при нарушении. Значит инвариант зафиксирован декларативно, но не operationalized.  
**Вариант исправления:** добавить отдельный AC и state-сценарий с двумя источниками и разными checkpoint, где обновление одного не меняет другой.

**Цитата из спеки:** `5. Консистентность FK: удаление Source каскадно удаляет связанный SyncCheckpoint (dependent: :destroy в AR + ON DELETE CASCADE в миграции).`  
**Проблема для AI-агента:** инвариант формулируется сразу через две реализации (`dependent: :destroy` и `ON DELETE CASCADE`), но не отделяет обязательное наблюдаемое поведение от способа его достижения.  
**Вариант исправления:** оставить инвариант в поведенческой форме: `после удаления source связанный checkpoint недоступен`, а реализационные детали перенести в grounding как необязательные варианты только если они действительно архитектурно закреплены.

### 7. Grounding и реализуемость: Fail

**Цитата из спеки:** `spec/models/source_spec.rb` и `spec/models/sync_checkpoint_spec.rb`  
**Проблема для AI-агента:** текущий репозиторий пока не содержит ни `spec/models/`, ни Rails test helper для модельных спеки. В [spec/spec_helper.rb](/home/aromanychev/edu/aida/ai-da-collect/spec/spec_helper.rb) настроено только `core/lib`-окружение и coverage по `core/lib`, поэтому grounding недостаточно привязан к реальной тестовой архитектуре проекта.  
**Вариант исправления:** явно указать, какой test helper должен быть добавлен или изменён для Rails model specs, и где он будет подключаться.

**Цитата из спеки:** `Тесты моделей подключают Rails-окружение и используют transactional fixtures или database_cleaner`  
**Проблема для AI-агента:** это предположение о существующей тестовой инфраструктуре, которого сейчас в проекте не видно. Спека не указывает реальный файл настройки (`spec/rails_helper.rb`, `spec/support/...`) и тем самым оставляет существенный пробел в реализуемости.  
**Вариант исправления:** привязать это к конкретным путям и текущей структуре проекта, например: `добавить spec/rails_helper.rb` и описать минимальную конфигурацию AR/RSpec, если её ещё нет.

**Цитата из спеки:** `app/models/application_record.rb | Родительский класс (уже существует, не изменяется)` и `core/lib/collect/errors.rb | Добавить Collect::CheckpointAmbiguityError < Collect::Error`  
**Проблема для AI-агента:** эти пути реальны, но сама спека не объясняет, как Rails-часть (`app/models`) должна взаимодействовать с `core/lib/collect/errors.rb` в текущей схеме загрузки. Без этого связь между двумя частями архитектуры остаётся подразумеваемой.  
**Вариант исправления:** явно указать, через какой загрузочный путь или require Rails-модели получают доступ к `Collect::Error`, и в каком существующем файле эта связка уже задаётся.
