# Spec Review: Source и SyncCheckpoint (Issue #0002)

Проверена спецификация `spec-v3.md`.

## Вердикты по критериям

### 1. T (Testable): Pass

Acceptance Criteria в основном описывают наблюдаемое поведение: создание `Source`, выборка через `for_sync`, чтение `sync_checkpoint`, идемпотентный `upsert_checkpoint!`, каскадное удаление и ошибка неоднозначности. По ним можно писать автотесты без проверки внутренних деталей реализации.

### 2. A (Ambiguous-free): Fail

**Цитата из спеки:** `| position: {} (пустой Hash) | Валидация проходит; хранится как {} |`

**Цитата из спеки:** `| app/models/sync_checkpoint.rb | AR: belongs_to :source, validates :position, presence: true |`

**Проблема для AI-агента:** спецификация одновременно требует принимать `position: {}` и предлагает реализацию через `validates :position, presence: true`. В Rails `presence: true` считает пустой `Hash` пустым значением, то есть такая реализация как раз отклонит `{}`. Для агента это не просто деталь реализации, а противоречие в контракте: непонятно, что считать корректным поведением.

**Вариант исправления:** зафиксировать одно правило. Если `{}` должно быть допустимым значением, заменить реализационную подсказку на проверку только `nil` (`validates :position, exclusion: { in: [nil] }` или эквивалентный контракт) и оставить AC на `{}`. Если `{}` не должно проходить, убрать этот сценарий из состояний и AC.

### 3. U (Uniform): Fail

**Цитата из спеки:** `#### 4. Сценарии ошибок и состояния`

**Проблема для AI-агента:** раздел перечисляет success/error/empty кейсы, но не фиксирует, что состояние `loading` для синхронных AR-операций здесь неприменимо. По правилам TAUS неприменимые состояния нельзя оставлять подразумеваемыми. Кроме того, для доменно-специфичного поля `position` не зафиксировано, что отсутствие ожидаемых ключей внутри JSON не валидируется на этом уровне и должно считаться допустимым состоянием, раз формат `position` вынесен за scope.

**Вариант исправления:** в разделе состояний явно добавить формулировки уровня: `loading — неприменимо, все операции синхронные и завершаются в рамках одного вызова` и `missing key внутри position — допустимо на уровне модели, так как формат position валидирует плагин`.

### 4. S (Scoped): Pass

Спецификация описывает одну фичу, не расползается на оркестрацию или API, укладывается в лимит шаблона (`1111` слов) и затрагивает ограниченное число модулей: миграции, AR-модели и тестовую инфраструктуру.

### 5. Scope явно ограничен: Pass

Раздел Scope явно отделяет то, что входит в задачу, от того, что реализовывать запрещено. Оркестрация, фоновые джобы, payload, API и провайдер-специфичная логика корректно вынесены за пределы issue.

### 6. Инварианты перечислены: Pass

Инварианты перечислены явно, и для каждого есть falsifiable сценарий или Acceptance Criterion: уникальность `(plugin_type, external_id)`, единственность checkpoint, неизменность сохранённого значения после мутации аргумента, независимость источников и каскадное удаление.

### 7. Grounding и реализуемость: Fail

**Цитата из спеки:** `spec/rails_helper.rb — новый helper для Rails model specs; загружает Rails-окружение, подключает core/lib через require 'collect', использует DatabaseCleaner со стратегией :transaction.`

**Проблема для AI-агента:** текущий репозиторий действительно содержит `spec/spec_helper.rb`, `config/application.rb` и `core/lib`, но в [Gemfile](/home/aromanychev/edu/aida/ai-da-collect/Gemfile) нет `database_cleaner`/`database_cleaner-active_record`. В таком виде grounding опирается на зависимость, которой проект сейчас не располагает. Агенту всё ещё придётся догадываться: нужно ли добавлять новый gem, или использовать встроенные Rails transactional fixtures/transactions без него.

**Вариант исправления:** либо явно включить добавление нужного gem в scope и grounding, либо убрать `DatabaseCleaner` из спецификации и зафиксировать реализацию через стандартные Rails transactions/fixtures, если это предпочтительный путь проекта.

## Итог

Есть Fail-критерии: `A`, `U`, `Grounding и реализуемость`.
