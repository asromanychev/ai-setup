# Spec Review: Source и SyncCheckpoint (Issue #0002)

Проверена спецификация `spec-v2.md`.

## Вердикты по критериям

### 1. T (Testable): Pass

Acceptance Criteria в целом сформулированы как наблюдаемое поведение: создание `Source`, выборка через `for_sync`, чтение `sync_checkpoint`, каскадное удаление и ошибка неоднозначности. По ним можно писать автотесты без проверки внутренних деталей реализации.

### 2. A (Ambiguous-free): Pass

Явных расплывчатых слов вроде `быстро`, `удобно`, `при необходимости` или `и т.д.` нет. Основные требования заданы предметно и в большинстве случаев допускают однозначную интерпретацию.

### 3. U (Uniform): Fail

**Цитата из спеки:** `Повторный source.upsert_checkpoint!(position: { cursor: 99 }) обновляет значение; в таблице sync_checkpoints остаётся ровно одна запись для данного источника`

**Проблема для AI-агента:** покрыт только повторный вызов с другим значением `position`, но не материализован boundary case `повторный вызов с теми же аргументами`, который явно требуется проверять для доменно-специфичных режимов. Без этого спека не фиксирует идемпотентное поведение `upsert_checkpoint!` и оставляет лазейку для реализации, которая ведёт себя по-разному на одинаковом повторном вызове.

**Вариант исправления:** добавить в `Сценарии ошибок и состояний` и в AC отдельный сценарий: повторный вызов `source.upsert_checkpoint!(position: { cursor: 42 })` с тем же значением сохраняет ровно одну запись и возвращает/делает доступным то же значение без побочных эффектов на другие записи.

**Цитата из спеки:** `Source#sync_enabled — boolean, default false; допустимые значения true или false; nil — ActiveRecord::RecordInvalid.`

**Проблема для AI-агента:** не зафиксировано, как система должна вести себя на граничных значениях boolean-поля, которые Rails обычно приводит типами: `"true"`, `"false"`, `"0"`, `"1"`, `0`, `1`. Сейчас определён только случай `nil`, а поведение на остальных `invalid state value` остаётся подразумеваемым.

**Вариант исправления:** явно указать допустимую границу для входа: либо разрешены только Ruby boolean `true/false`, а любые иные значения приводят к `RecordInvalid`, либо зафиксировать ожидаемое поведение AR-type-casting и добавить falsifiable AC хотя бы для одного не-boolean значения.

### 4. S (Scoped): Fail

**Цитата из спеки:** `#### 4. Сценарии ошибок и состояния`

**Проблема для AI-агента:** по формальному ограничению шаблона спецификация должна быть меньше 1500 слов, а текущий документ содержит 1530 слов. Фича остаётся одной и не выходит за пределы 3 модулей, но ограничение по размеру уже нарушено, значит документ разросся сильнее, чем допускает процесс.

**Вариант исправления:** сократить дублирование между разделами `Функциональные требования`, `Сценарии ошибок и состояния` и `Acceptance Criteria`: оставить в требованиях только контракты, а часть повторяющихся примеров перенести либо в AC, либо убрать.

### 5. Scope явно ограничен: Pass

Границы сформулированы явно: перечислено, что входит в задачу, и отдельно перечислено, что запрещено реализовывать в рамках issue. Оркестрация, фоновые джобы, API и провайдер-специфичные данные корректно вынесены за scope.

### 6. Инварианты перечислены: Pass

Инварианты заданы явно и для каждого из них в документе есть falsifiable сценарий или Acceptance Criterion: уникальность `Source`, единственность `SyncCheckpoint`, независимость checkpoint разных источников, неизменность сохранённой позиции после мутации аргумента и каскадное удаление.

### 7. Grounding и реализуемость: Fail

**Цитата из спеки:** `db/migrate/YYYYMMDDHHMMSS_create_sources.rb`  
**Цитата из спеки:** `db/migrate/YYYYMMDDHHMMSS_create_sync_checkpoints.rb`

**Проблема для AI-агента:** спецификация привязана к типичному Rails-пути миграций, но не учитывает текущее состояние проекта: в репозитории есть `db/cache_schema.rb`, `db/queue_schema.rb` и `db/seeds.rb`, но сейчас нет `db/migrate/` и не зафиксировано, как именно в этом приложении ведётся primary schema. Для AI-агента это оставляет существенную неопределённость в точке входа для персистентного слоя.

**Вариант исправления:** явно зафиксировать, что primary DB для доменных моделей вводится через новый каталог `db/migrate/`, либо указать существующий механизм ведения основной схемы, если он уже стандартизирован проектом.

**Цитата из спеки:** `spec/rails_helper.rb | Новый файл: загружает Rails-окружение ... Использует DatabaseCleaner со стратегией :transaction для изоляции тестов`

**Проблема для AI-агента:** текущая тестовая инфраструктура проекта опирается на `spec/spec_helper.rb` и проверку coverage только для `core/lib`. Спека добавляет Rails model specs и `DatabaseCleaner`, но не связывает это явно с существующим test setup: где должен жить новый helper, как он соотносится с `spec/spec_helper.rb`, и должен ли coverage-gate распространяться на Rails-часть. Реализуемость в этом месте всё ещё требует догадки.

**Вариант исправления:** дополнить grounding конкретным правилом для тестовой архитектуры: какие спеки остаются на `spec/spec_helper.rb`, какие переходят на `spec/rails_helper.rb`, как подключается `core/lib/collect`, и нужен ли отдельный coverage policy для `app/models`.
