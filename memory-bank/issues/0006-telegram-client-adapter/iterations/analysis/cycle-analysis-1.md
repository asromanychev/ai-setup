**Файл для обновления:** `01-1-generate-brief.md`  
**Условие (Триггер):** Критерий успеха содержит описание поведения при сбое/ошибке  
**Действие:** Перед написанием критерия успеха для error-случая зафиксировать одну семантику (semantics-first gate): запрещено оставлять формулировки вида «не падает с необработанным исключением за пределами зафиксированных случаев» без явного определения «зафиксированных случаев» в том же документе

## 2. Spec

**Файл для обновления:** `01-2-generate-spec.md`  
**Условие (Триггер):** Раздел «Uniform — состояния» содержит строку с новым error state  
**Действие:** После заполнения таблицы состояний выполнить mandatory state→AC traceability pass: для каждой строки таблицы (кроме «Неприменимо») убедиться, что в AC-списке есть numbered AC, который явно покрывает это состояние. Если нет — добавить AC перед сохранением v1  
**Избегать:** Оставлять состояние только в prose-описании строки таблицы без номерного AC  
**Обоснование:** В spec-v1 `Telegram::TimeoutError` присутствовал в таблице состояний, но не имел AC — что потребовало fix-итерации добавления AC-15.

---

**Файл для обновления:** `01-2-generate-spec.md`  
**Условие (Триггер):** Раздел «Инварианты» содержит invariant, отмеченный как труднотестируемый  
**Действие:** Для каждого инварианта, который нельзя проверить автоматически (логирование, side-channel), добавить явную пометку `*(Enforced by code review; not automated)*` прямо в тексте инварианта — и НЕ добавлять для него AC; для всех остальных инвариантов — обязательный falsifiable AC  
**Избегать:** Создавать AC для инвариантов, которые требуют захвата log output на уровне unit-spec (неоправданная сложность)  
**Обоснование:** В spec-v1 инвариант 4 (bot_token не логируется) не имел пометки, что создало неопределённость при ревью.

---

**Файл для обновления:** `02-2-review-spec.md`  
**Условие (Триггер):** Review проверяет `Acceptance Criteria`  
**Действие:** Добавить явный state-to-AC matrix check: построить таблицу «строка состояния → AC номер»; если хотя бы одна строка без Неприменимо остаётся без AC — это Fail по `testability`, независимо от остального текста  
**Избегать:** Засчитывать prose adversarial case как эквивалент numbered AC  
**Обоснование:** review-1 обнаружил этот gap, но мог бы быть более механистичным в его выявлении.

## 3. Plan

**Файл для обновления:** `01-3-generate-plan.md`  
**Условие (Триггер):** Код в delta шага содержит `sleep`, `wait`, `retry` с задержкой, `Kernel.sleep` или аналогичные timing-вызовы  
**Действие:** Обязательно добавить `retry_delay:` (или аналогичный injectable delay параметр) в конструктор/интерфейс класса; в Шаге с тестами явно описать, как передавать `retry_delay: 0` для изоляции retry-тестов от реальных задержек; объяснить почему (N sleep × M секунд = N*M сек на example)  
**Избегать:** Оставлять `sleep` в production-коде без механизма test-isolation  
**Обоснование:** В plan-v1 `sleep` в retry loop не имел test-isolation параметра, что потребовало fix-итерации plan-v2.

---

**Файл для обновления:** `02-3-review-plan.md`  
**Условие (Триггер):** Plan содержит timing/retry логику в delta шага  
**Действие:** Добавить явный test-timing check: если в delta шага есть `sleep`/retry-with-delay, проверить наличие injectable delay в конструкторе; если нет — это Fail по `test_materialization`  
**Избегать:** Засчитывать retry-логику без injectable delay как «готовую к тестам»  
**Обоснование:** plan-review-1 нашёл этот gap, но явный gate помог бы обнаружить его автоматически.

## 5. Кросс-цикловая оптимизация

**Правило 1: Table→AC mandatory pass gate**

Условие: применять на этапах Spec и Plan ко всем артефактам с таблицей состояний, состояниями ошибок или invariant-списком  
Действие: После завершения генерации документа, до сохранения — выполнить проход по всем таблицам и спискам: каждая строка (кроме явно «Неприменимо») должна иметь один numbered AC или явную пометку «enforced by X». Если gap найден — добавить AC до сохранения  
Избегать: Сохранять документ с непокрытыми состояниями в расчёте на то, что reviewer найдёт их  
Обоснование: В этом цикле 2 из 3 fix-итераций (spec-v2, plan-v2) были вызваны отсутствием трассировки state→AC; это gate убрал бы их до review.

---

**Правило 2: Pre-generate infrastructure scan**

Условие: применять до начала Spec-генерации для фич с internal timing/network/retry  
Действие: Перед написанием Spec выполнить проверку «timing isolation»: если фича содержит retry/sleep/network — план обязан предусматривать injectable timing parameter; зафиксировать это как инвариант в VibeContract («retry_delay injectable for test isolation»)  
Избегать: Откладывать это решение на этап Plan, где оно вызовет fix-итерацию  
Обоснование: Проблема с `sleep` в retry была обнаружена только на этапе Plan review; если бы Spec содержал `retry_delay` как инвариант — Plan v1 был бы готов к активации без fix.
