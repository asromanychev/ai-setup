## 1. Цель и Проблема

Сервис ai-da-collect не имеет компонента для получения сообщений из публичных Telegram-каналов. Telegram-плагин не может начать ingestion: нет fetch-слоя, который выполняет запросы к внешнему Telegram API, обходит пагинацию и возвращает сырые данные в формате, пригодном для передачи в ядро.

**Наблюдаемый симптом:** при попытке запустить sync для Telegram-источника компонент отсутствует — нет ни класса, ни интерфейса, который можно было бы вызвать с идентификатором канала и получить в ответ сообщения.

---

## 2. Стейкхолдер (Для кого)

**Разработчик ядра и плагина ai-da-collect.**

Им нужен изолированный fetch-компонент, который принимает идентификатор публичного канала и курсор пагинации, а возвращает структурированные сырые данные сообщений. Без этого компонента невозможно реализовать Telegram-плагин и начать ingestion.

Задача стоит в бэклоге под номером 6 и является прямым предусловием для следующего шага — написания Telegram-плагина.

---

## 3. Доменный контекст

- **ai-da-collect** использует плагинную архитектуру: каждый источник изолирован.
- **Клиент/адаптер** в этой архитектуре — компонент уровня fetch: он знает только о внешнем API источника и его формате ответа. Бизнес-логика (checkpoint, retry на уровне плагина, запись в БД) находится за пределами клиента.
- **DTO** (Data Transfer Object) — простой структурированный объект с сырыми полями сообщения (без обработки смысла, без нормализации).
- Telegram API возвращает сообщения постранично через курсор (offset). Клиент обязан поддерживать последовательный обход страниц.
- **Rate limit** — реальное ограничение Telegram API. Зафиксированная семантика обработки: клиент выполняет ограниченное число повторных попыток в соответствии с задержкой, указанной API; если все попытки исчерпаны — сигнализирует вызывающему ошибкой с семантикой «rate limit exceeded».
- Проект **не** выполняет нормализацию, AI-анализ, embeddings — только ingestion сырых данных.

---

## 4. Технический контекст

- **Стек:** Ruby 3.4, Rails API-only, Docker.
- **Размещение:** в директории, предназначенной для fetch-клиентов проекта (`app/clients/` или `app/adapters/`) согласно конвенциям.
- **Внешний API:** публичный Telegram API. Конкретный транспорт и версия протокола — решение уровня Spec, не Brief.
- **Тестирование:** тесты используют заглушки HTTP-ответов; в CI реальные запросы к Telegram API не выполняются.

## 6. Критерий успеха

1. **Сообщения доступны:** при вызове компонента с идентификатором публичного Telegram-канала и начальным курсором возвращается коллекция объектов с сырыми полями сообщения (как минимум: идентификатор сообщения, дата публикации, текстовое содержимое) — без интерпретации смысла.

2. **Пагинация работает:** при последовательных вызовах с курсором из предыдущего ответа каждый следующий вызов возвращает следующую страницу. Когда сообщений больше нет, компонент явно сигнализирует об этом вызывающему (признак «конец данных»).

3. **Rate limit не обрывает работу неожиданно:** при получении ответа об ограничении от API компонент выполняет повторные попытки согласно зафиксированной семантике (см. Доменный контекст); при исчерпании всех попыток вызывающий получает ошибку с семантикой «rate limit exceeded» — и никогда не получает необработанное сетевое исключение.

4. **Тесты проходят без реальной сети:** весь набор тестов выполняется в CI-среде без HTTP-запросов к Telegram API — результат детерминирован и воспроизводим.

---

## 7. Вне скоупа

- Реализация Telegram-плагина (запись в БД, вызов ядра, checkpoint).
- Интерпретация или нормализация содержимого сообщений.
- Загрузка вложений в S3.
- Работа с приватными каналами.
- Управление аутентификацией за рамками передачи credentials как параметра компонента.
- AI-анализ, NLP, embeddings.
