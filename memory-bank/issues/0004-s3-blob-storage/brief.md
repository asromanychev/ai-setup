# Brief: S3-Compatible Storage for Raw Blobs

## 1. Цель и Проблема

Система сбора данных уже исходит из того, что PostgreSQL должна хранить только метаданные и ссылки на сырые данные, а raw payload, которые по архитектурному правилу проекта не должны сохраняться в PostgreSQL целиком, должны жить во внешнем object storage. Для задачи ingestion это ещё не превращено в единое прикладное поведение, поэтому у системы нет зафиксированной и проверяемой границы, на которой raw blob передаётся во внешнее хранилище, а приложение получает обратно ссылку на уже сохранённый объект, пригодную для последующего сохранения в метаданных без помещения самого содержимого в PostgreSQL.

Наблюдаемые симптомы:
- Для raw blob, который по правилам проекта должен храниться вне PostgreSQL, нет согласованного пути хранения, совместимого с уже зафиксированной архитектурной границей "в object storage хранятся сырые данные, в PostgreSQL только ссылки".
- Из issue не следует единое ожидаемое поведение для сохранения raw blob: какой результат должен получить вызывающий код, чтобы затем однозначно сослаться из метаданных на тот же уже сохранённый объект.
- Отсутствует проверяемая граница, позволяющая показать, что сценарий хранения raw blob воспроизводим без зависимости от реального AWS-аккаунта и без размещения секретов доступа в репозитории.

## 2. Для кого

Основной стейкхолдер — разработчик ingestion-системы, которому нужен однозначный сценарий сохранения raw blob во внешнем object storage так, чтобы прикладной код мог сохранить в метаданных ссылку на уже сохранённый объект, а не переносить сами байты в PostgreSQL.

Это также затрагивает сопровождающего CI и инфраструктурные сценарии локальной разработки: им нужен воспроизводимый способ проверить поведение хранения без выдачи реальных облачных секретов и без требования живого AWS/S3 окружения.

Приоритет высокий, потому что issue отмечен как следующий шаг бэклога после базовых задач по контракту плагина, checkpoint и orchestration. Без этой границы последующие шаги ingestion будут либо хранить сырые данные вне зафиксированной архитектуры проекта, либо оставят неявным критический переход от "получили raw payload" к "можем безопасно сослаться на него из метаданных".

## 3. Доменный и Технический контекст

- Корневые правила проекта уже фиксируют архитектурную границу: сервис ingestion работает на Rails API-only и PostgreSQL, при этом object storage S3-compatible используется для сырых payload и вложений, а в PostgreSQL должны храниться только метаданные, checkpoint и ссылки.
- Ядро ingestion на текущем этапе не зависит от сети, Telegram API, PostgreSQL, S3 или Sidekiq напрямую; это означает, что задача хранения должна оставаться совместимой с существующим разделением ответственности и не затягивать сетевую или провайдерскую логику в места, где её раньше не было.
- В `Gemfile` уже присутствует зависимость `aws-sdk-s3`, а конфигурация проекта в целом следует правилу "конфиг только через ENV"; следовательно, задача не про появление нового класса проблем, а про материализацию уже заявленного storage-контекста в прикладном поведении системы.
- Issue явно ограничивает задачу хранением raw blob через S3-совместимое object storage и отдельно подчёркивает, что в CI нельзя требовать реальный AWS. Это вводит жёсткое ограничение на проверяемость: поведение хранения должно быть воспроизводимо без внешнего облачного аккаунта.
- В рамках этого brief "raw blob, который не должен храниться в PostgreSQL целиком" означает сырой payload, для которого прикладной сценарий хранения обязан завершаться сохранением содержимого во внешнем object storage и возвратом ссылки на сохранённый объект. Вложения и правила их обработки остаются вне рамок этой задачи.

## 4. Критерий успеха

Задача считается выполненной, когда на уровне поведения системы можно показать следующее:

1. Когда приложению требуется сохранить raw blob, который по правилам задачи не должен храниться в PostgreSQL целиком, система сохраняет содержимое во внешнем object storage и возвращает вызывающему коду ссылку на сохранённый объект, которую затем можно сохранить в метаданных и использовать для однозначной ссылки на тот же сохранённый объект.
2. Сценарий сохранения raw blob воспроизводим в непроизводственном контуре без отдельного AWS-аккаунта, без секретов, размещённых в репозитории, и без предварительно созданных внешних ресурсов, которые нужно подготавливать вручную специально для проверки этой задачи.
3. Решение задачи не требует размещения секретов доступа к object storage в репозитории; приложению передаются только внешние параметры доступа, не закреплённые в кодовой базе.
4. Успешное сохранение raw blob определено однозначно: задача считается решённой только в сценарии, где вызывающий код получает результат, достаточный для последующего сохранения ссылки на уже сохранённый объект в метаданных; сценарии, в которых такая ссылка не возникает, не считаются выполнением задачи.

## 5. Что не в скоупе

- Публичная раздача данных через presigned URL не входит в рамки этой задачи.
- Lifecycle policies, retention management и иные правила долгосрочного управления объектами в хранилище.
- Проектирование отдельной модели хранения blob в PostgreSQL не входит в рамки этой задачи; задача ограничена сценарием, в котором приложение может сослаться в метаданных на объект, уже сохранённый во внешнем object storage.
- Реальная интеграция CI с внешним AWS-аккаунтом или иным обязательным облачным окружением.
- Любая логика обработки содержимого payload после его сохранения, включая нормализацию, анализ и downstream-пайплайны.
- Любая логика хранения вложений, отличная от рассматриваемого сценария сохранения raw blob.
