# Cycle Analysis: Issue #0004 — S3-Compatible Raw Blob Storage Adapter

## 1. Brief

Файл для обновления: `01-1-generate-brief.md`  
Условие (Триггер): если issue использует более широкий архитектурный фон (`payload и вложения`, `blob и attachments`, `storage и persistence`), а сама задача фактически покрывает только один подтип артефакта или один переход.  
Действие (Что добавить в промпт): добавить `scope-canonicalization gate`: перед сохранением Brief агент обязан выбрать один канонический объект scope и провести его без подмены через `Цель и Проблема`, `Для кого`, `Контекст`, `Критерий успеха` и `Что не в скоупе`; более широкий фон разрешён только как архитектурный контекст с явной пометкой, что он не расширяет текущую задачу.  
Избегать (Граница/Антипаттерн): запрещено обещать в ранних разделах `raw blob и вложения`, а проверять в критерии успеха только `raw blob`.  
Обоснование: это предотвращает brief-churn v2-v4, где один и тот же документ одновременно обещал покрытие attachments и ограничивал success-path только blob storage.

Файл для обновления: `03-1-fix-brief.md`  
Условие (Триггер): если review указывает на недоопределённые термины в `Критерий успеха` или на конфликт между `Критерий успеха` и `Открытые вопросы`.  
Действие (Что добавить в промпт): fix-step обязан применять `term-closure gate`: каждый спорный термин либо переписывается в наблюдаемое поведение, либо выносится в открытый вопрос, который блокирует готовность документа; синонимическая замена не считается исправлением.  
Избегать (Граница/Антипаттерн): запрещено лечить замечание заменами вроде `идентификатор` -> `стабильная ссылка` -> `доменный результат`, если критерий всё ещё нельзя проверить без домысливания.  
Обоснование: это убирает повторяющийся churn brief-v1..v3, где fix-итерации меняли слова, но не закрывали сам success-path.

## 2. Spec

Файл для обновления: `01-2-generate-spec.md`  
Условие (Триггер): если в проектируемом API появляется factory/runtime branch, завязанная на `ENV`, secrets, bootstrap, client construction или другой внешний runtime, которого нет в active Brief.  
Действие (Что добавить в промпт): добавить `runtime-branch split gate`: любая такая ветка должна пройти одно из двух решений до сохранения спеки: либо быть полностью grounded в текущем issue с файлами, AC и allowlist, либо быть явно вынесенной в `Preconditions / out of scope / next issue`; третьего состояния нет.  
Избегать (Граница/Антипаттерн): запрещено одновременно держать в scope `from_env`-подобный путь и писать, что bootstrap/load-path/ENV wiring не меняются.  
Обоснование: это предотвращает главный дефект spec-v1, где скрытый runtime через `from_env` создал поздний разрыв между спецификацией и реально допустимым scope.

Файл для обновления: `02-2-review-spec.md`  
Условие (Триггер): если FR описывает union-type или duck-typed контракт (`String или IO-like`, `Hash`, optional fields) либо таблица состояний содержит adversarial case.  
Действие (Что добавить в промпт): ревью обязано строить матрицу `FR/input branch -> success AC -> invalid-branch AC` и считать fail любой типовой веткой, которая осталась только в prose или покрыта только happy-path/nil-case.  
Избегать (Граница/Антипаттерн): не засчитывать общий AC за покрытие отрицательных веток вроде `body` не `String` и не `read`-совместим, либо IO-case, который описан в таблице, но не материализован отдельным falsifiable AC.  
Обоснование: это ловит не только явный IO-gap из spec-review-1, но и позднее расширение тестов/кода для invalid body, которое active spec формально не материализовала.

## 3. Plan

Файл для обновления: `01-3-generate-plan.md`  
Условие (Триггер): если active spec содержит одновременно новый целевой suite и требование `existing tests remain green`.  
Действие (Что добавить в промпт): сделать `two-tier runtime gate` обязательным: план обязан разделять `focused verification` нового контракта и отдельный финальный `regression gate` по существующим suites с одной явной командой и pass/fail сигналом.  
Избегать (Граница/Антипаттерн): запрещено завершать план единственным запуском нового spec-файла, если active spec требует сохранить зелёным существующий suite.  
Обоснование: это устраняет defect plan-v1, где acceptance по новому сервису был материализован, а регрессионный контракт проекта нет.

Файл для обновления: `02-3-review-plan.md`  
Условие (Триггер): если verification-команда в плане задевает внешний runtime prerequisite: БД, Rails boot, unset secrets, filesystem, network isolation или другой execution dependency.  
Действие (Что добавить в промпт): ревью должно применять `verification-prerequisite closure gate`: для каждой финальной команды требовать более ранний observational proof именно тех prereq’ов, без которых команда не стартует, либо явный blocker в плане.  
Избегать (Граница/Антипаттерн): не одобрять план только потому, что команда конкретна; если финальный gate включает DB-backed specs, а ранний шаг доказал лишь загрузку Rails или отсутствие `AWS_*`, это всё ещё неполный план.  
Обоснование: это закрывает поздний runtime-unknown, когда plan-v2 уже был одобрен, но локальная PostgreSQL prerequisite для финального regression gate так и не была материализована.

## 4. Code

Файл для обновления: `02-4-review-code.md`  
Условие (Триггер): если по scorecard остаётся `runtime_unknown` или в markdown review есть хотя бы один открытый `runtime-unknown gate`.  
Действие (Что добавить в промпт): запретить финальную фразу `0 замечаний`, пока scorecard не имеет `verdict = ready_for_activation` и пока все execution-gates не материализованы; при `runtime_unknown` review обязано завершаться явным незакрытым gate, а не нулевым замечанием.  
Избегать (Граница/Антипаттерн): нельзя одновременно писать `Эквивалентно`, оставлять непроверенный regression gate и завершать review формулой `0 замечаний`.  
Обоснование: это устраняет прямое противоречие code-review-1, где scorecard честно оставила `runtime_unknown`, но markdown review прозвучал как готовность без оговорок.

Файл для обновления: `01-4-generate-code.md`  
Условие (Триггер): если active plan включает финальный runtime gate, завязанный на среду исполнения вне чистого unit-level path.  
Действие (Что добавить в промпт): добавить `pre-completion runtime gate`: до объявления реализации завершённой агент обязан либо материализовать среду для финальной команды, либо зафиксировать точный внешний blocker с командой, на которой он проявился; завершение только по локальному unit-pass запрещено.  
Избегать (Граница/Антипаттерн): не считать кодовый этап завершённым только потому, что новый spec зелёный, если план ещё требует DB-backed regression gate или другой внешний prerequisite.  
Обоснование: это переносит обнаружение execution blocker раньше и уменьшает churn между реализацией, code review и fix-contract.

## 5. Кросс-цикловая оптимизация

Файл для обновления: `workflow/system gate`  
Условие (Триггер): при переходе между любыми стадиями `Brief -> Spec -> Plan -> Code -> Analysis`.  
Действие (Что добавить в промпт): ввести `artifact-completeness gate`: стадия считается завершённой только если для неё существует канонический набор артефактов в `iterations/<stage>/` по правилам workflow; отсутствие rubric/scorecard/fix-contract/activation должно либо блокировать переход, либо быть явно зафиксировано как допустимое исключение.  
Избегать (Граница/Антипаттерн): не переходить к следующей стадии, когда анализ вынужден опираться только на markdown review, потому что scorecard или rubric не были созданы вообще.  
Обоснование: это снизит аналитическую неоднозначность текущего цикла, где Brief прошёл без канонических scorecard-артефактов.

Файл для обновления: `workflow/system gate`  
Условие (Триггер): при сохранении review, scorecard, fix-contract и activation-артефактов.  
Действие (Что добавить в промпт): ввести `canonical-path gate`: authoritative артефакты текущего цикла разрешено сохранять только в `iterations/<stage>/` с каноническим именованием; stray-файлы вне stage-папки должны либо перемещаться, либо явно маркироваться как неавторитетные и не открывающие gate.  
Избегать (Граница/Антипаттерн): не смешивать в одном issue несколько конкурирующих review-историй вроде `iterations/spec-review-1.md` и `iterations/spec/spec-review-1.md`.  
Обоснование: это уменьшает шум анализа и предотвращает ложные выводы из неканонических review-файлов.
