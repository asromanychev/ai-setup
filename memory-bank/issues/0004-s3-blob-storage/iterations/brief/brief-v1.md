# Brief: S3-Compatible Storage for Raw Blobs

## 1. Цель и Проблема

Система сбора данных уже исходит из того, что PostgreSQL должна хранить только метаданные и ссылки на сырые данные, а сами крупные raw payload и вложения должны жить во внешнем object storage. Для задачи ingestion это ещё не превращено в единое прикладное поведение, поэтому у системы нет зафиксированной и проверяемой границы, на которой крупный сырой контент может быть передан во внешнее хранилище, а приложение получить обратно стабильную ссылку на него без помещения самого содержимого в PostgreSQL.

Наблюдаемые симптомы:
- Для крупного raw payload нет согласованного пути хранения, совместимого с уже зафиксированной архитектурной границей "в object storage хранятся сырые данные, в PostgreSQL только ссылки".
- Из issue не следует единое ожидаемое поведение для сохранения сырого blob: какой результат должен получить вызывающий код на уровне доменного исхода и что считается успешным завершением операции хранения.
- Отсутствует проверяемая граница, позволяющая изолированно удостовериться, что секреты доступа не попадают в репозиторий и что сценарий хранения может проверяться в CI без зависимости от реального AWS-аккаунта.

## 2. Для кого

Основной стейкхолдер — разработчик ingestion-системы, который должен сохранять крупные raw payload и вложения так, чтобы прикладной код мог ссылаться на них через стабильный идентификатор хранения, а не переносить сами байты в PostgreSQL.

Это также затрагивает сопровождающего CI и инфраструктурные сценарии локальной разработки: им нужен воспроизводимый способ проверить поведение хранения без выдачи реальных облачных секретов и без требования живого AWS/S3 окружения в тестовом контуре.

Приоритет высокий, потому что issue отмечен как следующий шаг бэклога после базовых задач по контракту плагина, checkpoint и orchestration. Без этой границы последующие шаги ingestion будут либо хранить крупные данные вне зафиксированной архитектуры проекта, либо оставят неявным критический переход от "получили raw payload" к "можем безопасно сослаться на него из метаданных".

## 3. Доменный и Технический контекст

- Корневые правила проекта уже фиксируют архитектурную границу: сервис ingestion работает на Rails API-only и PostgreSQL, при этом object storage S3-compatible используется для сырых payload и вложений, а в PostgreSQL должны храниться только метаданные, checkpoint и ссылки.
- Ядро ingestion на текущем этапе не зависит от сети, Telegram API, PostgreSQL, S3 или Sidekiq напрямую; это означает, что задача хранения должна оставаться совместимой с существующим разделением ответственности и не затягивать сетевую/провайдерскую логику в места, где её раньше не было.
- В `Gemfile` уже присутствует зависимость `aws-sdk-s3`, а конфигурация проекта в целом следует правилу "конфиг только через ENV"; следовательно, задача не про появление нового класса проблем, а про материализацию уже заявленного storage-контекста в прикладном поведении системы.
- Issue явно ограничивает задачу хранением крупных raw payload через S3-совместимое object storage и отдельно подчёркивает, что в CI нельзя требовать реальный AWS. Это вводит жёсткое ограничение на проверяемость: поведение хранения должно быть воспроизводимо без внешнего облачного аккаунта.
- В issue недоопределено, что именно считается "крупным" payload, как формируется значение `storage_key`, и где проходит граница между raw payload и другими типами файлов. Эти термины достаточны для черновика brief, но их отсутствие не позволяет считать некоторые детали домена окончательно закрытыми.

## 4. Критерий успеха

Задача считается выполненной, когда на уровне поведения системы можно показать следующее:

1. Когда приложению нужно сохранить крупный raw payload во внешнее object storage, оно завершает операцию без помещения самих байтов в PostgreSQL и получает идентификатор хранения, на который затем можно ссылаться из метаданных.
2. Один и тот же прикладной сценарий хранения воспроизводимо проверяется в автоматических тестах и CI без требования реального AWS-аккаунта, реальных облачных секретов или ручной инфраструктурной подготовки.
3. Эксплуатационный контур задачи не требует хранения секретов доступа в репозитории: все чувствительные параметры, необходимые для обращения к object storage, подаются через конфигурацию окружения, а отсутствие секретов в кодовой базе остаётся проверяемым инвариантом задачи.
4. Контракт поведения хранения описан достаточно явно, чтобы вызывающий код мог однозначно понимать, какой наблюдаемый результат означает успешное сохранение сырого blob, а какой сценарий не считается решённым в рамках этой задачи.

## 5. Что не в скоупе

- Публичная раздача данных через presigned URL, если для этого нужна отдельная нетривиальная логика.
- Lifecycle policies, retention management и иные правила долгосрочного управления объектами в хранилище.
- Изменение доменной модели PostgreSQL сверх необходимости ссылаться на уже сохранённый blob.
- Реальная интеграция CI с внешним AWS-аккаунтом или иным обязательным облачным окружением.
- Любая логика обработки содержимого payload после его сохранения, включая нормализацию, анализ и downstream-пайплайны.

## 6. Открытые вопросы

- Что именно считается "крупным" raw payload в границах этой задачи: любой blob, превышающий практический предел для хранения в PostgreSQL, или конкретный класс данных/вложений?
- Должен ли `storage_key` быть только стабильным идентификатором для последующей ссылки, или для домена важны дополнительные наблюдаемые свойства этого значения, например предсказуемость, переносимость между окружениями или возможность восстановить тип объекта?
- Входит ли в задачу единая доменная семантика ошибок хранения на уровне вызывающего кода, или для brief достаточно зафиксировать только наблюдаемый success-path и требование проверяемости без реального AWS?
