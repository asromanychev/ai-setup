# Ревью Brief v1 — Issue #4: S3-совместимый адаптер для raw blob

## Замечание 1 (Блокирующее) — Критерий успеха опирается на недоопределённые термины

**Цитата:**
> Когда приложению нужно сохранить **крупный raw payload** во внешнее object storage, оно завершает операцию без помещения самих байтов в PostgreSQL и получает **идентификатор хранения**, на который затем можно ссылаться из метаданных.

> Контракт поведения хранения описан достаточно явно, чтобы вызывающий код мог однозначно понимать, какой **наблюдаемый результат** означает успешное сохранение сырого blob, а какой сценарий не считается решённым в рамках этой задачи.

**Проблема:**
Критерий успеха строится на терминах, которые сам Brief оставляет открытыми: что именно считается "крупным" payload, какие минимальные свойства обязан иметь "идентификатор хранения" и какой именно результат считается достаточным доменным исходом успешного сохранения. Это не cosmetic wording issue, а дефект проверяемости: без этих определений нельзя однозначно доказать, что задача выполнена, и нельзя отличить завершённый scope от недореализованного.

Проблему усиливает то, что раздел `Открытые вопросы` прямо фиксирует эти термины как незакрытые. Пока они остаются открытыми, критерий успеха не является верифицируемым и допускает разные трактовки одним и тем же командам разработки, ревью и реализации.

**Как исправить:**
Закрыть эти термины прямо в Brief на уровне наблюдаемого поведения, не уходя в implementation details. Например:

> 1. Когда приложению требуется сохранить raw blob, который по правилам задачи не должен храниться в PostgreSQL целиком, система сохраняет содержимое во внешнем object storage и возвращает стабильную ссылку на сохранённый blob, достаточную для последующего сохранения этой ссылки в метаданных.
>
> 4. Успешное сохранение raw blob определено однозначно: вызывающий код получает доменный результат, позволяющий сослаться на уже сохранённый blob из метаданных; сценарии, в которых такой ссылки не возникает, не считаются выполнением задачи.

После этого раздел `Открытые вопросы` нужно либо сузить, чтобы он не подрывал критерий успеха, либо удалить вопросы, которые уже обязаны быть закрыты для прохождения brief-review.

## Итог

Есть блокирующее замечание: критерий успеха пока не верифицируем из-за незакрытых терминов. Сначала нужно определить доменную границу success-path, и только после этого имеет смысл доводить wording.
