# Ревью Brief v3 — Issue #4: S3-совместимый адаптер для raw blob

## Замечание 1 (Блокирующее) — Критерий успеха всё ещё опирается на недоопределённые термины

**Цитата:**
> Когда приложению требуется сохранить raw blob, который по правилам задачи не должен храниться в PostgreSQL целиком, система сохраняет содержимое во внешнем object storage и возвращает вызывающему коду **стабильную ссылку** на уже сохранённый объект, достаточную для последующего сохранения этой ссылки в метаданных.

> Сценарий сохранения raw blob воспроизводим в **управляемом непроизводственном контуре** без требования реального AWS-аккаунта, **постоянных облачных секретов** или **ручной подготовки внешней инфраструктуры**.

> Успешное сохранение raw blob определено однозначно: задача считается решённой только в сценарии, где вызывающий код получает **доменный результат**, позволяющий сослаться из метаданных на уже сохранённый объект.

**Проблема:**
Brief снял прежнее `solution-artifact leakage`, но критерий успеха по-прежнему держится на терминах, которые документ не фиксирует достаточно жёстко для верификации. Сейчас из текста нельзя однозначно понять:

- какие минимальные свойства делают ссылку именно "стабильной";
- что именно считается тем самым "доменным результатом";
- какой контур считается "управляемым непроизводственным";
- где проходит граница между допустимой подготовкой и запрещённой "ручной подготовкой внешней инфраструктуры";
- что в рамках задачи считается "постоянными облачными секретами".

Это уже не wording-претензия, а дефект проверяемости критерия успеха. Несколько разных трактовок будут одновременно выглядеть совместимыми с Brief: opaque storage handle, URL, object key или иной идентификатор; локальный контур, CI-контур или заранее поднятая shared-инфраструктура. Пока эти понятия не закрыты на уровне наблюдаемого поведения, нельзя доказать, что задача выполнена одинаково для автора, ревьюера и следующего исполнителя.

Дополнительно проблема усиливается разделом `Открытые вопросы`: он утверждает, что незакрытых вопросов, влияющих на success-path, не осталось, хотя success-path всё ещё зависит от этих терминов.

**Как исправить:**
Нужно зафиксировать эти понятия прямо в Brief, не уходя в implementation details и не подменяя поведение инженерными артефактами. Например:

> 1. Когда приложению требуется сохранить raw blob, который по правилам задачи не должен храниться в PostgreSQL целиком, система сохраняет его во внешнем object storage и возвращает вызывающему коду ссылку на сохранённый объект, которую затем можно сохранить в метаданных и использовать для однозначной ссылки на тот же объект без хранения raw bytes в PostgreSQL.
>
> 2. Сценарий хранения считается воспроизводимым, если команда может запустить его в непроизводственном контуре без отдельного облачного аккаунта, без секретов, постоянно закреплённых за репозиторием, и без предварительно созданных внешних ресурсов, которые нужно подготавливать вручную специально для проверки этой задачи.
>
> 4. Успешное сохранение определено как получение вызывающим кодом результата, достаточного для последующего сохранения ссылки на уже сохранённый объект в метаданных; если такой ссылки на сохранённый объект не возникает, задача не считается выполненной.

После этого раздел `Открытые вопросы` нужно привести в соответствие новой формулировке: либо удалить спорные термины из числа "открытых", либо признать их незакрытыми и не заявлять критерий успеха полностью определённым.
