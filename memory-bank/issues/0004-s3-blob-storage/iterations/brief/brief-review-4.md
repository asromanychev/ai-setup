# Ревью Brief v3 — Issue #4: S3-совместимый адаптер для raw blob

## Замечание 1 (Блокирующее) — Внутреннее противоречие по границе задачи: `raw blob` против `raw blob и вложения`

**Цитата:**
> ...а сами **raw payload и вложения** ... должны жить во внешнем object storage.  
> [раздел `Цель и Проблема`]

> Основной стейкхолдер ... должен сохранять **raw blob и вложения** ...  
> [раздел `Для кого`]

> ...object storage S3-compatible используется для **сырых payload и вложений**...  
> [раздел `Доменный и Технический контекст`]

> Когда приложению требуется сохранить **raw blob**...  
> [раздел `Критерий успеха`]

**Проблема:**
Документ не удерживает одну и ту же границу scope во всех ключевых разделах. В проблеме, стейкхолдере и контексте задача описана как проблема хранения `raw payload и вложений`, но `Критерий успеха` проверяет только сценарий `raw blob`. Это блокирующее противоречие между `Цель и Проблема` и `Критерий успеха`: сейчас невозможно однозначно понять, должна ли задача закрывать оба типа артефактов или только один.

Из-за этого документ допускает две несовместимые трактовки:

- если вложения входят в задачу, критерий успеха неполон и не доказывает устранение заявленной проблемы;
- если вложения не входят в задачу, problem statement и stakeholder statement расширяют scope за пределы того, что задача обязана закрыть.

Это не cosmetic wording issue, а дефект границы задачи. Пока он не устранён, brief не задаёт однозначный контракт для следующего артефакта.

**Как исправить:**
Нужно выбрать одну границу и провести её одинаково через весь документ.

Если задача только про `raw blob`, переписать более широкие места так, чтобы они не обещали покрытие вложений. Например:

> Система уже исходит из того, что сырые данные хранятся вне PostgreSQL, но для сценария сохранения raw blob это ещё не превращено в единое прикладное поведение...

> Основной стейкхолдер — разработчик ingestion-системы, которому нужен однозначный сценарий сохранения raw blob во внешнем object storage...

Если же задача должна покрывать и вложения, тогда `Критерий успеха` нужно расширить до наблюдаемого поведения для обоих случаев, а не только для raw blob.

## Замечание 2 (Блокирующее) — Критерий успеха опирается на недоопределённые термины

**Цитата:**
> ...возвращает вызывающему коду **стабильную ссылку** на уже сохранённый объект...

> Сценарий сохранения raw blob воспроизводим в **управляемом непроизводственном контуре** без требования реального AWS-аккаунта, **постоянных облачных секретов** или **ручной подготовки внешней инфраструктуры**.

> ...вызывающий код получает **доменный результат**, позволяющий сослаться из метаданных на уже сохранённый объект.

**Проблема:**
Критерий успеха всё ещё строится на терминах, которые Brief не фиксирует достаточно жёстко для верификации. Из текста нельзя однозначно вывести:

- какие свойства делают ссылку именно "стабильной";
- что именно считается "доменным результатом";
- какой контур считается "управляемым непроизводственным";
- что здесь считается "постоянными облачными секретами";
- где проходит граница между допустимой подготовкой и запрещённой "ручной подготовкой внешней инфраструктуры".

Это блокирующий дефект проверяемости. Несколько разных реализаций и эксплуатационных сценариев будут одинаково выглядеть совместимыми с Brief, хотя реально задают разные контракты. Проблему усугубляет раздел `Открытые вопросы`, который утверждает, что незакрытых вопросов для success-path не осталось.

**Как исправить:**
Нужно закрыть эти термины прямо в Brief на уровне наблюдаемого поведения, не скатываясь в implementation artifacts. Например:

> 1. Когда приложению требуется сохранить raw blob, который не должен храниться в PostgreSQL целиком, система сохраняет содержимое во внешнем object storage и возвращает вызывающему коду ссылку на сохранённый объект, которую затем можно сохранить в метаданных и использовать для однозначной ссылки на тот же сохранённый объект.
>
> 2. Сценарий хранения считается воспроизводимым, если команда может запустить его в непроизводственном контуре без отдельного облачного аккаунта, без секретов, закреплённых за репозиторием, и без предварительно созданных внешних ресурсов, которые нужно вручную подготавливать специально для проверки этой задачи.
>
> 4. Успешное сохранение определено как получение вызывающим кодом результата, достаточного для последующего сохранения ссылки на уже сохранённый объект в метаданных; если такой ссылки на сохранённый объект не возникает, задача не считается выполненной.

После этого раздел `Открытые вопросы` нужно привести в соответствие новой формулировке и не заявлять success-path полностью закрытым, пока эти понятия не зафиксированы.

## Замечание 3 (Неблокирующее) — В `Что не в скоупе` есть размытая граница

**Цитата:**
> Публичная раздача данных через presigned URL, **если для этого нужна отдельная нетривиальная логика**.

**Проблема:**
Фраза `нетривиальная логика` не задаёт проверяемой границы. Разные участники команды будут по-разному трактовать, что считать тривиальным, а что нет. Для раздела `Что не в скоупе` это плохая формулировка, потому что он должен именно закрывать пространство домысливания.

**Как исправить:**
Убрать оценочное слово и зафиксировать границу без условий. Например:

> Публичная раздача данных через presigned URL не входит в рамки этой задачи.

Если нужен условный scope, его лучше описать через наблюдаемую границу, а не через субъективную оценку сложности.
