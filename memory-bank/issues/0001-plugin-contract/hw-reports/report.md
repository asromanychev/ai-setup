## 3. Качество результата агента

**Итог:** тесты проходят, реализация соответствует спецификации по всем AC.

**Что хорошо:**
- Агент точно следовал grounding'у (конкретные файлы и строки из spec.md).
- После чистого Brief и Spec spec агент быстро сгенерировал правильный план (4 итерации вместо ожидаемых 6+).
- Реализация `deep_dup` / `deep_freeze`, контракт `NullPlugin`, `PluginRegistry` — без архитектурных отклонений от плана.

**Где потребовалась правка:**
- Агент не проявлял инициативу в adversarial сценариях (мутабельные ключи Hash) до явного указания в Spec.
- В ревью кода смешивал `runtime-unknown gate` и `provable defect` — приходилось уточнять вердикт.

**Итоговая оценка качества: 8/10.** Функционально полноценный результат, без внешних зависимостей, с тестовым покрытием >80%.

## 4. Что нужно изменить в промптах

Рекомендации основаны на `cycle-analysis-1.md` (уже сохранён в `iterations/`).

### `01-1-generate-brief.md`
**Что добавить:** перед сохранением Brief — boundary-check для каждого термина в Problem и Success Criteria: либо описание через наблюдаемое поведение, либо пометка как `open question`. Запретить оставлять термин одновременно в SC и в Unresolved.

### `02-1-review-brief.md`
**Что добавить:** обязательную проверку цепочки `symptom → required boundary → scope`. Если симптом нельзя устранить без того, что вынесено в Out of Scope — блокирующее замечание до любых wording-правок.

### `01-2-generate-spec.md`
**Что добавить:** для каждого инварианта про immutability / determinism / isolation — немедленно сгенерировать adversarial сценарий (aliasing, внешняя мутация ключей, повторный вызов). Без такого сценария spec не считается готовой.

### `02-2-review-spec.md`
**Что добавить:** отдельно проверять три вещи: явно ли закрыты неприменимые состояния TAUS; материализован ли каждый инвариант в falsifiable scenario; есть ли edge cases для доменно-специфичных полей (`{}` vs `nil`, missing key, invalid cursor).

### `01-3-generate-plan.md`
**Что добавить:** для каждого edit-шага — delta-first формулировку `что уже есть → чего не хватает → что не трогать`. Любой conditional step обязан ссылаться на предшествующий observable gate (тест или команда).

### `02-3-review-plan.md`
**Что добавить:** блокирующее замечание, если план не завершается отдельным шагом с конкретной командой запуска и проверкой coverage threshold.

### `02-4-review-code.md`
**Что добавить:** явно различать `provable defect` (есть контрпример в коде) и `runtime-unknown gate` (только runtime может подтвердить). Вердикт `Неэквивалентно` — только при наличии доказуемого контрпримера.

## 5. Наблюдения по Context Engineering

- **Curated context vs raw:** чистый Brief резко сокращает итерации на Spec и Plan. Brief с 9 итерациями привёл к Spec с 2 итерациями — прямая зависимость.
- **Перегруженный контекст:** когда в одной сессии Brief + Spec + Plan, агент начинал подтягивать детали Spec в Brief. Разделение этапов на отдельные сессии снизило churn.
- **Grounding:** привязка к конкретным файлам и строкам (`spec.md:39-42`) в плане и ревью кода — существенно повысила точность анализа агента.
