# Spec Review: Plugin Contract (Issue #0001)

Проверена спецификация `spec-v2.md`.

## Вердикты по критериям

### 1. T (Testable): Pass

В спецификации есть явные Acceptance Criteria и проверяемые контрактные требования для `Collect::Plugin`, `Collect::PluginRegistry` и `Collect::NullPlugin`. AC описывают ожидаемое поведение и сообщения об ошибках так, что по ним можно написать автотесты без внешних зависимостей.

### 2. A (Ambiguous-free): Pass

В тексте нет расплывчатых формулировок вроде «быстро», «удобно», «при необходимости» или «и т.д.». Контракты входов, выходов, ошибок и специальных случаев сформулированы однозначно.

### 3. U (Uniform): Pass

Спецификация покрывает success-, error- и empty-сценарии, а также явно фиксирует, что состояние loading/in progress неприменимо для синхронного `sync_step`. Edge cases для `checkpoint_in` и `source_config[:records]` перечислены отдельно и не оставляют пространства для произвольной трактовки.

### 4. S (Scoped): Pass

Это одна узкая фича: контракт шага синка, реестр и тестовая заглушка. Объём текста укладывается в ограничение шаблона, а решение привязано к компактному срезу ядра и его тестов.

### 5. Scope явно ограничен: Pass

Раздел Scope явно отделяет входящий в задачу контрактный слой от того, что запрещено реализовывать в рамках issue: реальные плагины, БД, S3, джобы и персистентность checkpoint.

### 6. Инварианты перечислены: Pass

Инварианты сформулированы явно и проверяемо: иммутабельность входов и выхода, соответствие ключа реестра, детерминизм `NullPlugin` и изоляция от внешних систем.

### 7. Grounding и реализуемость: Pass

Спецификация привязана к реальным путям проекта (`core/lib/collect/*.rb`, `spec/core/collect/*.rb`, `spec/spec_helper.rb`), и перечисленные файлы существуют в репозитории. Предлагаемая реализация соответствует текущей архитектуре Ruby-ядра и тестового слоя.

0 замечаний, спека готова к реализации.
