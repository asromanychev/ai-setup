# Spec Review: Plugin Contract (Issue #0001)

Проверена спецификация `spec-v1.md`.

## Вердикты по критериям

### 1. T (Testable): Pass

В спецификации есть явные Acceptance Criteria и проверяемые контрактные требования для `Collect::Plugin`, `Collect::PluginRegistry` и `Collect::NullPlugin`. По ним можно написать автоматические тесты без обращения к внешним системам.

### 2. A (Ambiguous-free): Pass

В тексте нет размытых формулировок уровня «быстро», «удобно», «при необходимости», «и т.д.». Ошибки, типы входов/выходов и ожидаемые сообщения перечислены однозначно.

### 3. U (Uniform): Fail

**Цитата из спеки:**
> `NullPlugin`, `checkpoint_in: nil` | Возвращает батч из `source_config[:records]`, `cursor: 1`

**Проблема для AI-агента:**
Спека покрывает success, error и empty-сценарии, но не фиксирует несколько важных edge cases и не описывает отдельное состояние "loading/in progress" даже как явно неприменимое для синхронного `sync_step`. Для агента это создаёт пространство для произвольной трактовки контракта. Сейчас не зафиксировано:
- что происходит, если `checkpoint_in` — Hash без `:cursor`;
- что происходит, если `checkpoint_in[:cursor]` не Integer или отрицательный;
- что происходит, если `source_config[:records]` отсутствует или не является Array;
- почему состояние loading отсутствует и считается неприменимым.

**Вариант исправления:**
Добавить в раздел Uniform отдельный блок:
- `Loading/In progress: не применяется, потому что sync_step синхронный и возвращает итог одного шага за один вызов`;
- сценарии для `checkpoint_in: {}` и невалидного `cursor`;
- сценарии для невалидного `source_config[:records]`;
- явное правило, что отсутствие `:cursor` трактуется как `0` либо поднимает `ContractError`.

### 4. S (Scoped): Pass

Это одна фича, объём спецификации меньше 1500 слов, а изменения ограничены контрактом плагина, реестром и тестовой заглушкой. Перечень файлов шире трёх, но по смыслу речь идёт об одном небольшом срезе ядра и его тестах.

### 5. Scope явно ограничен: Pass

Раздел Scope явно перечисляет, что входит в реализацию, и отдельно фиксирует, что запрещено делать в рамках задачи.

### 6. Инварианты перечислены: Pass

Инварианты сформулированы явно: иммутабельность входов и выхода, соответствие ключа реестра, детерминизм заглушки и изоляция от внешних систем.

### 7. Grounding и реализуемость: Pass

Спека привязана к реальным путям проекта (`core/lib/collect/*.rb`, `spec/core/collect/*.rb`, `spec/spec_helper.rb`), и все перечисленные файлы существуют в репозитории. Предлагаемая реализация соответствует текущей архитектуре Ruby-ядра и тестов.
