# Brief Draft

## Цель и Проблема

Для первого vertical slice `ai-da-collect` нужно устранить неопределенность вокруг границы ingestion-плагина: без зафиксированного контракта команда не может стабильно писать и проверять плагины, а также собирать ранние тесты ядра без внешних зависимостей. Сейчас задача не в расширении функциональности, а в снятии архитектурной неоднозначности на самом базовом уровне: что считается одним шагом синка, в каком виде передается состояние между шагами и как система однозначно определяет нужный плагин по его идентификатору.

Из issue следует несколько важных неоднозначностей, которые и составляют problem space:
- семантика checkpoint пока не до конца зафиксирована: допустим и opaque, и структурированный payload, но выбор должен быть единым;
- первый тестовый вертикальный срез должен работать без сети, Telegram и БД, иначе ранние unit/integration tests будут нестабильны и дорогими в сопровождении;
- для неизвестного идентификатора плагина требуется явное, проверяемое поведение вместо неявного fallback.

## Для кого

Основной стейкхолдер — разработчик, который строит ядро ingestion и последующие source plugins для `ai-da-collect`. Также это затрагивает автора тестов и будущих интеграций: им нужна минимальная, предсказуемая граница взаимодействия с плагинами до подключения Telegram, PostgreSQL, S3 и фоновых задач.

Приоритет высокий, потому что issue помечен как первый шаг бэклога для SDD vertical slice: без этой договоренности дальнейшие задачи будут опираться на незафиксированные предположения и могут разойтись по интерфейсам и ожиданиям тестов.

## Доменный и Технический контекст

Доменный контекст задачи — ingestion данных из источников через плагины, где каждый источник должен обрабатываться пошагово и переносить состояние между вызовами. Для системы критично, чтобы шаг синка был воспроизводимым и пригодным для тестирования без внешних интеграций на раннем этапе разработки.

Технический контекст по репозиторию:
- проект находится на Ruby/Rails стеке;
- в репозитории уже выделен `core` как отдельная зона ответственности для минимального plugin core;
- инфраструктурные зависимости проекта в целом существуют или планируются (`Sidekiq`, PostgreSQL, S3, Telegram plugin), но эта задача должна быть изолирована от них;
- локальные и CI-проверки завязаны на автоматизируемые тесты, поэтому поведение ядра должно быть детерминированным и проверяемым без сети и внешних сервисов.

Ограничения и инварианты:
- задача относится только к ядру plugin boundary, а не к конкретному внешнему источнику;
- идентификация плагина должна быть однозначной и стабильной;
- поведение на неизвестный plugin id должно быть явным;
- тестовый плагин нужен как контролируемая среда для проверки контракта, а не как реальная интеграция;
- решение должно сохранить разделение между problem definition и будущей реализацией внешних коннекторов.

## Критерий успеха

Проблема считается решенной, когда команда может однозначно ответить на следующие вопросы без обращения к внешним интеграциям или неформальным договоренностям:
- что обязан предоставлять каждый ingestion-плагин на входе и выходе одного шага синка;
- как система разрешает нужный плагин по идентификатору и что происходит при неизвестном идентификаторе;
- как прогонять ранние unit/integration tests ядра в полностью локальном и детерминированном режиме.

Проверяемый результат на пользовательском уровне для внутреннего стейкхолдера: разработчик может написать тест, подключить тестовый плагин и получить воспроизводимое поведение ядра без Telegram, БД и сети.

## Что не в скоупе

- Реализация внешних интеграций, включая Telegram API.
- Модели и схема PostgreSQL.
- Хранилище в S3.
- Фоновые джобы и orchestration через Sidekiq.
- Любые нефункциональные расширения, не связанные с фиксацией минимальной границы plugin core для первого vertical slice.

## Открытые вопросы

1. В issue явно оставлен незакрытым выбор формата checkpoint: нужен один окончательный вариант, иначе последующие документы и тесты останутся двусмысленными.
2. Не до конца определено, входит ли в текущий scope регистрация каких-либо встроенных плагинов кроме тестового stub-плагина, или на этом шаге требуется только зафиксировать механизм регистрации как таковой.
3. Не уточнено, какие именно сценарии считаются минимально достаточными для "одного шага синка" с точки зрения доменной семантики батча и завершения прохода; это важно для однозначной спецификации следующего уровня.
