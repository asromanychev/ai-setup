## Замечание 1

**Цитата:**
> «Определение конкретного формата данных состояния (checkpoint): ядро не интерпретирует его содержимое — это решение принимается на уровне реализации плагина.»

**Проблема:**
Раздел «Что не в скоупе» содержит три элемента Solution Space:
1. Слово `checkpoint` в скобках — это технический термин реализации, отсутствующий в разделе «Цель и Проблема», где то же понятие корректно называется «формат состояния между шагами синка».
2. «ядро не интерпретирует его содержимое» — это описание поведения будущего решения (ядра).
3. «это решение принимается на уровне реализации плагина» — рассуждение на уровне архитектурного решения.

**Предложение:**
Убрать всю фразу-пояснение после двоеточия. Достаточно: «Определение конкретного формата состояния между шагами синка.»

## Замечание 2

**Цитата:**
> «Регистрация встроенных плагинов помимо stub-плагина для тестов: на данном шаге фиксируется только механизм регистрации как таковой.»

**Проблема:**
Два технических артефакта решения, которых не должно быть в бриф Problem Space:
1. `stub-плагин для тестов` — конкретный артефакт реализации.
2. `механизм регистрации` — название технического механизма.

Brief может говорить о том, что *проблема* затрагивает только минимальный сценарий регистрации, но не называть конкретные компоненты решения.

**Предложение:**
Переформулировать: «Регистрация более чем одного плагина: в рамках задачи достаточно проверки поведения ядра с одним источником.»
