# Brief Draft

## Цель и Проблема

В репозитории `ai-da-collect` отсутствует зафиксированный интерфейс ingestion-плагина: в ядре нет единого контракта для входа и выхода шага синка, что делает невозможным написание тестов ядра без привязки к конкретной реализации плагина. Из этого вытекают три конкретных наблюдаемых симптома:

- формат передаваемого состояния между шагами синка не определён однозначно, что делает невозможным написание стабильных тестов для ядра;
- первый тестовый вертикальный срез не может выполниться без сети, Telegram и БД, что делает ранние тесты нестабильными и дорогими в сопровождении;
- для неизвестного идентификатора плагина в текущем состоянии не зафиксировано единое проверяемое поведение, из-за чего обработка ошибки зависит от конкретной реализации.

## Для кого

Основной стейкхолдер — разработчик, который строит ядро ingestion и последующие source plugins для `ai-da-collect`. Также это затрагивает автора тестов и будущих интеграций: им нужна однозначно определённая граница взаимодействия с плагинами — входной и выходной формат одного шага синка — до подключения Telegram, PostgreSQL, S3 и фоновых задач.

Приоритет высокий, потому что issue помечен как первый шаг бэклога для SDD vertical slice: без этой договоренности дальнейшие задачи будут опираться на незафиксированные предположения и могут разойтись по интерфейсам и ожиданиям тестов.

## Доменный и Технический контекст

Доменный контекст задачи — ingestion данных из источников через плагины, где каждый источник должен обрабатываться пошагово и переносить состояние между вызовами. Для системы критично, чтобы шаг синка был воспроизводимым и контракт плагина был проверяемым без подключения внешних источников данных.

Технический контекст по репозиторию:
- проект находится на Ruby/Rails стеке;
- в репозитории уже выделен `core` как отдельная зона ответственности для минимального plugin core;
- инфраструктурные зависимости проекта в целом существуют или планируются (`Sidekiq`, PostgreSQL, S3, Telegram plugin), но для первого вертикального среза отсутствие изолированного контракта не позволяет проверить ядро отдельно от этих зависимостей;
- локальные и CI-проверки завязаны на автоматизируемые тесты, но в текущем состоянии ядро нельзя проверить детерминированно без сети и внешних сервисов;
- в текущем состоянии граница plugin boundary описана не полностью: формат состояния между шагами синка не зафиксирован, единое поведение для неизвестного plugin id не формализовано, а минимальный сценарий одного шага синка не определён одинаково для всех участников.

## Критерий успеха

Задача считается закрытой, когда поведение ядра при двух сценариях можно проверить без внешних зависимостей:

- один шаг синка завершается без ошибки и возвращает обновлённое состояние, которое может быть передано следующему вызову, без обращения к сети и базе данных;
- запрос плагина по неизвестному идентификатору возвращает конкретное значение (например, `nil` или специфическую ошибку), и это поведение проверено автоматическим тестом в CI;
- оба сценария воспроизводимы в CI без Telegram, сети и базы данных.

## Что не в скоупе

- Реализация внешних интеграций, включая Telegram API.
- Модели и схема PostgreSQL.
- Хранилище в S3.
- Фоновые джобы и orchestration через Sidekiq.
- Определение конкретного формата состояния между шагами синка.
- Регистрация более чем одного плагина: в рамках задачи достаточно проверки поведения ядра с одним источником.
