## Новые правила из ретроспективного применения нового процесса

Эти правила выявлены при прохождении issue #0001 по новому процессу и **не отражены** в cycle-analysis-1.md (создана до введения нового процесса).

### 1. VibeContract обязателен в плане

**Файл:** `01-3-generate-plan.md`  
**Изменение:** Добавить секцию VibeContract (Pre/Post/Invariants) как обязательный раздел плана до начала шагов реализации.  
**Основание:** plan-v4 не содержал VibeContract — это стало blocker при ретроспективном scorecard. Предусловия и постусловия существовали частично, но не были явно оформлены.  
**Не применять ретроактивно:** к уже закрытым issues — только к новым начиная с issue, для которого процесс введён.

### 2. Rubric создаётся при первом проходе этапа

**Файл:** `memory-bank/issues/README.md`  
**Изменение:** Уточнить правило «rubric создаётся при первом проходе» — добавить: если rubric для этапа отсутствует, генерация артефакта блокирована до его создания; исключение только для ретроспективного применения нового процесса к уже закрытым issues.  
**Статус:** Уже зафиксировано в README.md — дубликат, пропускаем.

### 3. Acceptance scenarios создаются до перехода к Plan

**Файл:** `MAKE_NEW_ISSUE.md`  
**Изменение:** Уже зафиксировано (Этап 2.5 BDD-СЦЕНАРИИ, gate ⛔).  
**Статус:** Дубликат, пропускаем.

### 4. Scorecard хранит ретроспективные исключения явно

**Файл:** `memory-bank/templates/scorecard-template.json`  
**Изменение:** Добавить необязательное поле `"_note"` для фиксации исключений при активации (например: blocker принят как допустимое исключение с обоснованием).  
**Статус:** ✅ Применено в `plan-score-4.json` — задокументировано на уровне поля `_note`.  
**Шаблон обновлён:** нет — поле `_note` использовано в конкретном scorecard, добавление в шаблон целесообразно при следующей ревизии шаблонов.
