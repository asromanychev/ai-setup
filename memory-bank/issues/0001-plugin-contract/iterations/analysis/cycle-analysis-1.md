## Brief

Файл для обновления: `01-1-generate-brief.md`
Условие (Триггер): если в Problem или Success Criteria появляется термин, который легко уводит документ в API-уровень (`checkpoint`, `plugin id`, `один шаг синка`, `детерминированный результат`, `минимальная граница`).
Действие (Что добавить в промпт): перед сохранением пройди boundary-check для каждого такого термина: либо опиши его через наблюдаемое поведение на уровне проблемы, либо пометь как открытый вопрос, блокирующий утверждение Brief; запрещено оставлять термин одновременно в success criteria и в unresolved/open questions.
Избегать (Граница/Антипаттерн): не перечисляй форматы возврата, типы значений, классы, тестовые артефакты, конкретные ошибки или варианты реализации как способ “уточнить” критерий успеха.
Обоснование: это убирает основной источник churn в Brief, где агент многократно лечил неоднозначность через Solution Space.

Файл для обновления: `02-1-review-brief.md`
Условие (Триггер): если Brief одновременно заявляет симптом в Problem и исключает связанную границу в Out of Scope.
Действие (Что добавить в промпт): добавь обязательную проверку `symptom -> required boundary -> scope`: если симптом нельзя устранить без того, что вынесено в out of scope, ревью должно ставить блокирующее замечание раньше любых wording-правок и требовать развести интерфейсный контракт и детали реализации.
Избегать (Граница/Антипаттерн): не пропускай внутреннее противоречие ради локальной формулировки “в целом критерий понятен”.
Обоснование: это предотвращает поздний межэтапный дефект, когда Brief одновременно требовал зафиксировать state contract и исключал его из scope.

## Spec

Файл для обновления: `01-2-generate-spec.md`
Условие (Триггер): если в Functional Requirements появляется инвариант про immutability, deep copy, normalization, determinism или isolation.
Действие (Что добавить в промпт): для каждого такого FR/invariant агент обязан сразу добавить в Uniform или AC минимум один adversarial сценарий, который ломает гарантию через aliasing, внешнюю мутацию ключей/значений, повторный вызов или нестандартную структуру входа; без такого сценария спеку нельзя считать готовой.
Избегать (Граница/Антипаттерн): не ограничивайся happy-path и типовыми негативными кейсами уровня `nil`/`wrong type`/`empty`.
Обоснование: это устраняет дефект, из-за которого первая версия спеки пропустила важные edge cases, а code review потом нашёл реальный контрпример с мутабельными ключами Hash.

Файл для обновления: `02-2-review-spec.md`
Условие (Триггер): если Uniform перечисляет success/error, но не фиксирует неприменимые состояния и boundary cases для кастомных инвариантов.
Действие (Что добавить в промпт): ревью обязано отдельно проверять три вещи: явно ли закрыты неприменимые состояния TAUS; материализован ли каждый инвариант в falsifiable scenario; есть ли edge cases для доменно-специфичных полей и режимов (`missing key`, `invalid cursor`, `{}` vs `nil`, constructor-time validation).
Избегать (Граница/Антипаттерн): не ставь Pass по Uniform только потому, что есть таблица ошибок общего вида.
Обоснование: это смещает обнаружение пробелов из позднего plan/code-этапа обратно в review спеки.

## Plan

Файл для обновления: `01-3-generate-plan.md`
Условие (Триггер): если план включает правку существующего файла, уже частично реализующего спецификацию.
Действие (Что добавить в промпт): каждый edit-step обязан начинаться с delta-first формулировки в виде `что уже есть -> чего не хватает -> что не трогать`; если файл, вероятно, уже соответствует spec, сначала зафиксируй это тестом или явной проверкой и только потом допускай условную правку.
Избегать (Граница/Антипаттерн): не формулируй шаг как повтор описания конечного состояния файла без указания конкретной дельты.
Обоснование: это убирает ранние plan-итерации, где агент предлагал переписывать уже корректный `plugin_registry.rb` и расплывчато описывал `plugin.rb`.

Файл для обновления: `01-3-generate-plan.md`
Условие (Триггер): если шаг в плане условный (`если найдено расхождение`, `при необходимости исправить`, `если тесты упадут`).
Действие (Что добавить в промпт): любой conditional step должен ссылаться на предшествующий наблюдаемый gate: конкретный тест, проверку или команду, которая создаёт решение о правке; источник решения обязан стоять раньше условного шага в порядке зависимостей.
Избегать (Граница/Антипаттерн): не оставляй условное редактирование файла без предшествующего test/probe шага.
Обоснование: это предотвращает неатомарные планы, где исполнитель должен догадываться, менять файл или нет.

Файл для обновления: `02-3-review-plan.md`
Условие (Триггер): если план содержит шаги про тесты, но не заканчивается явной runtime-верификацией acceptance gates.
Действие (Что добавить в промпт): ревью обязано ставить блокирующее замечание, если в конце нет отдельного шага с конкретной командой запуска и проверкой финальных ворот из spec, включая coverage и другие declared thresholds.
Избегать (Граница/Антипаттерн): не считать “мы добавили тесты” эквивалентом “мы доказали AC”.
Обоснование: это закрывает системный разрыв между планом изменений и фактической верификацией результата.

## Code

Файл для обновления: `01-4-generate-code.md`
Условие (Триггер): если реализация добавляет deep copy / deep freeze / normalization helpers или меняет семантику derived fields.
Действие (Что добавить в промпт): перед правкой агент обязан выписать для каждого такого инварианта конкретный adversarial test-кейс и реализовать его до или вместе с кодом; отдельно проверить не только значения, но и ключи Hash, aliasing входов и повторный вызов с теми же объектами.
Избегать (Граница/Антипаттерн): не считать инвариант закрытым, если тесты покрывают только значения коллекций и happy-path.
Обоснование: это переносит обнаружение контрпримеров вроде мутабельных ключей из code review в этап имплементации.

Файл для обновления: `02-4-review-code.md`
Условие (Триггер): если ревью видит, что все функциональные FR выглядят реализованными, а невыполненным остаётся только runtime-gate вроде coverage/green suite.
Действие (Что добавить в промпт): ревью должно явно маркировать такой случай как `runtime-unknown gate`, а не как полноценный архитектурный дефект; verdict `Неэквивалентно` разрешён только при наличии доказуемого контрпримера в коде, иначе вывод должен отделять `provable defect` от `неподтверждённого acceptance gate`.
Избегать (Граница/Антипаттерн): не смешивай отсутствие запуска тестов с найденным логическим багом.
Обоснование: это уменьшает churn в позднем ревью и делает замечания более прицельными.

## Кросс-цикловая оптимизация

Правило для системного промпта / workflow: перед переходом на следующий этап агент обязан пройти трассировку `Problem -> Success -> Scope -> FR -> Invariants -> AC -> Plan/Test Gate`; если на любом звене термин, граница или инвариант не материализован в проверяемое условие, этап не считается готовым.

Правило для системного промпта / workflow: для любых условных правок и любых гарантий уровня immutability/determinism workflow должен требовать `observe first, edit second`: сначала тест/проверка, которая делает расхождение наблюдаемым, потом точечная правка, потом отдельный финальный runtime-gate.
