# Brief — Issue #18: CollectionTask Data Model

## 1. Цель и Проблема

В сервисе `ai-da-collect` отсутствует центральная доменная сущность `CollectionTask`, без которой невозможно принять, отследить и завершить ни одно задание на сбор данных.

**Наблюдаемый симптом:** текущая схема БД построена вокруг устаревшей сущности `sources` (с FK `source_id` в `raw_ingest_items` и `sync_checkpoints`), которая не поддерживает:
- жизненный цикл задания (создано → в очереди → сбор → готово → потреблено → удалено / ошибка);
- изоляцию по заказчику (`requester_id`);
- конфигурируемое поведение сбора и политики хранения.

Пока этой сущности нет, ни один новый компонент MVP (Job, API, Webhooks) не может быть реализован — все задачи последующей фазы заблокированы.

## 2. Для кого

**Внешние агенты (requester)** — системы или пользователи, которые отправляют задания на сбор данных и ожидают жизненного цикла от создания до потребления результата.

**Разработчики и downstream-сервисы** — команды, реализующие Job, API и Webhooks, которые сейчас не могут стартовать из-за отсутствия контрактной модели.

**Причина приоритета:** задача имеет статус critical и стоит первой в бэклоге — до её закрытия остальные задачи фазы 7 не могут начаться.

## 3. Доменный контекст

`CollectionTask` — задание на сбор от конкретного заказчика (`requester_id`) через изолированный плагин (`plugin_type`). Каждое задание однозначно идентифицирует внешний объект сбора (`external_id`) в рамках одного заказчика и плагина.

**Бизнес-правила:**
- Жизненный цикл задания линеен и необратим: `created → queued → collecting → ready → consumed → deleted`; боковой путь `→ failed` возможен только из `collecting`.
- `collection_mode` (режим сбора) фиксируется при создании и не может изменяться — смена режима нарушила бы воспроизводимость сбора.
- `retention_policy` определяет, что происходит с собранными данными после потребления: удалить (`delete`) или сохранить (`retain`).
- Дублирующиеся задания (одинаковый `requester_id` + `plugin_type` + `external_id`) не допускаются — это защита от дублирования сбора.
- `consumed_at` и `failed_at` монотонны: после записи не обнуляются.

**Доменные термины:**
- `requester_id` — идентификатор заказчика задания (внешней системы или пользователя).
- `plugin_type` — тип изолированного плагина-сборщика (например, `telegram`).
- `external_id` — идентификатор объекта сбора в системе источника.
- `collection_mode` — способ сбора (`polling` или другой режим, определяемый плагином).
- `state` — текущее состояние задания в lifecycle.

## 4. Технический контекст

**Стек:** Ruby 3.4 / Rails 8 API-only / PostgreSQL / Redis + Sidekiq / Docker.

**Текущее состояние:**
- В схеме БД существуют таблицы `sources`, `raw_ingest_items` (FK `source_id`), `sync_checkpoints` (FK `source_id`).
- Модель `CollectionTask` отсутствует; связанные таблицы (`raw_ingest_items`, `sync_checkpoints`) всё ещё ссылаются на старую сущность.
- Никакого lifecycle-механизма для заданий нет.

**Ограничения среды:**
- `raw_ingest_items` и `sync_checkpoints` должны сохранить свои функции (хранение сырых данных и чекпойнтов), изменив только FK.
- Старые модели `Source` могут остаться как deprecated — их удаление не входит в текущую задачу.
- Сервис не содержит AI-компонентов; только ingestion.

## 5. Когда и временны́е зависимости

- **Дедлайн:** не задан, но задача Critical и первая в бэклоге.
- **Блокирует:** #7, #8, #10 и все задачи NEW-2 — NEW-6 (Job, API, Webhooks, плагины).
- **Блокируется:** нет внешних блокеров; задача может стартовать немедленно.
- **Порядковые зависимости:** должна завершиться до начала любой другой задачи фазы 7.

## 6. Критерий успеха

**CS-1. Жизненный цикл задания работает сквозь все состояния.**
- *Что наблюдаем:* задание можно провести от `created` до `consumed` и до `failed`, используя предусмотренные переходы.
- *Как измеряем:* каждый переход успешно выполняется; недопустимые переходы возвращают ошибку; поведение воспроизводимо при любом начальном состоянии.

**CS-2. Дублирующееся задание отклоняется системой.**
- *Что наблюдаем:* попытка создать второе задание с теми же `requester_id`, `plugin_type` и `external_id` вызывает явную ошибку уникальности.
- *Как измеряем:* система возвращает ошибку уникальности; данные не дублируются в хранилище.

**CS-3. Режим сбора неизменяем после создания.**
- *Что наблюдаем:* попытка изменить `collection_mode` у существующего задания отклоняется системой.
- *Как измеряем:* изменение не принимается системой; значение остаётся прежним.

**CS-4. Связанные данные сохраняют целостность при удалении задания.**
- *Что наблюдаем:* при удалении задания связанные записи в `raw_ingest_items` и `sync_checkpoints` также удаляются.
- *Как измеряем:* после удаления задания ни одной «осиротевшей» записи в связанных таблицах не остаётся.

**CS-5. Состояние `failed` фиксирует причину ошибки; повторная успешная попытка её очищает.**
- *Что наблюдаем:* при переходе в `failed` система сохраняет описание ошибки; при последующем успешном переходе описание ошибки сбрасывается.
- *Как измеряем:* наблюдаемое поведение воспроизводимо.

## 7. Вне скоупа

- API-эндпоинты для управления `CollectionTask` (issue NEW-3 / следующая задача).
- Удаление или миграция данных из старых таблиц `sources` (deprecated, не блокирует MVP).
- Реализация плагинов (Telegram и другие) — отдельные задачи фазы 7.
- Job-оркестрация (`CollectionJob`) — issue #7.
- Webhooks — отдельные задачи.
- Нормализация, embeddings, AI-анализ, UI — вне MVP.
