# Brief — Issue #0005: Raw Ingest Item Persistence

**status:** draft  
**version:** v1  
**date:** 2026-04-07

---

## 1. Цель и Проблема

Система ingestion собирает сырые данные из внешних источников (Telegram-каналы и другие плагины), но в настоящий момент каждый собранный объект нигде не фиксируется в PostgreSQL. Это означает:

- Нет возможности определить, был ли конкретный внешний объект уже обработан (дублирование при повторном запуске).
- Нет записи о том, какой source породил объект и где хранится его тело (ссылка на blob).
- При сбое цикла sync невозможно восстановить, какие объекты уже попали в систему.

**Наблюдаемый симптом:** при повторном запуске ingestion для одного источника система не может отличить новые объекты от уже собранных — нет записи об ingested объектах в БД.

---

## 2. Стейкхолдер (Для кого)

**Стейкхолдер:** компонент оркестрации ingestion (sync-оркестратор, фоновые jobs).

Без учёта собранных объектов невозможно реализовать идемпотентный ingestion: одна и та же запись от провайдера будет попадать в систему многократно. Задача открывает возможность для корректного downstream-pipeline (если таковой появится) и закрывает основной риск дублирования данных.

**Почему сейчас:** задача является прямым продолжением #2 (Source + SyncCheckpoint) и #4 (blob storage). Модель checkpoint фиксирует позицию в потоке, но не фиксирует отдельные объекты. Blob storage фиксирует тело, но без связи с источником и внешним идентификатором.

---

## 3. Доменный контекст

- **Source** — источник данных; каждый объект привязан ровно к одному Source.
- **External id** — идентификатор объекта на стороне провайдера (например, id сообщения в Telegram). Уникален в рамках одного Source.
- **Storage key** — ссылка на тело объекта в blob-хранилище. Присваивается в момент сохранения blob и может отсутствовать для объектов без тела.
- **Idempotent ingestion** — семантика, при которой повторная попытка записать один и тот же объект (по Source + external id) не создаёт дублей. Конкретная семантика (upsert с обновлением полей vs skip-with-log при дубле) должна быть явно зафиксирована.

---

## 4. Технический контекст

- **Стек:** Ruby on Rails API, PostgreSQL, ActiveRecord.
- **Существующие модели:** `Source` (источник), `SyncCheckpoint` (позиция в потоке). Нет модели для отдельного ingested объекта.
- **Соглашение проекта:** модели живут в `app/models/`, миграции — в `db/migrate/`.
- **Предшествующие компоненты:** `Source` уже существует и содержит идентификатор плагина и external id источника; blob storage (issue #4) предоставляет `storage_key`.

Для установления идемпотентности нужна запись, связывающая Source + external_id + storage_key (опционально) + метаданные. Без неё при повторном запуске sync нет наблюдаемого способа понять, был ли объект уже обработан.

---

## 5. Когда и временны́е зависимости

- **Дедлайн:** нет.
- **Блокирует:** корректную идемпотентную вставку в downstream pipeline; полную замкнутость ingestion-цикла.
- **Блокируется:** концептуально зависит от #2 и #4 (можно начинать независимо, как только Source существует).
- **Порядковые зависимости:** Source (`#2`) должен существовать. Blob storage (#4) желателен, но storage_key опционален.

---

## 6. Критерий успеха

1. **Идемпотентность зафиксирована:** для пары (source + external_id) существует ровно одна запись в системе независимо от числа попыток записать один и тот же объект. Поведение при дубле (перезапись или пропуск с логом) задокументировано и единообразно.

2. **Связь с источником:** каждая запись об ingested объекте содержит ссылку на конкретный Source. По этой ссылке можно однозначно определить, из какого источника получен объект.

3. **Ссылка на тело:** каждая запись может содержать ссылку на тело объекта в blob-хранилище. Объекты без тела допустимы; отсутствие ссылки не является ошибкой.

4. **Метаданные сохранены:** произвольные метаданные объекта, поступившие от провайдера, сохраняются вместе с записью и не теряются.

---

## 7. Вне скоупа

- Семантическая нормализация и парсинг метаданных.
- Embeddings, RAG, AI-анализ.
- Downstream pipeline обработки.
- Управление retention / удаление старых записей.
- API-эндпоинты для чтения ingested объектов.
- Интеграция с конкретным плагином (Telegram и др.) — только контракт модели.
