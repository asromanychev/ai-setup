# Brief: Оркестрация одного шага синка

## 1. Цель и Проблема

После появления контракта плагина (#1) и персистентных `Source`/`SyncCheckpoint` (#2) в системе всё ещё отсутствует единое прикладное поведение для выполнения одного инкрементального шага синка от имени ядра. Сейчас есть отдельные части процесса, но нет зафиксированного пользовательски наблюдаемого результата для сценария "взять источник, использовать его последнюю позицию возобновления, выполнить один шаг и сохранить новый результат так, чтобы состояние системы осталось согласованным".

Наблюдаемые симптомы:
- Разработчик не может проверить на уровне одного вызова ядра, что после успешного шага синка система перешла из состояния "есть источник и предыдущая позиция" в состояние "есть обновлённая позиция и согласованные последствия шага" без ручного склеивания нескольких низкоуровневых операций.
- При ошибке во время шага синка не зафиксировано единое ожидаемое поведение для уже существующей позиции возобновления: из issue не следует, должна ли система сохранять частичный результат или обязана оставаться в прежнем согласованном состоянии.
- Повторный запуск одного и того же шага после сбоя или ретрая не имеет зафиксированной границы безопасного повтора на уровне ядра, поэтому нельзя однозначно показать, что повтор не ломает состояние источника.

## 2. Для кого

Основной стейкхолдер — разработчик ядра ingestion, который должен связывать контракт плагина с постоянным состоянием источника и проверять поведение одного шага синка без Telegram, сети и ручного вмешательства в БД.

Это также затрагивает автора автотестов и будущую orchestration-часть: им нужен один проверяемый сценарий выполнения шага, на который можно опираться перед добавлением планировщика джоб, API и внешних интеграций.

Приоритет высокий, потому что issue помечен как третий шаг бэклога и напрямую зависит от результатов задач #1 и #2. Без этой границы следующие задачи будут строиться на неявных предположениях о том, когда шаг считается успешно завершённым, а когда состояние должно считаться неизменённым.

## 3. Доменный и Технический контекст

- Домен ingestion в проекте построен вокруг инкрементального обхода источников: каждый вызов обрабатывает один шаг и либо продвигает позицию возобновления, либо завершает шаг без продвижения, если успех не достигнут.
- Контракт плагина из задачи #1 уже определяет наблюдаемое поведение одного шага через входное состояние, набор собранных записей, новую позицию возобновления и признак завершения прохода. Для ядра теперь критично связать этот шаг с внутренним состоянием системы, а не только с изолированным вызовом плагина.
- Задача #2 уже ввела постоянное хранение источников и последней позиции возобновления. Это создаёт предпосылку для следующего шага, но само по себе не задаёт, как ядро должно переводить систему из одного согласованного состояния в другое после вызова плагина.
- В issue используется формулировка "атомарно сохранить новый checkpoint и побочные эффекты". На уровне Problem Space здесь определено только требование согласованности: после завершения одного шага не должно оставаться состояния, в котором позиция возобновления отражает новый шаг, а связанные с этим шагом последствия отражают старый, или наоборот.
- Открытый вопрос, который нельзя однозначно вывести из issue: что именно считается "побочными эффектами" этого шага в первом вертикальном срезе. В текущем brief это трактуется как наблюдаемые изменения внутреннего состояния системы, которые должны быть согласованы с позицией возобновления, но конкретный состав этих изменений ещё не зафиксирован в исходной задаче.
- Вне текущего среза уже явно остаются полное планирование джоб и API endpoints; следовательно, brief не должен опираться на внешние триггеры запуска и описывает только поведение одного вызова внутри ядра.

## 4. Критерий успеха

Задача считается выполненной, когда на уровне поведения системы можно показать следующее:

1. Для зарегистрированного источника с сохранённой позицией возобновления ядро может выполнить один шаг синка, используя эту позицию как входную, и после успешного завершения шага внутренняя позиция источника наблюдаемо соответствует именно результату этого шага.
2. Если шаг завершается ошибкой, система не остаётся в смешанном состоянии, где новая позиция возобновления уже сохранена, а связанные с шагом внутренние изменения не завершены, или наоборот.
3. Повторный запуск одного и того же шага после сбоя не переводит источник в повреждённое или противоречивое состояние: после повтора система остаётся пригодной для следующего корректного шага синка.
4. Happy path и сценарий сбоя воспроизводимы автоматическими тестами без Telegram, сети и внешних API.

## 5. Что не в скоупе

- Полное планирование и расписание фоновых запусков синка.
- Публичные API endpoints или внешний интерфейс управления запуском.
- Интеграция с реальным Telegram-плагином и другими сетевыми источниками.
- Определение продуктовой семантики всех возможных побочных эффектов за пределами внутреннего согласованного состояния, необходимого для одного шага синка.
- Глобальное межзапросное состояние, не относящееся к одному вызову шага синка.
