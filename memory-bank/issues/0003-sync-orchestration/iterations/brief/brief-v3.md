# Brief: Оркестрация одного шага синка

## 1. Цель и Проблема

После появления контракта плагина (#1) и персистентных `Source`/`SyncCheckpoint` (#2) в системе всё ещё отсутствует единое прикладное поведение для выполнения одного инкрементального шага синка от имени системы ingestion. Сейчас есть отдельные части процесса, но нет зафиксированного пользовательски наблюдаемого результата для сценария "взять источник, использовать его последнюю позицию возобновления, выполнить один шаг и сохранить новый результат так, чтобы состояние системы осталось согласованным".

Наблюдаемые симптомы:
- Разработчик не может проверить на уровне одного вызова системы, что после успешного шага синка система перешла из состояния "есть источник и предыдущая позиция" в состояние "есть обновлённая позиция возобновления источника" без ручного склеивания нескольких низкоуровневых операций.
- При ошибке во время шага синка не зафиксировано единое ожидаемое поведение для уже существующей позиции возобновления: из issue не следует, должна ли система сохранять частичный результат или обязана оставлять позицию источника в прежнем состоянии.
- Повторный запуск одного и того же шага после сбоя или ретрая не имеет зафиксированной границы безопасного повтора на уровне системы, поэтому нельзя однозначно показать, что повтор не нарушает согласованность позиции возобновления источника.

## 2. Для кого

Основной стейкхолдер — разработчик ingestion-системы, который должен связать контракт плагина с постоянным состоянием источника и проверять поведение одного шага синка без ручного вмешательства в БД.

Это также затрагивает автора автотестов и будущую orchestration-часть: им нужен один проверяемый сценарий выполнения шага, на который можно опираться перед добавлением планировщика джоб, API и внешних интеграций.

Приоритет высокий, потому что issue помечен как третий шаг бэклога и напрямую зависит от результатов задач #1 и #2. Без этой границы следующие задачи будут строиться на неявных предположениях о том, когда шаг считается успешно завершённым, а когда состояние должно считаться неизменённым.

## 3. Доменный и Технический контекст

- Домен ingestion в проекте построен вокруг инкрементального обхода источников: каждый вызов обрабатывает один шаг и либо продвигает позицию возобновления, либо завершает шаг без продвижения, если успех не достигнут.
- Контракт плагина из задачи #1 уже определяет наблюдаемое поведение одного шага через входное состояние, набор собранных записей, новую позицию возобновления и признак завершения прохода. Для системы теперь критично связать этот шаг с внутренним состоянием источника, а не только с изолированным вызовом плагина.
- Задача #2 уже ввела постоянное хранение источников и последней позиции возобновления. Это создаёт предпосылку для следующего шага, но само по себе не задаёт, как система должна переводить источник из одного согласованного состояния в другое после шага синка.
- В issue используется формулировка "атомарно сохранить новый checkpoint и побочные эффекты". Для текущего вертикального среза в brief фиксируется минимальная проверяемая инварианта: позиция возобновления источника не должна переходить в новое значение, если шаг не завершился успешно. Более широкий состав побочных эффектов в этой итерации не фиксируется.
- Вне текущего среза уже явно остаются полное планирование джоб и API endpoints; следовательно, brief описывает только поведение одного шага синка внутри системы ingestion.

## 4. Критерий успеха

Задача считается выполненной, когда на уровне поведения системы можно показать следующее:

1. Для зарегистрированного источника с сохранённой позицией возобновления система может выполнить один шаг синка, используя эту позицию как входную, и после успешного завершения шага позиция возобновления источника совпадает с позицией, которую плагин зафиксировал как итог успешного шага.
2. Если шаг завершается ошибкой, позиция возобновления источника не обновляется: источник остаётся в том же состоянии, что и до вызова шага.
3. Повторный запуск одного и того же шага после сбоя не нарушает инварианту из критерия 2: источник сохраняет позицию возобновления, не изменённую относительно состояния до сбоя, и остаётся доступным для следующего шага синка.
4. Инварианта перехода состояния источника при успешном шаге и при сбое одинакова и не зависит от конкретного плагина или типа источника данных.

## 5. Что не в скоупе

- Полное планирование и расписание фоновых запусков синка.
- Публичные API endpoints или внешний интерфейс управления запуском.
- Интеграция с реальным Telegram-плагином и другими сетевыми источниками.
- Определение полного состава побочных эффектов шага за пределами инварианты позиции возобновления источника в рамках одного вызова.
- Глобальное межзапросное состояние, не относящееся к одному вызову шага синка.
