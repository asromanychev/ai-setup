# Ревью Brief v5 — Issue #3: Оркестрация одного шага синка

## Замечание 1 (Блокирующее) — Нарушение границы Problem/Solution Space

**Цитата:**
> В issue используется формулировка "атомарно сохранить новый checkpoint и побочные эффекты".

**Проблема:**
Слово "атомарно" описывает механизм реализации — транзакционный принцип. Обёртка "в issue используется формулировка" не выводит это слово из документа: оно присутствует в теле Brief и вводит в Problem Space solution-space термин. Разработчик читает Brief и получает имплицитный сигнал: "нужна транзакция". Это не факт о проблеме, а факт о реализации.

**Как исправить:**
Убрать цитату с "атомарно" или переформулировать на доменный язык. Если цель — зафиксировать инварианту, нужную для понимания контекста, это должно звучать как наблюдаемое доменное поведение:

> Существующее описание в issue подразумевает, что позиция возобновления источника не должна изменяться частично: либо она переходит в новое значение по итогам успешного шага, либо остаётся прежней.

Такая формулировка остаётся в Problem Space и не задаёт механизм достижения.

## Замечание 2 (Блокирующее) — Незакрытый термин "успешный результат"

**Цитата:**
> Если шаг завершается неуспешно (плагин не вернул успешный результат), позиция возобновления источника не обновляется. (КУ 2)

**Проблема:**
В разделе 3 успех определён двусоставно:
> "Шаг считается успешным, если плагин завершился без ошибки **и** вернул позицию возобновления как итог шага."

КУ 2 оперирует термином "успешный результат", который не совпадает с этим определением ни лексически, ни логически. Возникает пограничный случай: плагин завершился без ошибки, но не вернул позицию. По разделу 3 — это не успех. По КУ 2 — неясно: "не вернул успешный результат" — это условие только об (b), или об обоих?

Поскольку на этом термине строится инварианта раздела 4, незакрытость блокирует однозначную проверяемость критерия.

**Как исправить:**
В КУ 2 явно использовать определение из раздела 3:

> Если шаг завершается неуспешно — то есть плагин завершился с ошибкой **или** не вернул позицию возобновления как итог шага — позиция возобновления источника не обновляется.

## Замечание 3 (Среднее) — КУ 4: необоснованное обобщение через неопределённый термин

**Цитата:**
> Инварианта из критериев 1-3 выполняется для любого зарегистрированного плагина и **любого типа источника данных**. (КУ 4)

**Проблема:**
Термин "тип источника данных" нигде в документе не определён. Ни один из трёх симптомов в разделе 1 не фиксирует проблему, специфичную для разных типов источников. КУ 4 вводит новое обобщение без обоснования в Problem Space — это расширяет scope неявно, без соответствующего симптома.

Дополнительно: сама формулировка "для любого зарегистрированного плагина" уже достаточна для обобщения, поскольку плагин и источник данных — связанные концепции в доменном контексте проекта.

**Как исправить:**
Либо удалить часть "и любого типа источника данных":

> Инварианта из критериев 1-3 выполняется для любого зарегистрированного плагина.

Либо, если разнообразие типов источников принципиально, сначала ввести этот термин в разделе 3 и зафиксировать соответствующий симптом в разделе 1.

## Итог

**3 замечания:** 2 блокирующих, 1 среднее. Brief не готов к работе.

Приоритет исправлений:
1. Убрать solution-space термин "атомарно" из раздела 3.
2. Закрыть термин "успешный результат" в КУ 2 через уже существующее определение из раздела 3.
3. Уточнить или убрать "любого типа источника данных" в КУ 4.
