# Cycle Analysis: Issue #0003 — One-Step Sync Orchestration

## 1. Brief

Файл для обновления: `01-1-generate-brief.md`  
Условие (Триггер): если в `Критерии успеха` появляются слова, которые описывают тесты, изоляцию окружения, bootstrap, API, классы, return values или технические механизмы проверки.  
Действие (Что добавить в промпт): перед сохранением прогонять `success-criterion leakage gate`: каждый пункт `Критерия успеха` обязан описывать только наблюдаемое состояние системы или доменный переход; если пункт описывает способ проверки, среду выполнения или инженерный артефакт, переписать его в Problem Space.  
Избегать (Граница/Антипаттерн): запрещено оставлять в Brief формулировки вроде `автотесты`, `без сети/API`, `плагин возвращает`, `bootstrap`, `ядро`, если они не выражены как наблюдаемое системное поведение.  
Обоснование: это предотвращает повторяющийся `solution-artifact leakage`, который гонял brief-v1/v2 через несколько лишних ревью.

Файл для обновления: `01-1-generate-brief.md`  
Условие (Триггер): если критерий успеха использует слова типа `корректный`, `пригодный`, `одинаковый`, `связанные изменения`, `согласованность`, `повреждённый`.  
Действие (Что добавить в промпт): добавить `undefined-term gate`: любой термин из `Критерия успеха` должен либо быть раскрыт через конкретную инварианту в самом Brief, либо удалён до сохранения; запрещено оставлять термин одновременно как критерий и как нераскрытое допущение.  
Избегать (Граница/Антипаттерн): не лечить такие места косметическим rewording; если термин не сводится к наблюдаемому поведению, агент обязан упростить критерий до уже определённой инварианты.  
Обоснование: это убирает затяжной churn brief-v1..v7 вокруг `корректного шага`, `пригодности` и ложной одинаковости success/error переходов.

Файл для обновления: `02-1-review-brief.md`  
Условие (Триггер): если несколько wording-замечаний маскируют одно и то же смысловое противоречие в `Критерии успеха`.  
Действие (Что добавить в промпт): заставить ревьюера сначала сводить замечания к одному корневому semantic defect и запрещать разгонять документ по нескольким wording-правкам, пока не устранён этот дефект.  
Избегать (Граница/Антипаттерн): не выдавать 3-4 мелких замечания на один и тот же незакрытый инвариант, если можно сформулировать один блокирующий boundary defect.  
Обоснование: это уменьшит churn в Brief-цикле, где один смысловой разрыв порождал серию последовательных wording-итераций.

## 2. Spec

Файл для обновления: `01-2-generate-spec.md`  
Условие (Триггер): если спецификация ссылается на runtime-константы, load-path, bootstrap, helpers или иные внешние prerequisites, которые не гарантированы active-артефактами проекта.  
Действие (Что добавить в промпт): перед сохранением выполнять `precondition-hygiene gate`: для каждого внешнего prerequisite агент обязан выбрать ровно одно из двух решений: либо оформить его как явный `Preconditions + stop condition`, либо явно включить минимальный remedy в scope; скрытые prerequisites запрещены.  
Избегать (Граница/Антипаттерн): запрещено одновременно запрещать bootstrap/runtime-change в scope и молча опираться на него в grounding.  
Обоснование: это предотвращает главный межэтапный дефект цикла, когда code-stage уткнулся в `Collect` runtime, не зафиксированный честно на уровне active spec.

Файл для обновления: `01-2-generate-spec.md`  
Условие (Триггер): если FR описывает shape-контракт, duck-typed вход, constructor-time validation или special-case значения (`nil`, `{}`, wrong type, missing key).  
Действие (Что добавить в промпт): добавить `boundary-to-AC expansion gate`: каждый нетривиальный boundary case обязан быть материализован триадой `FR -> error/state scenario -> отдельный AC`; общие корзины вида `invalid result` или `build raises` недостаточны.  
Избегать (Граница/Антипаттерн): не объединять в один AC несколько независимых веток контракта (`missing key`, `wrong type`, `constructor invalid config`, `{}` vs `nil`).  
Обоснование: это устраняет повторяющиеся spec-review замечания про недоматериализованные shape/adversarial cases, которые позже всплывали в Plan.

Файл для обновления: `02-2-review-spec.md`  
Условие (Триггер): если в спеки уже есть общий success/error сценарий, но неясно, материализованы ли все boundary cases входа и результата.  
Действие (Что добавить в промпт): ревью обязано строить таблицу `boundary class -> отдельный scenario/AC` и считать дефектом любую ветку, которая осталась только в общем описании без собственного falsifiable наблюдения.  
Избегать (Граница/Антипаттерн): не засчитывать общий `error`-bucket как покрытие constructor-time invalid config, missing keys и non-Hash field cases.  
Обоснование: это сокращает количество поздних plan-fix итераций, вызванных тем, что спека была формально хорошей, но не fully materialized.

## 3. Plan

Файл для обновления: `01-3-generate-plan.md`  
Условие (Триггер): если шаг плана использует shell pipeline или числовой gate (`0 pending`, число examples, dry-run count) как prerequisite/final verification.  
Действие (Что добавить в промпт): ввести `observable gate rule`: любой observational или final gate обязан отдельно проверять, что runtime/suite вообще успешно загрузился (`errors_outside_of_examples_count == 0`, boot success, exit semantics), и только потом считать производные метрики вроде pending/examples/count.  
Избегать (Граница/Антипаттерн): запрещено использовать pipeline, который может вернуть валидный счётчик при сломанном boot или при status только последней команды.  
Обоснование: это предотвращает многократный churn plan-v1..v10 вокруг ложных `0 pending` и ложных counts при не поднявшемся suite.

Файл для обновления: `01-3-generate-plan.md`  
Условие (Триггер): если план требует проверить инвариант rollback, изоляцию переменной, одноразовый вызов, duck-typed независимость или иной нетривиальный behavioural invariant.  
Действие (Что добавить в промпт): обязать план делать для таких кейсов `test materialization note`, где явно фиксируются: наблюдаемый rollback boundary, единственная меняющаяся переменная, стартовое состояние, и почему тест не может пройти по случайной эквивалентности.  
Избегать (Граница/Антипаттерн): не оставлять сценарии уровня `stub method and raise`, `run twice on same source`, `one generic missing key case`, если они не доказывают сам инвариант.  
Обоснование: это убирает повторяющиеся plan-review дефекты по savepoint rollback, AC 17 duck-typing и неполной материализации AC 10/`checkpoint_out`.

Файл для обновления: `02-3-review-plan.md`  
Условие (Триггер): если шаг плана формально есть, но его pass/fail неотличим от соседнего runtime-фейла или от неполной матрицы тестов.  
Действие (Что добавить в промпт): ревью должно строить проверку `step goal -> observable signal -> unique failure mode`; если сигнал не уникален, шаг считается дефектным даже при формально правильной команде.  
Избегать (Граница/Антипаттерн): не одобрять шаг, который путает `suite not loaded` с `0 pending`, или `same source second run` с проверкой независимости от класса плагина.  
Обоснование: это переводит review из перечисления команд в проверку наблюдаемости и резко сокращает plan churn.

## 4. Code

Файл для обновления: `01-4-generate-code.md`  
Условие (Триггер): если перед началом реализации в active spec/plan есть prerequisite или stop condition, завязанные на runtime boot, test DB, load-path, coverage/pending gate.  
Действие (Что добавить в промпт): сделать `pre-edit runtime gate` обязательным: до первой правки агент должен либо материализовать prerequisite проверкой, либо остановиться с `implementation blocker`; запрещено начинать код и особенно тестовые обходные правки, пока prerequisite не подтверждён или не эскалирован.  
Избегать (Граница/Антипаттерн): не лечить неподтверждённый prerequisite внеплановыми `.env`, `database.yml`, boot-файлами или иными infra-изменениями без явной scope-эскалации.  
Обоснование: это предотвращает кодовый churn, где позднее пришлось убирать scope breaches и только потом честно фиксировать минимальный bootstrap.

Файл для обновления: `03-4-fix-code.md`  
Условие (Триггер): если последний code review содержит только `scope breach` или `runtime-unknown gate`, а доказуемого функционального дефекта нет.  
Действие (Что добавить в промпт): fix-step обязан сначала классифицировать замечания по типу и действовать по разным путям: `scope breach` лечить минимизацией change set, `runtime-unknown gate` закрывать исполнением/эскалацией, а не переписыванием бизнес-логики.  
Избегать (Граница/Антипаттерн): не продолжать кодовые правки по инерции, если review уже говорит `Эквивалентно`, а открыты только environment/scope issues.  
Обоснование: это сокращает лишние fix-итерации, когда проблема была не в `sync_step!`, а в allowlist и непроверенных gates.

Файл для обновления: `02-4-review-code.md`  
Условие (Триггер): если review выявляет, что active spec/plan сами не совпадают с фактически необходимым минимальным runtime remedy.  
Действие (Что добавить в промпт): обязать ревьюера явно различать два случая: `illegal workaround` и `legitimate missing scope/precondition`; если remedy минимален, фактически нужен для прохождения declared prerequisite и не меняет доменную логику, review должен рекомендовать вернуть цикл на обновление active spec/plan, а не только копить `scope breach` на code-step.  
Избегать (Граница/Антипаттерн): не оставлять цикл в состоянии бесконечной борьбы между корректным кодом и заведомо устаревшими active-документами.  
Обоснование: это предотвращает поздний churn между code-review-7 и фактическим исправлением active spec/plan.

## 5. Кросс-цикловая оптимизация

Файл для обновления: `workflow/system gate`
Условие (Триггер): перед переходом `Spec -> Plan` и `Plan -> Code` для задач, завязанных на runtime, boot, DB или suite gates.
Действие (Что добавить в промпт): ввести общий `materialized prerequisite gate`: если документ содержит prerequisite, он должен быть либо подтверждён наблюдаемой командой до следующей стадии, либо оформлен как внешний blocker с stop condition; скрытые prerequisites запрещены.
Избегать (Граница/Антипаттерн): не переносить в следующий этап требования, которые существуют только как текстовое допущение без materialized evidence или явной эскалации.
Обоснование: это срежет целый класс поздних `03-fix`, вызванных не кодом, а невалидными предпосылками.

Файл для обновления: `workflow/system gate`
Условие (Триггер): при любом финальном gate, который опирается на suite counts, pending, coverage или example counts.
Действие (Что добавить в промпт): обязать все такие gates иметь двухфазную структуру `boot/load success -> metric threshold`, а review и fix должны использовать ту же структуру без сокращений.
Избегать (Граница/Антипаттерн): не считать метрику доказательством готовности, если не доказана корректная загрузка того, из чего эта метрика получена.
Обоснование: это уменьшит churn одновременно в Plan и Code, потому что ложноположительные и ложноотрицательные gates исчезнут на уровне workflow.
