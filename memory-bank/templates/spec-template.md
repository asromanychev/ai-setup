# Шаблон и требования к Specification (Спецификации)

**Ключевое правило:** Спецификация — это дизайн решения. Она переводит проблему в четкие технические контракты. Документ должен быть меньше 1500 слов и затрагивать не более 3 модулей системы.

### Rubric Gate
Для Spec обязана существовать отдельная rubric/scorecard-оценка по 5 измерениям:
- `testability`
- `ambiguity_free`
- `state_uniformity`
- `precondition_hygiene`
- `grounding_realism`

При активации Spec не допускаются измерения со `score <= 2`.

### [Название фичи]

#### 1. Цель решения
Одно предложение: как технически мы решаем заявленную в Брифе проблему.

#### 1.1 Потребитель решения (Who / When)
Опишите, кто вызывает эту функциональность, в каком порядке и с какой частотой:
- **Кто вызывает:** роль, компонент или внешняя система (`Sidekiq job`, `HTTP endpoint`, `CLI`, `другой плагин`).
- **Когда вызывается:** триггер (`по расписанию`, `на событие`, `по запросу пользователя`, `при старте приложения`).
- **Ожидаемая частота / объём:** примерное количество вызовов или объём данных (`~100 сообщений/мин`, `однократно при инициализации`).
- **Ожидаемая модель ответа:** синхронный результат, фоновая задача, webhook, side-effect в БД.

#### 2. Scope (Границы решения)

Используй стабильные идентификаторы для трассировки требований:

* `REQ-1` [Конкретный функционал, который реализуется]
* `REQ-2` [Ещё один обязательный capability]
* `NS-1` НЕ входит: [Связанные вещи, которые реализовывать запрещено]
* `NS-2` НЕ входит: [Ещё одно ограничение скоупа]

#### 2.1 Preconditions
Если реализация зависит от внешнего prerequisite, который ещё не материализован в проекте, перечислите его отдельно с явным stop condition.

#### 3. Функциональные требования и Критерии TAUS
*В этом разделе не должно быть слов «быстро», «удобно», «и т.д.», «при необходимости».*
1. [Однозначное требование 1]
2. [Однозначное требование 2]

#### 4. Uniform (Единообразие и полнота состояний)
Опишите все возможные состояния (success, loading, error, empty).
* **Сценарий ошибки 1 (невалидные данные):** [Ожидаемое поведение]
* **Сценарий ошибки 2 (нет прав/сети):** [Ожидаемое поведение]
* **Edge cases:** [Граничные значения, пустые списки]

#### 5. Инварианты
Свойства системы, которые ДОЛЖНЫ сохраняться после ваших изменений (например: «Ни один запрос не должен обходить middleware авторизации»).

#### 6. Ограничения на реализацию и Grounding
Укажите конкретные пути к существующим файлам, которые нужно будет изменить. Укажите, какие архитектурные паттерны проекта использовать.

#### 7. Acceptance Criteria (AC) / Exit Criteria

*AC описывают поведение, а не детали кода. По каждому SC можно написать автотест. Задача не завершена, пока все CHK не pass.*

**Acceptance Scenarios:**
* `SC-1` [Основной позитивный сценарий — end-to-end]
* `SC-2` [Негативный или edge-case сценарий]

**Traceability:**
| SC | Покрывает REQ |
|----|---------------|
| SC-1 | REQ-1 |
| SC-2 | REQ-2 |

**Checks & Evidence:**
* `CHK-1` [ ] [Проверяемое условие — команда или observable fact]
* `CHK-2` [ ] Все существующие тесты остаются зелёными (`bin/ci`)
* `EVID-1` [Carrier: путь к spec-файлу / CI run / вывод команды]
