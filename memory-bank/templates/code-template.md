# Правила кодогенерации и Verification Checklist

**Ключевое правило:** Код является прямым производным от Спецификации и Плана. Агенту запрещено проявлять самодеятельность, выходящую за рамки этих документов.

### Rubric Gate
Для Code обязана существовать отдельная rubric/scorecard-оценка по 5 измерениям:
- `scope_compliance`
- `invariant_preservation`
- `test_adequacy`
- `runtime_readiness`
- `regression_risk`

Если любой из этих пунктов получает `score <= 2`, код не считается готовым.

### Часть 1. Требования к процессу (Implement)
1. **Curated Context:** Следуйте глобальным правилам из файла `CLAUDE.md`. Изменяйте только файлы, строго указанные в Плане реализации.
2. **Анти-Gold Plating:** Запрещено писать код «на будущее», добавлять лишние методы или рефакторить соседний код, не относящийся к задаче напрямую.
3. **Безопасность:** Проверяйте входные данные, не хардкодьте секреты, защищайтесь от типичных уязвимостей (SQLi, XSS).

### Часть 2. Agent-first Code Review (Semi-Formal Reasoning)
*До передачи кода на ревью человеку, агент-исполнитель обязан проанализировать свой код и ответить на следующие вопросы:*

1. **Инварианты и контракты:** Сохраняются ли ключевые свойства системы? Не нарушен ли чужой контракт?
2. **Трассировка путей выполнения:** Пройден ли happy-path? Корректно ли отрабатывают все указанные в спеке error-path?
3. **Риски и регрессии:** Назовите возможные риски вашего изменения (severity: high/medium/low) и укажите конкретную строку кода.
4. **Вердикт эквивалентности:** Выполнены ли все Acceptance Criteria на 100%? Если нет — требуется доработка кода до завершения сессии.
5. **Тесты:** Покрывают ли добавленные тесты бизнес-поведение из спеки (а не внутреннюю реализацию методов)? Будут ли они зелеными?
