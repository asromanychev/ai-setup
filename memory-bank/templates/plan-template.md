# Шаблон и требования к Implementation Plan (Плану реализации)

**Ключевое правило:** План декомпозирует Спецификацию на конкретные, атомарные шаги реализации. Ни один шаг не должен требовать от агента догадок.

### Rubric Gate
Для Plan обязана существовать отдельная rubric/scorecard-оценка по 5 измерениям:
- `dependency_order`
- `step_atomicity`
- `delta_specificity`
- `runtime_gate_quality`
- `test_materialization`

При активации Plan не допускаются измерения со `score <= 2`.

#### 1. Паттерн оркестрации
Укажите, как задача будет выполняться: один агент последовательно или декомпозиция на нескольких параллельных агентов.

#### 1.5 Preconditions

* `PRE-1` [Внешнее условие, которое должно выполниться до начала — миграция, зависимость, merged PR]
* `PRE-2` [Ещё одно предусловие или «нет предусловий»]

#### 2. Grounding (Проверка привязки к коду)
* [ ] Подтверждено, что все файлы, упомянутые в плане, реально существуют в кодовой базе проекта.
* [ ] План совместим с текущей архитектурой (нет конфликтов).
* [ ] План технически осуществим в текущем стеке.

#### 3. Граф зависимостей (When)
Визуализируйте порядок шагов и жёсткие зависимости между ними. Используйте нотацию: `Шаг N -> Шаг M` означает, что M нельзя начать до завершения N.

```
Шаг 1 -> Шаг 2 -> Шаг 4
Шаг 3 -> Шаг 4
Шаг 4 -> Шаг 5 (финальная верификация)
```

Если шаги независимы и могут идти параллельно — укажите это явно.

#### 4. Пошаговый план (Атомарные шаги)
*Каждый шаг должен затрагивать конкретный файл или модуль. Шаги отсортированы по жестким зависимостям (например: сначала миграция БД, потом логика).*

Формат каждого шага — используй стабильные идентификаторы `STEP-*`:

```
* `STEP-N` [S|M|L]: [Файл / Модуль] -> [Конкретное действие]
  - Implements: REQ-X (из spec)
  - Depends on: STEP-X или «нет зависимостей»
  - Rollback: [Как отменить] или «нет side-effects»
  - Signal: [Наблюдаемый pass/fail — команда или факт]
```

Маркер сложности: `S` — до 30 мин, `M` — до 2 часов, `L` — полдня и более.

* `STEP-1` [S]: [Файл / Модуль] -> [Конкретное действие]
  - Implements: REQ-1
  - Depends on: нет зависимостей
  - Rollback: [описание]
  - Signal: [команда pass/fail]
* `STEP-2` [M]: [Файл / Модуль] -> [Конкретное действие]
  - Implements: REQ-2
  - Depends on: STEP-1
  - Rollback: [описание]
  - Signal: [команда pass/fail]

#### 5. План тестирования
Укажите, в каких файлах должны быть добавлены Unit и Integration тесты, чтобы покрыть все Acceptance Criteria из Спецификации.

#### 6. Финальный Runtime Gate

Последний шаг плана обязан быть чистым runtime gate — только команды и criteria:

* `CHK-1` `bundle exec rspec [путь к spec]` → все зелёные
* `CHK-2` `bundle exec rubocop [затронутые файлы]` → no offenses
* `CHK-3` `bin/ci` → зелёный

* `EVID-1` [Carrier: CI run URL / вывод команды / путь к файлу]

Без дополнительных правок «по месту», если команда упала — стоп и диагностика.
