# Инструкция для AI-агента (Task Definition): Генерация Spec

Пользователь инициировал создание Спецификации для конкретного Git Issue. Твоя задача — самостоятельно собрать контекст, прочитать утвержденный Brief и сгенерировать первую draft-версию Спецификации.

## Шаг 1. Сбор контекста (Grounding on Memory Bank)
1. Извлеки номер Issue из запроса пользователя.
2. Сформируй 4-значный префикс: `1 -> 0001`.
3. Просканируй директорию `memory-bank/issues/` и найди папку, которая начинается с этого префикса, например `0001-task-name`.
4. Перейди в эту папку. Это твой рабочий контекст.
5. Прочитай утвержденный файл `brief.md` в корне папки issue.
6. Если `brief.md` отсутствует, остановись и сообщи пользователю, что Brief ещё не утвержден. Не используй draft-файлы из `iterations/` как замену active-документу.

## Шаг 2. Изучение правил
1. Прочитай `.claude/CLAUDE.md`.
2. Прочитай `.codex/CODEX.md`, если файл существует.
3. Прочитай шаблон `memory-bank/templates/spec-template.md`, если он существует.
4. Прочитай `memory-bank/templates/rubric-template.json`, если файл существует.
5. Кратко изучи кодовую базу и существующие файлы проекта, которые относятся к проблеме из Brief. Решение должно быть реализуемо в текущей архитектуре.

## Факторы выполнения
- execution metadata: фиксируй `system`, `model`, `provider`, `execution_date`, `prompt_id`; если среда не раскрывает значение, записывай `unknown`.
- telemetry capture: фиксируй `started_at`, `finished_at`, `elapsed_seconds`; если runtime отдает usage, добавляй `input_tokens`, `output_tokens`, `total_tokens`; если billing или лимиты недоступны, записывай `not available in current runtime`.
- version overwrite: отключен; существующие версии draft и review не перезаписываются.

## Шаг 3. Выполнение задачи (Solution Space)
0. Если `iterations/spec/spec-rubric.json` ещё не существует, создай его.
   Рубрика должна содержать ровно 5 измерений:
   - `testability`
   - `ambiguity_free`
   - `state_uniformity`
   - `precondition_hygiene`
   - `grounding_realism`
   Сумма весов должна быть `1.0`.
На основе `brief.md` и кодовой базы создай Спецификацию решения. Она должна строго соответствовать критериям TAUS:
- T (Testable): есть проверяемые Acceptance Criteria, по которым можно написать автотесты.
- A (Ambiguous-free): нет двусмысленных слов вроде «быстро», «удобно», «и т.д.», «по возможности».
- U (Uniform): описаны все состояния (success, error, loading, empty), сценарии ошибок и edge cases.
- S (Scoped): это ровно одна фича, текст меньше 1500 слов.

Выведи спецификацию по следующей структуре:
1. Цель решения:
   как технически решается проблема из Brief.
2. Scope:
   что входит в реализацию и что не входит.
3. Функциональные требования:
   однозначные и конкретные.
4. Сценарии ошибок и состояния:
   детальное описание поведения при невалидных данных, отсутствии прав, пустых списках и других особых состояниях.
5. Инварианты:
   ключевые свойства системы, которые должны сохраняться после изменений.
6. Grounding (ограничения на реализацию):
   укажи пути к конкретным существующим файлам проекта, которые придётся затронуть; укажи паттерны, которые нужно использовать.
7. Acceptance Criteria (AC):
   чеклист проверяемых условий. Обязательно добавь пункты:
   - все существующие тесты остаются зелёными;
   - инварианты не нарушены.

Дополнительные жёсткие правила:
- Если в Grounding, Scope, Functional Requirements, тестовой архитектуре или AC появляется новый helper-файл, gem-зависимость, bootstrap/load-path механизм, внешний error-класс или инфраструктурный модуль вне основной фичи, сначала выполни grounding-check:
  - файл реально существует в репозитории;
  - зависимость уже есть в `Gemfile`/lock-файле;
  - механизм загрузки или bootstrap уже материализован в проекте.
- Precondition-hygiene gate:
  - если реализация зависит от внешнего runtime prerequisite, который сейчас не материализован в проекте, не описывай его как часть текущего Scope или Grounding;
  - такой prerequisite обязан быть вынесен в отдельный раздел `Preconditions` с явным условием остановки и признаком, что без него текущая спека неисполняема;
  - запрещено маскировать prerequisite формулировками вроде «добавить по пути реализации», если это отдельная инфраструктурная работа.
- Runtime-branch split gate:
  - если в API, factory, bootstrap или runtime-path появляется ветка, завязанная на `ENV`, secrets, client construction, load-path или другой внешний runtime, до сохранения спеки выбери один из двух вариантов:
    - либо эта ветка полностью grounded в текущем issue через файлы, FR, AC и допустимый scope;
    - либо она явно вынесена в `Preconditions`, `Out of scope` или отдельный следующий issue;
  - третьего состояния нет: запрещено одновременно держать такую ветку в scope и писать, что соответствующий bootstrap/env wiring не меняется.
- Если grounding-check не пройден, нельзя молча включать этот элемент в реализацию текущей фичи.
  Варианты допустимы только такие:
  - оформить элемент как явный `Precondition`;
  - вынести его в отдельный issue;
  - сузить спецификацию до уже существующих модульных зон.
- Boundary-to-AC expansion gate:
  - если Functional Requirements вводят структурный контракт, доменный режим или гарантию уровня `determinism`, `normalization`, `deep copy`, `isolation`, `idempotence`, для него обязаны появиться explicit AC или error scenarios, которые проверяют boundary conditions этого контракта;
  - нельзя оставлять boundary cases только в prose-описании FR без последующей материализации в `Сценариях ошибок и состояний` или `Acceptance Criteria`.
- Не описывай `database_cleaner`, интеграцию через `config/application.rb`, дополнительные load-path, `core/lib`-подключение или внешние исключения как часть текущей фичи без явного подтверждения, что они уже существуют и входят в scope.
- Для каждого инварианта про копирование, иммутабельность, изоляцию, нормализацию или детерминизм добавь минимум один adversarial edge case, который пытается сломать эту гарантию через внешнюю мутацию, aliasing, повторный вызов или нестандартную структуру данных.
- Не ограничивай edge cases только типовыми сценариями уровня `nil`, `string`, `empty`.
- Если какое-то состояние из TAUS (`loading`, `empty`, `in progress`) неприменимо, перечисли его явно и объясни, почему оно отсутствует в этой модели выполнения.
- Если в Functional Requirements появляется инвариант или гарантия уровня `deep copy`, `freeze`, `normalization`, `determinism`, `isolation`, `idempotence`, для него сразу должен появиться хотя бы один falsifiable adversarial scenario в `Сценариях ошибок и состояний` или в `Acceptance Criteria`.
- Для структурных контрактов проверяй не только значения, но и boundary cases на уровне формы данных: мутабельные ключи Hash, aliasing между входом и внутренним состоянием, различие `{}` и `nil`, constructor-time validation против runtime validation.
- Bulk-insert validation bypass gate:
  - если FR описывает idempotent insert семантику через bulk-операцию (`insert`, `upsert`, `INSERT ... ON CONFLICT`), добавить в FR явное разграничение двух уровней защиты:
    - AR validations, которые проверяются до bulk-операции (и которые bulk-операция НЕ вызывает автоматически);
    - DB constraints, которые обеспечиваются индексами или NOT NULL на уровне схемы;
  - для каждого AR validation, который не вызывается автоматически при bulk-insert, добавить falsifiable AC или error scenario, который проверяет именно этот случай;
  - запрещено описывать `validates :X, presence: true` как достаточную защиту для bulk-insert пути без явного упоминания, что validation должен быть вызван вручную перед вставкой.

## Шаг 4. Финализация и Сохранение
1. Сохрани `spec-rubric.json` в `iterations/spec/`, если файла ещё нет.
2. Сохрани draft-версию в подпапку `iterations/spec/` текущей папки issue под именем `spec-v1.md`, если это первая версия.
3. Если `spec-v1.md` уже существует, создай следующую версию `spec-v{N+1}.md`.
4. Не записывай результат в корень папки issue как `spec.md` на этом шаге: это draft.
5. В финальном ответе сообщи:
   - путь к созданному файлу;
   - номер версии;
   - была ли создана `spec-rubric.json`;
   - какие реальные файлы проекта использовались для grounding;
   - блок `Execution Metadata` со значениями `system`, `model`, `provider`, `execution_date`, `prompt_id`;
   - блок `Runtime Telemetry` со значениями `started_at`, `finished_at`, `elapsed_seconds`, `input_tokens`, `output_tokens`, `total_tokens`, `estimated_cost`, `limit_context`, где недоступные поля помечены как `unknown` или `not available in current runtime`.
