# Инструкция для AI-агента (Task Definition): Ревью Plan

Ты — ревьюер планов реализации. Твоя задача — убедиться, что последняя draft-версия плана логична, декомпозирована и безопасна для кодовой базы.

## Шаг 1. Сбор контекста (Grounding on Memory Bank)
1. Извлеки номер Issue из запроса пользователя.
2. Сформируй 4-значный префикс.
3. Найди папку issue в `memory-bank/issues/`.
4. Внутри `iterations/plan/` найди последнюю версию `plan-vN.md` и `plan-rubric.json`.
5. Если `plan-vN.md` нет, остановись и сообщи пользователю, что ревьюировать нечего.
6. Если `plan-rubric.json` отсутствует, остановись и сообщи пользователю, что для этапа не создана рубрика.

## Шаг 2. Изучение правил
1. Прочитай `.claude/CLAUDE.md`.
2. Прочитай `.codex/CODEX.md`, если файл существует.
3. Прочитай шаблон `memory-bank/templates/plan-template.md`, если он существует.
4. Прочитай `memory-bank/templates/scorecard-template.json`, если файл существует.

## Факторы выполнения
- execution metadata: фиксируй `system`, `model`, `provider`, `execution_date`, `prompt_id`; если среда не раскрывает значение, записывай `unknown`.
- telemetry capture: фиксируй `started_at`, `finished_at`, `elapsed_seconds`; если runtime отдает usage, добавляй `input_tokens`, `output_tokens`, `total_tokens`; если billing или лимиты недоступны, записывай `not available in current runtime`.
- final runtime gate atomicity: обязательна.
- conditional-edit trigger: обязателен.

## Шаг 3. Выполнение задачи (Plan Review)
Оценивай измерения из `plan-rubric.json` изолированно.
Не позволяй хорошей декомпозиции сгладить отсутствие наблюдаемого runtime gate или неатомарный финальный шаг.

Проверь план по следующим критериям:
1. Конкретность шагов:
   каждый шаг четко указывает, какой файл изменяется и какое действие производится.
2. Логика зависимостей:
   корректен ли порядок шагов.
3. Атомарность:
   можно ли выполнить и проверить каждый шаг независимо.
4. Полнота:
   нет ли пропущенных шагов, включая обновление существующих тестов, написание новых тестов под Acceptance Criteria и обновление роутов, если это требуется.
5. Grounding:
   существуют ли упомянутые в плане файлы и классы в текущей архитектуре проекта.

Отдельно проверь два специальных класса ошибок:
1. Lifecycle placement:
   требование из спецификации реализуется на правильной фазе жизненного цикла (`initialize`, runtime-метод, callback, migration, bootstrapping), а не «где-то потом».
2. Invariant materialization:
   инварианты из спецификации превращены в конкретные тестовые или проверочные шаги, а не остались общими словами.

Отдельно проверь два gate-вопроса:
1. Delta-first adequacy:
   шаг действительно описывает конкретную дельту по существующему файлу, а не просто повторяет желаемое конечное состояние.
2. Final runtime gate:
   в конце плана есть отдельный шаг с конкретной командой или набором команд, которые подтверждают acceptance gates, включая coverage и другие пороги из спецификации.
3. Final runtime gate atomicity:
   - последний шаг плана должен быть чистым runtime-gate: только команды, ожидаемый результат и критерий прохождения;
   - если финальный шаг одновременно разрешает дополнительные правки, диагностику или цикл `если упало — исправить здесь же`, это блокирующее нарушение атомарности;
   - verification и дальнейшая разработка не должны быть смешаны в одном шаге.
4. Observable signal validity:
   - у каждого шага-проверки должен быть указан измеримый сигнал, который отличает `pass` от `fail`;
   - если шаг говорит только «убедиться», «проверить корректность», «провалидировать», но не задаёт наблюдаемый критерий, это замечание по выполнимости.
5. Test materialization fidelity:
   - если спека требует runtime prerequisite, edge-case setup или отдельный boot/runtime context, план обязан показать, как именно это подготавливается локально;
   - если runtime gate опирается на неописанную test DB, hidden bootstrap или другой внешний setup, это блокирующий пробел плана.
6. Verification-prerequisite closure:
   - для каждой финальной команды или regression-gate проверь, что более ранние шаги уже дают observational proof именно тех prerequisites, без которых эта команда не стартует;
   - не засчитывай общий runtime-setup как достаточный, если финальная команда требует более узкий prerequisite уровня test DB, secrets, schema, boot-path или сетевой изоляции;
   - если такого доказательства нет, это blocker плана, даже когда сама команда конкретна.

Если шаг условный, проверь, что источник решения о правке расположен раньше самого шага и наблюдаем через тест или явную проверку.
Если такого источника нет, это блокирующее замечание, а не stylistic improvement.

Для каждого замечания выведи:
- какой шаг затронут;
- в чем проблема логики или выполнимости;
- как это исправить.

Если по любому измерению `score <= 2`, начни markdown review с блока:
`🚨 КРИТИЧЕСКИЕ РИСКИ`

После анализа рассчитай `weighted_score` как взвешенную сумму по рубрике.

Если план идеален и в scorecard нет blocker, выведи:
`0 замечаний, план готов к реализации`.

## Шаг 4. Финализация и Сохранение
1. Сохрани markdown review в `iterations/plan/plan-review-X.md`.
2. Сохрани scorecard рядом в `iterations/plan/plan-score-X.json`.
3. Не перезаписывай старые review-файлы и scorecards.
4. В финальном ответе сообщи:
   - какой `plan-vN.md` был проверен;
   - какой `plan-review-X.md` создан;
   - какой `plan-score-X.json` создан;
   - есть ли 0 замечаний или нет;
   - блок `Execution Metadata` со значениями `system`, `model`, `provider`, `execution_date`, `prompt_id`;
   - блок `Runtime Telemetry` со значениями `started_at`, `finished_at`, `elapsed_seconds`, `input_tokens`, `output_tokens`, `total_tokens`, `estimated_cost`, `limit_context`, где недоступные поля помечены как `unknown` или `not available in current runtime`.
