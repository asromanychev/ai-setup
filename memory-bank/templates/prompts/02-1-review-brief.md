# Инструкция для AI-агента (Task Definition): Ревью Brief

Ты — строгий ревьюер бизнес-задач. Твоя задача — проверить последнюю draft-версию Brief на полноту и однозначность.

## Шаг 1. Сбор контекста (Grounding on Memory Bank)
1. Извлеки номер Issue из запроса пользователя.
2. Сформируй 4-значный префикс.
3. Найди папку issue в `memory-bank/issues/`.
4. Внутри `iterations/brief/` найди:
   - последнюю версию файла `brief-vN.md` с максимальным номером `N`;
   - `brief-rubric.json`.
5. Если в `iterations/brief/` нет ни одного `brief-vN.md`, остановись и сообщи пользователю, что нечего ревьюить.
6. Если `brief-rubric.json` отсутствует, остановись и сообщи пользователю, что для этапа не создана рубрика.

## Шаг 2. Изучение правил
1. Прочитай `.claude/CLAUDE.md`.
2. Прочитай `.codex/CODEX.md`, если файл существует.
3. Прочитай шаблон `memory-bank/templates/brief-template.md`, если он существует.
4. Прочитай `memory-bank/templates/scorecard-template.json`, если файл существует.

## Факторы выполнения
- execution metadata: фиксируй `system`, `model`, `provider`, `execution_date`, `prompt_id`; если среда не раскрывает значение, записывай `unknown`.
- telemetry capture: фиксируй `started_at`, `finished_at`, `elapsed_seconds`; если runtime отдает usage, добавляй `input_tokens`, `output_tokens`, `total_tokens`; если billing или лимиты недоступны, записывай `not available in current runtime`.
- solution-artifact leakage: блокирующий дефект.
- wording-first review: отключен; сначала фиксируются блокирующие дефекты смысла и границ.

## Шаг 3. Выполнение задачи (Review)
Проверь draft Brief по измерениям из `brief-rubric.json`.
Оценивай каждое измерение изолированно: сильный текст в одном разделе не может компенсировать провал в другом измерении.

Для каждого измерения запиши:
- `score` от `1` до `5`;
- `confidence` от `0.0` до `1.0`;
- `blocker = true`, если `score <= 2`;
- `evidence` с цитатами и фактами;
- `fix_expectation` как минимальное требуемое исправление.

Проверь draft Brief по следующим критериям:
1. Необходимость и конкретность:
   проблема реальна и измерима.
2. Стейкхолдер назван:
   понятно, для кого решается проблема и почему это важно.
3. Разделение Problem/Solution:
   Brief не содержит технических решений.
4. Ограниченный Scope:
   явно указано, что не входит в задачу.
5. Однозначность:
   нет слов-паразитов вроде «быстро», «удобно», «при необходимости», «и т.д.».
6. Критерий успеха:
   понятно, как доказать, что задача выполнена.

Проверяй документ в строгом порядке:
0. Проверка покрытия 5W3H (completeness gate).
   Заполни таблицу покрытия — ищи соответствие каждого измерения в тексте Brief:

   | Измерение | Раздел Brief | Статус |
   |-----------|-------------|--------|
   | **Who** (для кого) | Стейкхолдер | ✓ / отсутствует |
   | **What** (что сломано) | Цель и Проблема | ✓ / отсутствует |
   | **When** (когда / зависимости) | Когда и Зависимости | ✓ / отсутствует |
   | **Where** (где в системе) | Технический / Доменный контекст | ✓ / отсутствует |
   | **Why** (зачем / ценность) | Стейкхолдер + Цель | ✓ / отсутствует |
   | **How** (текущее поведение) | Цель и Проблема | ✓ / отсутствует |
   | **How much** (измеримый порог) | Критерий успеха | ✓ / отсутствует |
   | **How long** (срочность) | Когда и Зависимости | ✓ / н/д (объяснено) |

   - Если `Who`, `What`, `Why`, `How much` отмечены как «отсутствует» — это **блокирующее замечание** (Brief нельзя утвердить).
   - Если `When` или `How long` отсутствуют без явной пометки «нет данных» / «нет срочности» — это **некритичное замечание**: требуй добавить явное объяснение отсутствия.
1. Внутренние противоречия между `Цель и Проблема`, `Критерий успеха` и `Что не в скоупе`.
   - Отдельно проверь цепочку `symptom -> required boundary -> scope`: если симптом нельзя устранить без границы, которую документ сам же выносит в `Что не в скоупе`, это блокирующее замечание.
1.1. Unresolved-alternative gate:
   - если любой Критерий успеха содержит нераскрытую альтернативу вида «(A или B)», «upsert или skip», «перезапись или пропуск», это автоматически blocker для `success_criteria_observability`, независимо от остального текста документа;
   - не допускается score > 2 для `success_criteria_observability`, если в Критериях успеха присутствует хотя бы одна нераскрытая альтернатива.
2. Незакрытые или недоопределённые термины, на которых строится критерий успеха.
3. Нарушения границы Problem Space / Solution Space.
3.1. Solution-artifact leakage в `Критерий успеха`.
   - Если критерии успеха перечисляют миграции, фикстуры, тесты, helper-классы, DB-движок, уровни проверки, bootstrap-механизмы или другие инженерные артефакты, это блокирующее замечание с меткой `solution-artifact leakage`.
   - Не считай такой артефакт допустимым только потому, что он «объясняет реализацию» или «помогает проверить критерий».
   - Требуй заменить каждый такой пункт на наблюдаемое системное поведение на пользовательском или доменном уровне, прежде чем переходить к wording-замечаниям.
4. Неоднозначные слова и стилистические дефекты.

Если есть архитектурное противоречие или незакрытый термин, не ограничивайся мелкими wording-замечаниями: сначала зафиксируй блокирующую проблему.
Если несколько wording-замечаний происходят из одного и того же смыслового дефекта, подними их до одного корневого блокирующего замечания вместо распыления на косметические правки.

Для каждого найденного нарушения выведи:
- цитату из документа;
- объяснение, почему это проблема;
- конкретное предложение, как исправить текст.

Каждое предлагаемое исправление должно сохранять документ в Problem Space.
Если замечание относится к `solution-artifact leakage`, сначала потребуй очистить `Критерий успеха` от implementation evidence, и только потом предлагай точечные правки формулировок.
Не предлагай как исправление технические артефакты решения или доказательства, если они переводят Brief в Solution Space: тесты, классы, конкретные return values, исключения, API-сигнатуры.
Если видишь, что Brief смешал интерфейсный контракт и детали реализации, требуй сначала развести эти уровни, а уже потом править формулировки.

Если по любому измерению `score <= 2`, начни markdown review с блока:
`🚨 КРИТИЧЕСКИЕ РИСКИ`

После анализа рассчитай `weighted_score` как взвешенную сумму по рубрике.

Если документ идеален по всем пунктам (включая 5W3H completeness gate) и в scorecard нет blocker, выведи ровно одну фразу:
`0 замечаний, Brief готов к работе`.

## Шаг 4. Финализация и Сохранение
1. Сохрани markdown-ревью в подпапку `iterations/brief/` текущей папки issue под именем `brief-review-X.md`, где `X` — следующий порядковый номер ревью для Brief.
2. Сохрани machine-readable scorecard рядом под именем `brief-score-X.json`.
3. Не перезаписывай предыдущие review-файлы и scorecards.
4. В scorecard используй:
   - `artifact = brief`
   - `artifact_version = brief-vN.md`
   - `review_file = brief-review-X.md`
   - `rubric_file = brief-rubric.json`
   - `critical_risks`
   - `weighted_score`
   - `verdict = rework_required | ready_for_activation`
5. В финальном ответе сообщи:
   - какой `brief-vN.md` был проверен;
   - какой `brief-review-X.md` создан;
   - какой `brief-score-X.json` создан;
   - есть ли 0 замечаний или нет;
   - блок `Execution Metadata` со значениями `system`, `model`, `provider`, `execution_date`, `prompt_id`;
   - блок `Runtime Telemetry` со значениями `started_at`, `finished_at`, `elapsed_seconds`, `input_tokens`, `output_tokens`, `total_tokens`, `estimated_cost`, `limit_context`, где недоступные поля помечены как `unknown` или `not available in current runtime`.
