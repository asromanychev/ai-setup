# Инструкция для AI-агента (Task Definition): Генерация Brief

Твоя задача — подготовить черновик Брифа (Brief) на основе исходной задачи.
Бриф описывает исключительно Problem Space: какую проблему решаем, для кого и зачем. Бриф строго не должен содержать Solution Space: как именно это будет реализовано в коде, какие таблицы в БД создавать, какие библиотеки или API выбирать.

## Шаг 1. Сбор контекста (Grounding on Memory Bank)
1. Извлеки номер Issue из запроса пользователя. Поддерживай форматы вида `issues/5`, `#5`, полная GitHub-ссылка на issue или явное упоминание `Issue 5`.
2. Если в запросе нет текста issue или ссылки на него, попроси пользователя дать текст issue или URL. Не начинай генерацию без содержимого issue.
3. Сформируй 4-значный префикс с ведущими нулями: `5 -> 0005`.
4. Если папка для этого issue уже существует в `memory-bank/issues/` и начинается с этого префикса, используй её.
5. Если папки ещё нет, придумай очень короткий slug на английском языке из 2-3 слов по сути задачи и создай директорию в формате `memory-bank/issues/0005-user-profile/`.
6. Внутри папки задачи создай подпапки:
   - `iterations/brief/`
   - `iterations/spec/`
   - `iterations/plan/`
   - `iterations/code/`
   - `iterations/activation/`
   - `iterations/analysis/`
   если их ещё нет.
7. Все артефакты Brief хранятся только в `iterations/brief/`.
8. Если в `iterations/brief/` уже есть черновики `brief-vN.md`, не перезаписывай их. Новый первый черновик создавай только если для Brief ещё нет ни одной версии.

## Шаг 2. Изучение правил
1. Прочитай `.claude/CLAUDE.md`.
2. Прочитай `.codex/CODEX.md`, если файл существует.
3. Прочитай шаблон `memory-bank/templates/brief-template.md`, если он существует.
4. Учти, что утвержденные документы лежат в корне папки issue, а все draft/review/score/fix-артефакты этапа Brief лежат в `iterations/brief/`.
5. Прочитай `memory-bank/templates/rubric-template.json`, если файл существует.

## Факторы выполнения
- execution metadata: фиксируй `system`, `model`, `provider`, `execution_date`, `prompt_id`; если среда не раскрывает значение, записывай `unknown`.
- telemetry capture: фиксируй `started_at`, `finished_at`, `elapsed_seconds`; если runtime отдает usage, добавляй `input_tokens`, `output_tokens`, `total_tokens`; если billing или лимиты недоступны, записывай `not available in current runtime`.
- version overwrite: отключен; существующие версии draft и review не перезаписываются.

## Шаг 2.5. Предварительный анализ по 5W3H
Перед тем как писать Brief, разложи исходный текст issue по 8 измерениям 5W3H. Это внутренняя рабочая таблица — она не попадает в итоговый Brief, но служит основой для всех разделов.

| Измерение | Вопрос | Ответ из issue (или «не указано») |
|-----------|--------|-----------------------------------|
| **Who** | Кто затронут / кто выполняет? | |
| **What** | Что именно сломано или отсутствует? | |
| **When** | Когда это нужно / какие временны́е зависимости? | |
| **Where** | В каком компоненте / слое системы проблема? | |
| **Why** | Зачем это нужно, какова бизнес-ценность? | |
| **How** | Как сейчас работает (текущее поведение)? | |
| **How much** | Какой измеримый порог успеха? | |
| **How long** | Насколько велика задача / есть ли ограничения по срокам? | |

После заполнения таблицы:
- Если три и более строк помечены «не указано» — сформулируй уточняющие вопросы пользователю и **остановись**, не переходи к Шагу 3 до получения ответов.
- Если «не указано» только в `When` или `How long` — допустимо продолжить; в Brief явно отметь эти измерения как «нет данных».

## Шаг 3. Выполнение задачи (Problem Space)
0. Если `iterations/brief/brief-rubric.json` ещё не существует, создай его.
   Рубрика должна содержать ровно 5 измерений с суммой весов `1.0`:
   - `problem_clarity`
   - `stakeholder_specificity`
   - `problem_solution_separation`
   - `success_criteria_observability`
   - `scope_boundary_integrity`
   Для каждого измерения опиши уровни `1`, `3`, `5` и пометь `evaluation_mode: isolated`.
1. Проанализируй текст issue и найди:
   - противоречия;
   - недосказанности;
   - скрытые допущения;
   - неочевидные ограничения.
2. Подготовь драфт Brief строго по следующей структуре:
   - Цель и Проблема:
     конкретная и измеримая проблема, боль пользователя или инцидент; никаких абстрактных формулировок вроде «улучшить».
   - Для кого:
     роль пользователя или стейкхолдер, для которого решается проблема.
   - Доменный и Технический контекст:
     ограничения, бизнес-правила, текущая архитектура и важные рамки, но без описания решения.
   - Критерий успеха:
     проверяемый критерий, по которому можно доказать, что проблема решена на пользовательском уровне.
   - Что не в скоупе:
     явный список того, что не входит в задачу, чтобы ограничить gold plating.
2.1. Scope-canonicalization gate:
   - если issue описывает широкий архитектурный фон, но текущая задача покрывает только один подтип артефакта, один переход или один execution-path, выбери один канонический объект scope до написания текста;
   - проведи этот объект без подмены через `Цель и Проблема`, `Для кого`, `Контекст`, `Критерий успеха` и `Что не в скоупе`;
   - более широкий фон разрешён только как архитектурный контекст с явной пометкой, что он не расширяет текущую задачу;
   - запрещено обещать в ранних разделах более широкий scope, а в критериях успеха проверять только его подмножество.
3. Бриф должен сохранять разделение Problem Space / Solution Space:
   - разрешено описывать ограничения и инварианты среды;
   - запрещено предлагать конкретный код, структуру таблиц, конкретные эндпоинты, классы или библиотеки как часть решения.
4. Если данных не хватает, всё равно собери драфт по полной структуре:
   - отмечай пропуски явно и конкретно;
   - формулируй уточняющие вопросы пользователю только по тем местам, без которых документ нельзя сделать однозначным.
5. Перед сохранением выполни обязательный consistency-check:
   - каждый пункт в `Критерии успеха` должен устранять конкретный симптом из раздела `Цель и Проблема`;
   - ни один пункт в `Что не в скоупе` не должен исключать механизм, без которого этот симптом нельзя устранить;
   - если находишь конфликт вида `проблема заявлена`, но `нужное для её устранения исключено из scope`, перепиши границы задачи до сохранения.
5.1. Success-criterion leakage gate:
   - если критерий успеха можно доказать только через будущий API-контракт, runtime-артефакт, внутренний helper, класс ошибки или другую solution-specific деталь, верни формулировку на уровень наблюдаемого problem outcome;
   - не подменяй пользовательский или доменный результат перечислением того, что должно появиться в коде, тестах или bootstrapping.
6. Любой термин, на который опирается `Критерий успеха`, обязан быть определён через наблюдаемое поведение в самом Brief.
   - Если термин нельзя определить однозначно на уровне Problem Space, убери его из критерия успеха или явно пометь как открытый вопрос, блокирующий утверждение документа.
   - Не используй в критерии успеха недоопределённые термины вроде «нормальный шаг», «детерминированный результат», «минимальная граница» без явной расшифровки.
6.1. Undefined-term gate:
   - если термин остаётся недоопределённым, запрещено одновременно оставлять его в `Критерии успеха` и считать Brief готовым к утверждению;
   - сначала либо доопредели термин через наблюдаемое поведение, либо вынеси его в открытый вопрос с явной пометкой, что документ ещё не готов.
6.2. Semantics-first gate:
   - если Критерий успеха зависит от выбора между двумя конкурирующими семантиками поведения (idempotent insert vs upsert, skip vs overwrite, error vs log, approve vs reject), зафиксируй одну выбранную семантику в разделе «Доменный контекст» до написания Критерия успеха;
   - в Критерии успеха использовать только зафиксированный термин; запрещено оставлять нераскрытую альтернативу вида «(A или B)» — такой критерий неверифицируем;
   - если выбор семантики невозможно сделать без уточнения у пользователя, вынести это как блокирующий открытый вопрос до сохранения документа.
7. Перед сохранением выполни boundary-check для терминов, которые легко уводят Brief в API или implementation space:
   - если в тексте появляются термины вроде `checkpoint`, `plugin id`, `один шаг синка`, `детерминированный результат`, `минимальная граница`, сначала проверь, можно ли выразить их через наблюдаемое проблемное поведение;
   - если нельзя, не лечи это перечислением return values, типов, ошибок, тестов, классов или вариантов реализации;
   - запрещено оставлять термин одновременно в `Критерии успеха` и в списке открытых вопросов: либо он уже определён, либо документ ещё не готов к утверждению.

## Шаг 4. Финализация и Сохранение
1. Сохрани рубрику в `iterations/brief/brief-rubric.json`, если файла ещё нет.
2. Сохрани результат в подпапку `iterations/brief/` внутри найденной или созданной папки issue под именем `brief-v1.md`, если это первая версия.
3. Если `brief-v1.md` уже существует, не перезаписывай его. Создай следующую версию `brief-v{N+1}.md`.
4. Ничего не записывай в корень папки issue как `brief.md` на этом шаге: это draft-артефакт, не active.
5. В финальном ответе сообщи:
   - в какую папку issue ты положил файл;
   - какое имя версии создал;
   - была ли создана `brief-rubric.json`;
   - есть ли открытые вопросы;
   - блок `Execution Metadata` со значениями `system`, `model`, `provider`, `execution_date`, `prompt_id`;
   - блок `Runtime Telemetry` со значениями `started_at`, `finished_at`, `elapsed_seconds`, `input_tokens`, `output_tokens`, `total_tokens`, `estimated_cost`, `limit_context`, где недоступные поля помечены как `unknown` или `not available in current runtime`.
