# Инструкция для AI-агента (Task Definition): Имплементация Code

Твоя задача — написать код для фичи строго по утвержденной Спецификации и Плану.

## Шаг 1. Сбор контекста (Grounding on Memory Bank)
1. Извлеки номер Issue из запроса пользователя.
2. Сформируй 4-значный префикс.
3. Найди соответствующую папку в `memory-bank/issues/`.
4. Прочитай утвержденные `spec.md` и `plan.md` в корне папки issue.
5. Если хотя бы один из этих файлов отсутствует, остановись и сообщи пользователю, что нельзя начинать реализацию без утвержденных Spec и Plan.

## Шаг 2. Изучение правил
1. Прочитай `.claude/CLAUDE.md`.
2. Прочитай `.codex/CODEX.md`, если файл существует.
3. Прочитай шаблон `memory-bank/templates/code-template.md`, если он существует.
4. Прочитай `memory-bank/templates/rubric-template.json`, если файл существует.
5. Проверь реальные файлы проекта, указанные в `plan.md`, прежде чем вносить изменения.

## Факторы выполнения
- execution metadata: фиксируй `system`, `model`, `provider`, `execution_date`, `prompt_id`; если среда не раскрывает значение, записывай `unknown`.
- telemetry capture: фиксируй `started_at`, `finished_at`, `elapsed_seconds`; если runtime отдает usage, добавляй `input_tokens`, `output_tokens`, `total_tokens`; если billing или лимиты недоступны, записывай `not available in current runtime`.
- scope expansion: только через явную эскалацию; allowlist файлов из `plan.md` обязателен.
- gold plating: отключен.
- blocker handling: локальные блокеры внутри текущего scope подлежат разрешению до завершения реализации; внешние prerequisites и scope change подлежат эскалации с явной причиной.

## Шаг 3. Выполнение задачи (Code Implementation)
0. Если `iterations/code/code-rubric.json` ещё не существует, создай его.
   Рубрика должна содержать ровно 5 измерений:
   - `scope_compliance`
   - `invariant_preservation`
   - `test_adequacy`
   - `runtime_readiness`
   - `regression_risk`
   Сумма весов должна быть `1.0`.
Реализуй код строго по `spec.md` и `plan.md` с соблюдением следующих ограничений:
1. Никакого Gold Plating:
   запрещено писать логику на будущее, добавлять новые зависимости или рефакторить соседний код, который не нужен для Acceptance Criteria.
2. Изоляция:
   изменяй только те файлы, которые явно указаны в плане реализации.
2.1. Scope lock:
   перед первой правкой зафиксируй allowlist файлов из `plan.md` и сравни его с фактическими изменениями, которые собираешься вносить.
   - Если для прохождения AC нужен файл вне allowlist, останови реализацию и верни задачу к пересмотру `spec.md`/`plan.md`.
   - Не лечи внешний prerequisite правкой файла, который был помечен как out of scope или вообще не фигурировал в утверждённом плане.
   - Не добавляй внеплановые изменения только потому, что они выглядят безопасными, удобными или «помогают запустить тесты».
2.2. Pre-edit runtime gate:
   до первой правки проверь, материализованы ли runtime prerequisites из `spec.md`/`plan.md`: boot/load-path, test DB, env/config, test helpers, schema.
   - Если prerequisite уже существует и просто не активирован, разрешено активировать его в рамках allowlist-плана.
   - Если prerequisite отсутствует как артефакт проекта, это не локальный фикс, а повод остановить кодинг и вернуть задачу к пересмотру active `spec.md`/`plan.md`.
3. Инварианты:
   соблюдай инварианты из Спецификации. Ни один существующий контракт не должен быть сломан.
4. Безопасность:
   валидируй входные данные, не хардкодь секреты, корректно обрабатывай null-значения.

Перед изменениями составь внутреннюю трассировку `FR/Invariant -> code path -> test case`.
- Для каждого нетривиального инварианта (`deep copy`, `freeze`, `determinism`, `isolation`, `idempotence`, `normalization`) добавь хотя бы один adversarial test, который пытается сломать гарантию через aliasing, внешнюю мутацию, повторный вызов или нестандартный вход.
- Не считай инвариант закрытым, если у тебя есть только happy-path и типовые негативные кейсы уровня `nil`/`wrong type`.
- Если меняешь helper'ы уровня `deep_dup`, `deep_freeze`, normalization или derived fields, adversarial tests обязаны проверять не только значения, но и ключи Hash, aliasing между входом и внутренним состоянием, а также повторное использование тех же объектов между вызовами.
- Если локальный runtime доступен, выполняй проверки в два прохода: сначала узкий набор тестов по изменённым файлам, затем финальный набор/gate из `spec.md` с coverage или другими acceptance-порогами.
- Bulk-insert validation guard:
  - если реализация использует bulk `insert` (AR class method) внутри идемпотентного метода, явно добавь validation guard до вызова `insert` для каждого AR validation, который `insert` обходит (presence, format, custom);
  - запрещено полагаться только на DB constraint (NOT NULL, unique index) как единственную защиту от пустых/nil значений, если Spec требует AR-level error message (например `ActiveRecord::RecordInvalid`).
- Human-attribute-name consistency:
  - если тест ожидает error message вида «X id can't be blank» (human-readable attribute name), проверь, что модель имеет `human_attribute_name` override, возвращающий нужное имя;
  - если существующие модели проекта уже имеют такой override (например Source), следуй тому же паттерну для новой модели с аналогичными атрибутами.
- Pre-completion runtime gate:
  - если active `plan.md` содержит финальный runtime gate вне чистого unit-level path, не объявляй реализацию завершённой до попытки пройти именно этот gate;
  - до завершения этапа обязан либо материализовать среду для финальной команды, либо зафиксировать точный внешний blocker с командой, на которой он проявился;
  - завершение только по локальному unit-pass запрещено, если план всё ещё требует DB-backed regression gate, boot/runtime prerequisite или другой execution-level финальный шаг.
- Если в процессе реализации появляется блокер, сначала попытайся разрешить его в рамках текущих `spec.md`, `plan.md` и allowlist файлов.
  - Если блокер можно снять без выхода за scope и без закрытия внешнего prerequisite, разрули его и доведи реализацию до конца.
  - Если выясняется, что active `spec.md`/`plan.md` устарели относительно минимального необходимого runtime remedy, не придумывай silent fix: сначала верни цикл к обновлению active-документов, потом продолжай реализацию.
  - Если блокер требует внешний prerequisite, изменение утвержденного scope или правку файла вне allowlist, не маскируй проблему локальной импровизацией: зафиксируй `implementation blocker`, опиши, что уже проверено, и остановись с явной эскалацией к пересмотру `spec.md`/`plan.md`.

Exit Criteria:
- все Acceptance Criteria из Спецификации покрыты рабочим кодом;
- написаны тесты для новых happy-path и error-path сценариев;
- существующие тесты остаются зелёными.
- фактический набор изменённых файлов не нарушает allowlist из `plan.md`, либо нарушение явно эскалировано как scope change до продолжения реализации.
- финальный runtime gate из `plan.md` либо пройден, либо остановлен на явно зафиксированном внешнем blocker без ложного объявления готовности.

## Шаг 4. Финализация и Сохранение
1. Внеси изменения в исходники проекта и тесты в рабочем дереве.
2. Не создавай отдельный `code-v1.md` в `iterations/`: результатом этого шага являются изменённые файлы проекта.
3. Сохрани `code-rubric.json` в `iterations/code/`, если файла ещё нет.
4. Если у тебя есть возможность локально выполнить релевантные тесты, сделай это и зафиксируй результат.
5. В финальном ответе сообщи:
   - какие файлы были изменены;
   - была ли создана `code-rubric.json`;
   - какие тесты добавлены или обновлены;
   - какие проверки были выполнены и что не удалось проверить;
   - какие блокеры встретились, как они были разрешены или почему потребовали эскалации;
   - блок `Execution Metadata` со значениями `system`, `model`, `provider`, `execution_date`, `prompt_id`;
   - блок `Runtime Telemetry` со значениями `started_at`, `finished_at`, `elapsed_seconds`, `input_tokens`, `output_tokens`, `total_tokens`, `estimated_cost`, `limit_context`, где недоступные поля помечены как `unknown` или `not available in current runtime`.
