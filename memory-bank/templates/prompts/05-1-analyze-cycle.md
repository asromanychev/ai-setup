# Инструкция для AI-агента (Task Definition): Анализ Цикла и Дистилляция Правил

Твоя задача — выполнить postmortem-анализ полного цикла реализации задачи (`Brief -> Spec -> Plan -> Code`) и дистиллировать из логов конкретные lessons learned, чтобы улучшить базовые шаблоны в `memory-bank/templates/prompts/` и сократить число будущих шагов `03-fix`.

Этот шаг выполняется после завершения реализации кода и использует весь накопленный контекст по issue: draft-версии, review-файлы, fix-итерации и утвержденные active-документы.

## Шаг 1. Сбор контекста (Grounding on Memory Bank)
1. Извлеки номер Issue из запроса пользователя.
2. Сформируй 4-значный префикс.
3. Найди папку issue в `memory-bank/issues/`, которая начинается с этого префикса.
4. Внутри папки issue:
   - прочитай все файлы в `iterations/brief/`, `iterations/spec/`, `iterations/plan/`, `iterations/code/`, `iterations/activation/`, `iterations/analysis/`, если они существуют;
   - прочитай active-документы в корне issue, если они существуют (`brief.md`, `spec.md`, `plan.md`);
   - используй `*-score-*.json` и `*-fix-contract-*.json` как часть анализа;
   - при наличии code review и изменённого кода используй их как часть анализа поздних этапов.
5. Если папка issue или `iterations/` отсутствуют, остановись и сообщи пользователю, что контекста для анализа недостаточно.
6. Ничего не анализируй выборочно: нужно прочитать все файлы текущего цикла, включая scorecards и fix-contracts.

## Шаг 2. Изучение правил
1. Прочитай `.claude/CLAUDE.md`.
2. Прочитай `.codex/CODEX.md`, если файл существует.
3. Прочитай актуальные шаблоны в `memory-bank/templates/prompts/`, которые потенциально нужно улучшить:
   - `01-1-generate-brief.md`
   - `02-1-review-brief.md`
   - `03-1-fix-brief.md`
   - `01-2-generate-spec.md`
   - `02-2-review-spec.md`
   - `03-2-fix-spec.md`
   - `01-3-generate-plan.md`
   - `02-3-review-plan.md`
   - `03-3-fix-plan.md`
   - `01-4-generate-code.md`
   - `02-4-review-code.md`
   - `03-4-fix-code.md`
   - `04-activate-artifact.md`, если файл существует

## Факторы выполнения
- execution metadata: фиксируй `system`, `model`, `provider`, `execution_date`, `prompt_id`; если среда не раскрывает значение, записывай `unknown`.
- telemetry capture: фиксируй `started_at`, `finished_at`, `elapsed_seconds`; если runtime отдает usage, добавляй `input_tokens`, `output_tokens`, `total_tokens`; если billing или лимиты недоступны, записывай `not available in current runtime`.
- selective reading: отключен; анализ должен покрывать все файлы текущего цикла в `iterations/`.

## Шаг 3. Выполнение задачи (Cycle Analysis)
Выполни анализ в 3 слоя:

1. Corrective Analysis:
   - сравни, что было сгенерировано на шагах `01`, какие замечания появились на шагах `02`, и что пришлось исправлять на шагах `03`;
   - отдельно анализируй расхождения между markdown review и `*-score-*.json`;
   - найди, какие именно систематические ошибки допускал агент на этапе генерации;
   - ищи не единичные wording-ошибки, а повторяющиеся паттерны мышления, декомпозиции, ревью или тест-дизайна.

2. Cross-stage Analysis:
   - найди, какие поздние проблемы (`Plan`, `Code`) были вызваны неполнотой ранних этапов (`Brief`, `Spec`);
   - отдельно проверь разрывы трассировки:
     `Problem -> Success -> Scope -> Functional Requirements -> Invariants -> Acceptance Criteria -> Plan -> Code -> Tests`;
   - если поздняя ошибка стала возможной потому, что ранний документ не зафиксировал нужную границу, инвариант, adversarial case, delta или verification gate, зафиксируй это как межэтапный дефект.

3. Enhancement Analysis:
   - оцени качество самих review-файлов;
   - оцени качество рубрик, scorecards и fix-contracts: ловят ли они реальный blocker или создают churn;
   - различай review, которые находили реальные архитектурные или логические проблемы, и review, которые создавали churn за счёт мелких формулировок;
   - если review пропустил архитектурный дефект, но гонял документ по wording, отрази это как рекомендацию для review prompt, а не для generate prompt.

## Шаг 4. Формат дистилляции (Вывод)
Не пересказывай логи по шагам и не делай хронологический отчёт.
Выдай только готовые рекомендации для обновления `memory-bank/templates/prompts/`.

Разбей ответ на 5 блоков:
1. Brief
2. Spec
3. Plan
4. Code
5. Кросс-цикловая оптимизация

Для блоков `Brief`, `Spec`, `Plan`, `Code` используй только такой формат:

Файл для обновления: `[имя prompt-файла]`
Условие (Триггер): `[когда это правило должно срабатывать]`
Действие (Что добавить в промпт): `[жёсткая инструкция, которую нужно встроить в шаблон]`
Избегать (Граница/Антипаттерн): `[что агенту запрещено делать или где он должен остановиться]`
Обоснование: `[1 короткое предложение, какую ошибку цикла это предотвращает]`

Правила вывода:
- Не предлагай абстрактные советы уровня «быть внимательнее».
- Каждое правило должно быть привязано к конкретному prompt-файлу в `memory-bank/templates/prompts/`.
- Если проблема относится к качеству ревью, обновляй `02-*`; если к качеству генерации — `01-*`; если к качеству исправления — `03-*`.
- Предпочитай boundary rules, которые заставляют агента остановиться и проверить условие до перехода к следующему этапу.
- Если одна и та же ошибка может быть предотвращена на более раннем этапе, предлагай правило для самого раннего релевантного prompt.

Для блока `Кросс-цикловая оптимизация`:
- предложи 1-2 общих правила для системного промпта или общего workflow;
- эти правила должны уменьшать количество шагов `03-fix` в следующем цикле;
- формулируй их как универсальные gates, а не как советы под одну конкретную задачу.

## Шаг 5. Финализация и Сохранение
1. Сохрани результат в подпапку `iterations/analysis/` текущей папки issue под именем `cycle-analysis-X.md`, где `X` — следующий порядковый номер такого анализа.
2. Не перезаписывай предыдущие `cycle-analysis-X.md`.
3. В финальном ответе сообщи:
   - какую папку issue ты проанализировал;
   - какой `cycle-analysis-X.md` создан;
   - какие prompt-файлы затронуты рекомендациями;
   - блок `Execution Metadata` со значениями `system`, `model`, `provider`, `execution_date`, `prompt_id`;
   - блок `Runtime Telemetry` со значениями `started_at`, `finished_at`, `elapsed_seconds`, `input_tokens`, `output_tokens`, `total_tokens`, `estimated_cost`, `limit_context`, где недоступные поля помечены как `unknown` или `not available in current runtime`.
