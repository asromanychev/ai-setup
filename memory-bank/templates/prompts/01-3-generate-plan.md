# Инструкция для AI-агента (Task Definition): Генерация Plan

Твоя задача — на основе утвержденной Спецификации создать draft Implementation Plan. План — это пошаговая инструкция, декомпозирующая Спецификацию на конкретные изменения в коде.

## Шаг 1. Сбор контекста (Grounding on Memory Bank)
1. Извлеки номер Issue из запроса пользователя.
2. Сформируй 4-значный префикс.
3. Найди папку issue в `memory-bank/issues/`, которая начинается с этого префикса.
4. Прочитай утвержденный `spec.md` в корне папки issue.
5. Если `spec.md` отсутствует, остановись и сообщи пользователю, что Спецификация ещё не утверждена. Не подменяй её draft-файлами из `iterations/`.

## Шаг 2. Изучение правил
1. Прочитай `.claude/CLAUDE.md`.
2. Прочитай `.codex/CODEX.md`, если файл существует.
3. Прочитай шаблон `memory-bank/templates/plan-template.md`, если он существует.
4. Прочитай `memory-bank/templates/rubric-template.json`, если файл существует.
5. Проверь существование всех файлов и модулей, которые потенциально будут затронуты планом.

## Факторы выполнения
- execution metadata: фиксируй `system`, `model`, `provider`, `execution_date`, `prompt_id`; если среда не раскрывает значение, записывай `unknown`.
- telemetry capture: фиксируй `started_at`, `finished_at`, `elapsed_seconds`; если runtime отдает usage, добавляй `input_tokens`, `output_tokens`, `total_tokens`; если billing или лимиты недоступны, записывай `not available in current runtime`.
- version overwrite: отключен; существующие версии draft и review не перезаписываются.

## Шаг 3. Выполнение задачи (Implementation Decomposition)
0. Если `iterations/plan/plan-rubric.json` ещё не существует, создай его.
   Рубрика должна содержать ровно 5 измерений:
   - `dependency_order`
   - `step_atomicity`
   - `delta_specificity`
   - `runtime_gate_quality`
   - `test_materialization`
   Сумма весов должна быть `1.0`.
Построй план на основе `spec.md` по следующим требованиям:
1. Атомарность:
   каждый шаг должен затрагивать конкретный файл или модуль и описывать одно логическое изменение.
2. Grounding:
   все упомянутые файлы и классы должны реально существовать в текущей кодовой базе, либо шаг должен явно создавать новый файл в допустимом месте.
3. Зависимости:
   порядок шагов должен быть технически корректным. Например, миграции, интерфейсы и контракты должны появляться до их использования в сервисах или API.
4. Полнота:
   включи шаги по добавлению или обновлению unit/integration tests для проверки Acceptance Criteria.
5. Delta-first:
   для каждого файла сначала зафиксируй, что уже реализовано в текущем коде, чего не хватает по `spec.md`, и что менять нельзя.
   Если файл уже соответствует спецификации, пометь его как `no-op` и вынеси подтверждение этого факта в тестовый или проверочный шаг, а не в шаг редактирования.
6. Условные правки:
   условный шаг разрешён только если перед ним есть отдельный шаг-проверка или шаг-тест, который выявляет расхождение.
   Нельзя писать шаг вида «проверить и при необходимости исправить» без явного источника решения о правке.
7. Финальная верификация:
   план обязан заканчиваться отдельным шагом runtime-проверки с конкретной командой или набором команд, которые подтверждают зелёный тестовый набор и все gate-условия из Acceptance Criteria, включая coverage, если он заявлен в спецификации.
7.1. Two-tier runtime gate:
   - если `spec.md` одновременно требует новый целевой suite и правило `все существующие тесты остаются зелёными`, план обязан разделить их на два разных verification-слоя;
   - сначала укажи `focused verification` нового или изменённого контракта;
   - затем отдельным последним шагом укажи `regression gate` по существующим suites с одной явной командой и наблюдаемым pass/fail сигналом;
   - запрещено завершать план единственным запуском нового spec-файла, если active spec требует сохранить зелёными существующие suites.
8. Observe first, edit second:
   если шаг условный, он обязан ссылаться на наблюдаемый trigger из более раннего шага: конкретный тест, команду или проверку, по результату которой принимается решение о редактировании файла.
   Источник решения должен идти раньше условного шага в порядке зависимостей.
8.1. Observable gate rule:
   для каждого verification/gate шага укажи, какой именно наблюдаемый сигнал он производит: `pass/fail`, конкретный exception, diff в файле, наличие/отсутствие константы, количество примеров, coverage threshold.
   Запрещено оставлять шаги вида «проверить, что всё работает» без измеримого сигнала, по которому следующий шаг принимает решение.
9. Executable edge-case / prerequisite materialization:
   если `spec.md` содержит внешний prerequisite, повреждённое состояние, недопустимое по штатным инвариантам БД, или edge-case, который нельзя получить обычным happy-path, план обязан материализовать это отдельным шагом:
   - как проверить prerequisite;
   - как временно создать запрещённое состояние для теста;
   - как восстановить инвариант после проверки, если это требуется.
   Нельзя оставлять такие случаи текстовым предупреждением без исполнимого шага проверки или setup.
10. Test materialization notes:
   если спецификация опирается на runtime gate вне unit-level логики, план обязан отдельно указать, каким минимальным setup этот gate достигается локально: boot file, env var, test DB, seed/state preparation, отдельная команда prepare.
   Не оставляй такие prerequisites неявными знаниями среды или «само собой разумеющимися» шагами.

Выведи план строго в виде нумерованного списка шагов, где для каждого шага указаны:
- целевой файл(ы);
- суть изменения;
- какие зависимости этот шаг разрешает для последующих шагов.

Для каждого шага редактирования формулируй дельту явно в виде:
- что уже есть в текущем коде;
- чего не хватает по `spec.md`;
- что в этом файле запрещено переписывать без отдельного наблюдаемого расхождения.

10.1. Bulk-insert validation delta:
   если шаг создаёт метод с bulk `insert` и `spec.md` требует AR-level error message (например `ActiveRecord::RecordInvalid`) для невалидных входных данных, в delta этого шага явно указать:
   - какие AR validations не вызываются автоматически при `insert`;
   - как именно validation guard реализуется до `insert` (explicit instance creation + validate, или ручная проверка + raise);
   - нельзя оставлять это «реализацией по усмотрению агента».

## Шаг 4. Финализация и Сохранение
1. Сохрани `plan-rubric.json` в `iterations/plan/`, если файла ещё нет.
2. Сохрани draft-план в подпапку `iterations/plan/` текущей папки issue под именем `plan-v1.md`, если это первая версия.
3. Если `plan-v1.md` уже существует, создай `plan-v{N+1}.md`.
4. Не записывай результат в корень папки issue как `plan.md` на этом шаге.
5. В финальном ответе сообщи:
   - путь к созданному файлу;
   - номер версии;
   - была ли создана `plan-rubric.json`;
   - были ли в плане шаги создания новых файлов;
   - блок `Execution Metadata` со значениями `system`, `model`, `provider`, `execution_date`, `prompt_id`;
   - блок `Runtime Telemetry` со значениями `started_at`, `finished_at`, `elapsed_seconds`, `input_tokens`, `output_tokens`, `total_tokens`, `estimated_cost`, `limit_context`, где недоступные поля помечены как `unknown` или `not available in current runtime`.
