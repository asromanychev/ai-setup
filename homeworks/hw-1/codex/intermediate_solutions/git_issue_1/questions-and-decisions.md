## Автоматически найденные неоднозначности

### 1. Нужно ли в Brief сразу выбирать форму checkpoint?
- Наблюдение: issue допускает выбор между opaque и структурированным payload.
- Принятое решение: в Brief не фиксировать конкретную форму, только зафиксировать сам checkpoint как обязательный вход/выход шага.
- Почему: это снимает лишнюю детализацию на уровне Brief и оставляет решение для Spec.

### 2. Что считать минимальным результатом одного шага синка?
- Наблюдение: issue говорит про "один шаг синка" и checkpoint на входе/выходе, но не перечисляет все поля результата.
- Принятое решение: на уровне Brief считать минимально обязательными `records`, `checkpoint_out`, `finished`.
- Почему: без этого критерии успеха не были бы проверяемыми.

### 3. Нужно ли закладывать Telegram в текущий артефакт?
- Наблюдение: в issue Telegram упомянут как следующий плагин, но не часть текущего объёма.
- Принятое решение: Telegram явно оставить вне объёма Brief.
- Почему: иначе получится scope creep относительно issue.

### 4. Нужно ли разрешать fallback для неизвестного plugin_id?
- Наблюдение: issue требует явную ошибку для неизвестного идентификатора.
- Принятое решение: fallback и `nil`-ответы запрещены.
- Почему: это делает поведение реестра тестируемым и однозначным.
