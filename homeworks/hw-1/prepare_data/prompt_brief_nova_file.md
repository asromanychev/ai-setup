# SDD-Агент: Полный цикл Brief → Spec → Plan → Implement

Ты — опытный технический мен
еджер и архитектор, применяющий методологию Spec-Driven Development (SDD). Твоя задача — провести меня через полный цикл реализации фичи по GitHub issue, создавая и улучшая документы на каждом этапе.

## Контекст задачи

GitHub Issue: https://github.com/asromanychev/ai-da-collect/issues/1

Методология: SDD (Spec-Driven Development) — итерируем на документах (Brief, Spec, Plan), а не на коде. Каждый документ проходит ревью до 0 замечаний перед переходом на следующий шаг.

## Шаг 1: Анализ GitHub Issue и генерация Brief

Прочитай содержание issue и сгенерируй Brief по следующей структуре:

### Формат Brief (до 500 слов):

```
## Brief: [название фичи]

### Проблема
[Что именно не работает или чего не хватает — 2-3 предложения]

### Цель
[Что должно быть после реализации — измеримо]

### Входные данные
[Что получает система/пользователь на входе]

### Ожидаемый результат
[Что выдаёт система на выходе]

### Критерии готовности (Definition of Done)
- [ ] критерий 1
- [ ] критерий 2
- [ ] критерий 3

### Ограничения и риски
[Что нельзя ломать, что может пойти не так]

### Связанные документы
- GitHub Issue: [ссылка]
- Spec: memory-bank/features/001/spec.md (будет создан)
- Plan: memory-bank/features/001/plan.md (будет создан)
```

После генерации Brief выполни **Brief Review** — задай себе 5 вопросов:
1. Понятна ли проблема без дополнительного контекста?
2. Измеримы ли критерии готовности?
3. Есть ли неоднозначности в ожидаемом результате?
4. Покрыты ли граничные случаи?
5. Достаточно ли информации для написания Spec?

Выведи результат ревью:
```
### Brief Review — Замечания:
- [замечание 1 или «замечаний нет»]
- [замечание 2]

### Исправленный Brief (если были замечания):
[обновлённая версия]
```

## Шаг 2: Генерация Spec на основе Brief

Только после того, как Brief прошёл ревью (0 замечаний), сгенерируй Spec.

### Формат Spec (до 1000 слов):

```
## Specification: [название фичи]

### Назначение
[Что делает этот модуль/фича — 1 абзац]

### Интерфейс
#### Входные данные
| Параметр | Тип | Обязательный | Описание |
|----------|-----|--------------|----------|
| ... | ... | да/нет | ... |

#### Выходные данные
| Параметр | Тип | Описание |
|----------|-----|----------|
| ... | ... | ... |

### Поведение
#### Основной сценарий (Happy Path)
1. шаг 1
2. шаг 2
3. шаг 3

#### Граничные случаи
- Если [условие] → [поведение]
- Если [условие] → [поведение]

#### Запрещённое поведение
- Нельзя делать X
- Нельзя возвращать Y без Z

### Критерии правильного результата
- [ ] тест-критерий 1
- [ ] тест-критерий 2
- [ ] покрытие тестами >80%

### Зависимости
[Что должно существовать до реализации]
```

После генерации выполни **Spec Review** — проверь по 5 критериям:
1. Можно ли по Spec написать тесты без дополнительных вопросов?
2. Описаны ли все граничные случаи из Brief?
3. Есть ли неоднозначности в интерфейсе?
4. Покрыты ли все критерии готовности из Brief?
5. Можно ли дать эту Spec другому разработчику без объяснений?

Выведи:
```
### Spec Review — Замечания:
- [замечание или «замечаний нет»]

### Исправленная Spec (если были замечания):
[обновлённая версия]
```

## Шаг 3: Генерация Implementation Plan на основе Spec

Только после того, как Spec прошла ревью (0 замечаний), сгенерируй Plan.

### Формат Implementation Plan:

```
## Implementation Plan: [название фичи]

### Файловая структура
[какие файлы создаём / изменяем]

### Задачи (в порядке выполнения)

#### Задача 1: [название]
- Файл: path/to/file.py
- Что делаем: [конкретное действие]
- Тест: [как проверим]
- Зависит от: [задача N или «ничего»]

#### Задача 2: [название]
...

### Тесты
- Unit-тесты: [что тестируем]
- Интеграционные тесты: [что тестируем]
- Ожидаемое покрытие: >80%

### CI/CD
- [ ] Добавить GitHub Actions workflow
- [ ] Добавить badges в README.md

### Критерии завершения
- [ ] все задачи выполнены
- [ ] тесты проходят
- [ ] покрытие >80%
- [ ] CI зелёный
```

После генерации выполни **Plan Review** — проверь:
1. Можно ли отдать Plan агенту без дополнительных объяснений?
2. Каждая задача атомарна и завершена?
3. Порядок задач логичен (нет зависимостей, нарушающих последовательность)?
4. Покрыты ли все требования Spec?
5. Есть ли тесты для каждого критерия из Spec?

Выведи:
```
### Plan Review — Замечания:
- [замечание или «замечаний нет»]

### Исправленный Plan (если были замечания):
[обновлённая версия]
```

## Шаг 4: Инструкция для агента-реализатора

Когда Plan прошёл ревью, сформируй финальный промпт для агента-реализатора:

```
### Промпт для агента:

Ты — senior Python разработчик. Реализуй фичу строго по Implementation Plan ниже.

Правила:
- Не отступай от Plan без явного запроса
- Пиши тесты параллельно с кодом
- После каждой задачи из Plan сообщай: «Задача N выполнена: [что сделано]»
- Если встречаешь неоднозначность — останови работу и задай вопрос
- Целевое покрытие тестами: >80%

[вставить финальный Implementation Plan]
```

# Объяснение

## Принцип работы промпта

### Основная логика

Промпт реализует полный конвейер SDD (Spec-Driven Development) для одной фичи из GitHub issue. Вместо того чтобы сразу просить агента написать код, промпт выстраивает **цепочку документов с встроенными ревью-шлюзами**: каждый документ (Brief → Spec → Plan) проходит самопроверку по 5 критериям, прежде чем стать основой для следующего.

Ключевая идея из исследования здесь реализована буквально: **итерируем на задании (документе), а не на результате (коде)**. Агент-реализатор получает на вход уже выверенный Plan, а не размытый запрос.

### Использованные техники из исследования

Промпт опирается на три главных принципа Spec-Driven Development:
- Стабильный артефакт — это документ, а не результат
- Итерация происходит на спецификации через встроенное ревью
- Тест на воспроизводимость: «можно ли дать документ другому агенту без объяснений?»

---

## Техники из исследования

### Техника 1: Итерация на задании, а не на выходе

#### Как реализована в промпте

Каждый из трёх документов (Brief, Spec, Plan) содержит встроенный **Review-блок** с 5 конкретными вопросами. Агент не переходит к следующему шагу, пока не закроет все замечания. Это прямая реализация принципа «закрой дырку в спецификации, регенерируй» — только применённого к каждому слою документации последовательно.

Если Brief содержит неоднозначность, она не попадёт в Spec. Если Spec неполная, Plan не будет написан. Ошибка изолируется на самом раннем слое.

### Техника 2: Описание ЧТО, а не КАК

#### Практическое применение

В формате Spec явно задана структура: **интерфейс (входы/выходы), поведение, запрещённые действия, критерии правильного результата** — без упоминания конкретных библиотек, паттернов или технических решений. Это соответствует требованию исследования: «не называй инструменты — только поведение». Технические решения остаются за агентом-реализатором, которому передаётся Plan.

### Техника 3: Тест на воспроизводимость

#### Конкретная реализация

В каждом ревью-блоке есть вопрос вида: «Можно ли отдать этот документ другому разработчику/агенту без дополнительных объяснений?» Это прямое применение метода из исследования: **дай спецификацию второму агенту — результат должен совпасть**. Если нет — в документе есть неоднозначность.

### Техника 4: Накопление знаний через итерации

#### Как работает в цикле

Шаг 5 (Отчёт) фиксирует не только результат, но и **дырки в документах** и **правки промптов для следующей итерации**. Это реализует принцип из исследования: каждая правка спецификации — зафиксированное понимание крайнего случая. После 3–10 итераций у пользователя накапливается база улучшенных промптов для Brief, Spec и Plan — именно то, что курс называет «базой частых промптов» для продвинутого уровня.

### Техника 5: Лимит длины документов

#### Контроль объёма

Промпт явно задаёт ограничения: Brief до 500 слов, Spec до 1000 слов. Это соответствует правилу исследования «не больше 1500 слов» — документ должен читаться за один раз, иначе теряется контроль над его содержимым.

## Руководство по применению

### Когда использовать

Промпт оптимален для:
- Реализации конкретной фичи из GitHub issue с нуля
- Первых 3–10 итераций курса (Неделя 1)
- Случаев, когда нужно выработать шаблоны документов под свой проект

Промпт менее подходит для:
- Hotfix/bugfix с очевидным решением (цикл избыточен)
- Рефакторинга без чётких критериев (сложно написать Brief)

### Как адаптировать под свои задачи

#### Для разных типов задач

**Для бага:** В Brief замени раздел «Цель» на «Ожидаемое поведение» и «Фактическое поведение». Добавь раздел «Шаги воспроизведения».

**Для рефакторинга:** В Spec добавь раздел «Инварианты» — что должно остаться неизменным после рефакторинга. Критерии готовности должны включать «поведение системы не изменилось».

**Для другого стека:** В формате Plan замени упоминание Python на свой язык/фреймворк. Раздел CI/CD адаптируй под свой пайплайн.

#### Управление детализацией

Если агент генерирует слишком общие документы — добавь в начало промпта контекст проекта:
```
Контекст проекта: [краткое описание стека, архитектуры, ключевых соглашений]
```

Если документы слишком детальные и перегруженные — сократи лимиты слов: Brief до 300, Spec до 600.

#### Интеграция с файловой структурой курса

Промпт уже содержит пути `memory-bank/features/001/` — для каждой следующей фичи просто меняй номер папки (002, 003...). Отчёт из Шага 5 напрямую соответствует требованиям `homeworks/hw-1/report.md`.
