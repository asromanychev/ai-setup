## Шаг 1: Генерация Brief

Сгенерируй Brief в формате Markdown, строго следуя структуре ниже. Brief описывает ЧТО нужно сделать — не КАК технически реализовать.

### Структура Brief (файл: brief.md)

```markdown
# Brief: [Название фичи из Issue]

**Issue:** [ссылка на GitHub Issue]
**Дата:** [текущая дата]
**Система генерации:** [CODEX / CLAUDE]

---

## Контекст
[Почему эта задача существует. Какую проблему решает. Кто от этого выигрывает.]

## Цель
[Одно предложение: чего мы хотим достичь в результате реализации этой фичи.]

## Пользователь / Стейкхолдер
[Кто использует результат. Что для него важно.]

## Входные данные
[Что получает система / агент / функция на входе. Конкретно и измеримо.]

## Выходные данные
[Что должно быть результатом. Конкретно и измеримо.]

## Обязательные требования
- [Требование 1 — что ДОЛЖНО быть]
- [Требование 2]
- [Требование N]

## Ограничения и запреты
- [Что НЕ должно происходить]
- [Что выходит за рамки этой задачи]

## Критерии успеха
- [ ] [Критерий 1 — измеримый, проверяемый]
- [ ] [Критерий 2]
- [ ] [Критерий N]

## Связанные документы
- Issue: [ссылка]
- Spec: [будет создан на следующем шаге]
- Plan: [будет создан после Spec]
```

## Шаг 2: Brief Review (автоматическое ревью)

После генерации Brief — немедленно проведи его ревью по следующим критериям. Для каждого критерия выставь статус: ✅ Пройден / ❌ Не пройден / ⚠️ Требует уточнения.

### Чеклист ревью Brief

**Полнота:**
- [ ] Все разделы заполнены (нет пустых или заглушек типа "TBD")
- [ ] Цель сформулирована в одном предложении
- [ ] Входные и выходные данные конкретны и измеримы

**Качество требований:**
- [ ] Обязательные требования описывают поведение, а не технологии
- [ ] Ограничения явно исключают out-of-scope сценарии
- [ ] Нет противоречий между разделами

**Критерии успеха:**
- [ ] Каждый критерий проверяем (можно сказать "да" или "нет")
- [ ] Критерии покрывают happy path и хотя бы один edge case
- [ ] Нет субъективных критериев типа "работает хорошо"

**Связность с Issue:**
- [ ] Brief полностью отражает суть Issue
- [ ] Нет требований, которых нет в Issue (scope creep)
- [ ] Ссылка на Issue проставлена

**Spec-Driven качество:**
- [ ] Brief не содержит упоминания конкретных библиотек, фреймворков, инструментов
- [ ] Описывает ЧТО, а не КАК
- [ ] Объём не превышает 1000 слов

## Шаг 3: Итерация на Brief (если есть замечания)

Если ревью выявило замечания (❌ или ⚠️):
1. Для каждого замечания — найди конкретную дырку в Brief
2. Исправь только Brief (не результат ревью)
3. Повтори ревью по обновлённому Brief
4. Продолжай до тех пор, пока все критерии не получат статус ✅
5. НЕ переходи к следующему шагу (Spec), пока ревью не пройдено

## Правила работы агента

- Итерируй на Brief, а не на результатах ревью
- Не переходи к следующему шагу без прохождения ревью
- Не упоминай технологии в Brief (только поведение)
- Держи Brief до 1000 слов
- Каждая дырка в Brief — это зафиксированное знание, добавь её в отчёт
- Задай 3 уточняющих вопроса если что-то в Issue неоднозначно — ДО генерации Brief
