## Контекст

Сервис ai-da-collect собирает данные из внешних источников через плагины. Прежде чем подключать первый реальный источник (Telegram), необходимо зафиксировать единый контракт, которому обязан следовать каждый плагин. Без такого контракта ядро оркестрации не может управлять синхронизацией единообразно, а тесты вынуждены зависеть от внешних сервисов. Эта задача закрывает первый вертикальный срез: контракт + реестр + тестовая заглушка.

## Цель

Определить и зафиксировать минимальный поведенческий контракт плагина, механизм регистрации и разрешения плагинов по идентификатору, а также тестовую заглушку, позволяющую проверять ядро без сети, базы данных и внешних зависимостей.

## Пользователь / Стейкхолдер

- **Разработчик ядра** — использует контракт для оркестрации синхронизации, не зная деталей источника.
- **Разработчик плагина** — реализует контракт, чтобы подключить новый источник данных.
- **Автоматические тесты** — используют заглушку для проверки поведения ядра изолированно.

## Входные данные

Один шаг синка получает:
- **Идентификатор источника** — строковое значение, уникальное для данного плагина.
- **Конфигурацию источника** — произвольный набор параметров, достаточных для выполнения шага (учётные данные, адреса, лимиты и т.п.).
- **Checkpoint предыдущего запуска** — opaque payload произвольной структуры; при первом запуске равен null.

Реестр получает:
- **Идентификатор** — строка для поиска зарегистрированного плагина.

Тестовая заглушка получает при создании:
- **Список заранее заданных записей** — массив, разбитый на батчи, которые она будет возвращать пошагово.

## Выходные данные

Один шаг синка возвращает:
- **Массив сырых записей** за данный шаг (может быть пустым).
- **Следующий checkpoint** — opaque payload, который ядро передаст в следующий шаг; null если источник исчерпан.
- **Флаг наличия следующего батча** — явный признак того, есть ли данные за этим шагом.

Реестр возвращает:
- Зарегистрированную реализацию плагина по идентификатору.
- Явную ошибку при запросе незарегистрированного идентификатора (не null, не дефолтный плагин).

Тестовая заглушка возвращает:
- Сконфигурированные записи пошагово, батч за батчем.
- После отдачи всех записей — флаг исчерпания и null в качестве checkpoint.

## Обязательные требования

- Контракт плагина определяет: уникальный идентификатор источника, структуру входной конфигурации, метод выполнения одного шага синка принимающий checkpoint и возвращающий записи + следующий checkpoint + флаг продолжения.
- Checkpoint передаётся как opaque payload: ядро не интерпретирует его содержимое — только хранит между шагами и передаёт обратно плагину.
- Реестр поддерживает регистрацию произвольного числа плагинов и разрешение по идентификатору.
- При запросе незарегистрированного идентификатора реестр возбуждает явное исключение с читаемым сообщением.
- Тестовая заглушка реализует тот же контракт, что и боевые плагины.
- Тестовая заглушка конфигурируется при создании списком записей; возвращает их пошагово; после исчерпания сигнализирует завершение.
- Контракт документирован встроенными комментариями непосредственно в коде.
- Контракт, реестр и тестовая заглушка покрыты юнит-тестами, запускаемыми без внешних зависимостей.

## Ограничения и запреты

- В рамках этой задачи не реализуется: работа с базой данных, объектным хранилищем, сетевыми запросами, фоновыми задачами.
- Конкретные источники данных (Telegram и другие) не упоминаются и не реализуются в контракте.
- Ядро не должно знать о внутреннем устройстве checkpoint конкретного плагина.
- Тестовая заглушка не обращается к внешним сервисам или файловой системе.
- Нормализация, трансформация и анализ данных выходят за рамки задачи.

## Критерии успеха

- [ ] Контракт плагина определён и задокументирован в коде.
- [ ] Реестр возвращает зарегистрированный плагин по его идентификатору.
- [ ] Реестр возбуждает явное исключение при запросе незарегистрированного идентификатора.
- [ ] Один шаг синка принимает checkpoint на входе и возвращает новый checkpoint + флаг продолжения.
- [ ] Тестовая заглушка возвращает сконфигурированные записи пошагово, батч за батчем.
- [ ] Тестовая заглушка сигнализирует исчерпание после отдачи всех записей.
- [ ] Все юнит-тесты проходят без сетевых запросов и подключения к внешним сервисам.
- [ ] Тестовая заглушка с одними и теми же входными данными даёт воспроизводимый результат.
