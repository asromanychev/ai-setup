## Контекст

В `ai-da-collect` пока нет зафиксированной границы между ядром ingestion и подключаемыми источниками данных. Из-за этого нельзя безопасно развивать первый vertical slice и писать ранние unit/integration tests без привязки к Telegram, сети, БД и другим внешним зависимостям. Эта задача вводит минимальное ядро плагинов, достаточное для первых воспроизводимых тестов и последующего подключения реального плагина источника.

## Цель

Зафиксировать минимальный поведенческий контракт для плагина источника, реестр разрешения плагинов по `plugin_id` и детерминированный `NullPlugin`, позволяющий выполнять один шаг синка и тестировать ядро без внешних сервисов.

## Пользователь / Стейкхолдер

- Разработчик ядра ingestion, которому нужен единый способ вызывать плагины и передавать между шагами checkpoint.
- Разработчик плагина источника, которому нужен понятный и стабильный контракт подключения.
- Автотесты проекта, которым нужна локальная детерминированная реализация без сети, Telegram и БД.

## Входные данные

Система получает:
- `plugin_id` для поиска реализации через реестр.
- `source_config` как конфигурацию конкретного источника.
- `checkpoint_in` предыдущего состояния синка или `nil` для первого запуска.
- Запрос на выполнение ровно одного шага синка.

## Выходные данные

Система должна возвращать:
- Реализацию плагина по `plugin_id` через реестр.
- Результат одного шага синка с массивом `records`.
- Новый `checkpoint_out`, пригодный для следующего шага.
- Флаг `finished`, показывающий, завершён ли текущий проход.
- Явную ошибку для неизвестного `plugin_id` или нарушения контракта.

## Обязательные требования

- Контракт `Collect::Plugin` обязан описывать обязательные входы и выходы одного шага синка.
- Один шаг синка обязан принимать `checkpoint_in` и возвращать `records`, `checkpoint_out`, `finished`.
- Реестр обязан разрешать зарегистрированный встроенный плагин по `plugin_id`.
- Для неизвестного `plugin_id` реестр обязан выбрасывать явную тестируемую ошибку.
- `NullPlugin` обязан реализовывать тот же контракт, что и боевой плагин.
- `NullPlugin` обязан работать полностью локально, без сети, Telegram, БД, S3 и фоновых задач.
- При одинаковых `source_config` и `checkpoint_in` `NullPlugin` обязан давать одинаковый результат.
- Контракт должен быть задокументирован в коде и покрыт unit-тестами.

## Ограничения и запреты

- В рамках задачи не реализуются PostgreSQL, S3, Telegram API, Sidekiq и любая сетевая логика.
- Нельзя подменять неизвестный `plugin_id` молчаливым fallback-плагином или `nil`.
- Нельзя делать `NullPlugin` зависимым от внешнего состояния, не переданного явно через входные данные.
- Нельзя описывать здесь техническую реализацию конкретного первого боевого плагина.
- Точная форма `checkpoint` должна быть зафиксирована в следующем документе Spec, а не оставлена неоднозначной.

## Критерии успеха

- [ ] Контракт плагина задокументирован в коде и однозначно описывает обязательные входы и выходы одного шага синка.
- [ ] Реестр возвращает зарегистрированный плагин по `plugin_id`.
- [ ] Реестр выбрасывает явную ошибку для неизвестного `plugin_id`.
- [ ] Один шаг синка принимает входной checkpoint и возвращает новый checkpoint вместе с результатом шага.
- [ ] `NullPlugin` реализует тот же контракт, что и основной плагинный API.
- [ ] `NullPlugin` работает без внешних зависимостей и даёт детерминированный результат на одинаковых входных данных.
- [ ] Unit/integration tests для ядра можно запускать локально без сети, Telegram и БД.
