# Postmortem: настройка окружения с AI-агентом

## Разбор конкретного прогона: statusline + диагностика PATH

### Задача
Настроить строку статуса Claude Code (ccstatusline) чтобы показывала детальную
информацию по контексту — количество токенов, стоимость сессии, rate limits.

### Что агент сделал

1. Прочитал существующий `statusline-command.sh`
2. Предложил добавить временный дамп входящего JSON в `/tmp/statusline-debug.json`
   чтобы увидеть реальную структуру данных
3. Внёс изменение в скрипт, попросил написать следующее сообщение чтобы триггернуть
   обновление строки
4. Прочитал `/tmp/statusline-debug.json` — увидел реальные поля
5. Обнаружил, что в первой версии скрипта использовались несуществующие поля
   (`used_tokens`, `remaining_tokens`) — они отсутствовали в JSON
6. Переписал скрипт под реальную структуру: `current_usage.input_tokens`,
   `current_usage.output_tokens`, `current_usage.cache_read_input_tokens`,
   `cost.total_cost_usd`, `rate_limits.five_hour.used_percentage`

**Файлы, которые менялись:** `~/.claude/statusline-command.sh`

**Как проверялся результат:** запуск скрипта вручную с реальным JSON через pipe,
вывод сравнивался со скриншотом желаемого формата.

### Где агент ошибся

**Критическая ошибка с путём.** Когда я попросил скопировать шаблон проекта
в `ai-da-collect` и дал путь `/home/aromanychev/dev/aitlas/ai-da-collect` —
агент его принял и записал туда файлы. Правильный путь был
`/home/aromanychev/edu/aida/ai-da-collect`. Агент не переспросил, не проверил
существует ли директория и соответствует ли она контексту разговора.

Пришлось явно написать "зачем ты записывал не туда" — только тогда агент удалил
лишние файлы и добавил в нужное место.

**Несуществующие поля в первой версии скрипта.** Агент написал код использующий
`used_tokens` и `remaining_tokens` не проверив их наличие в реальном JSON.
Потребовалась дополнительная итерация с дампом чтобы это исправить.

### Когда хотелось остановить агента

Когда агент начал предлагать правки в несколько файлов подряд без паузы на проверку —
нужно было прервать и убедиться что каждый шаг верный, прежде чем двигаться дальше.

---

## Границы доверия

**Архитектурные решения.** Агент заполнил `CLAUDE.md` по `architecture.md` —
результат хороший, но требовал вычитки: некоторые детали он интерпретировал
по-своему. Финальное слово по архитектуре всегда за мной.

**Пути и директории.** Агент берёт путь из последнего упоминания в разговоре
и не сверяет его с контекстом проекта. Всегда указывать полный абсолютный путь
и не давать агенту угадывать.

**Деструктивные операции.** Удаление файлов, `rm`, сброс git — всегда проверяю
вручную прежде чем подтвердить.

**Тексты от моего лица.** Этот отчёт написал агент на основе нашей сессии.
Я его вычитал и скорректировал — но сам факт требует осознанности.

---

## Правила на следующий заход

1. **Всегда давать полный абсолютный путь** — агент не угадывает намерения по
   неполному пути.
2. **Проверять реальную структуру данных перед кодом** — сначала дамп/лог,
   потом реализация. Агент напишет код под предполагаемую структуру, а не реальную.
3. **Один шаг — одна проверка** — не давать агенту менять несколько файлов подряд
   без паузы на ревью каждого изменения.
4. **После записи файла — открыть и прочитать** — агент может написать правильно,
   но нужно убедиться самому.
5. **Давать агенту контекст из реальных файлов, а не из памяти** — "смотри
   architecture.md" работает лучше чем пересказ архитектуры своими словами.
